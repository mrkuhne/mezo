Modell A · két lapos tábla

Két tábla, pontosan a FE két tömbjére: pantry_ingredient (étel: makró, micros jsonb, nova, ár, készlet/lejárat, source) és supplement_stash_item (dózis, protokoll, timing, készlet). A stashRefId FK köti az ételt a supplementhez.
  • + Legkevesebb nullable mező, mindkét tábla feszes, olvasható.
  • + Read = két SELECT, nulla join → 1:1 a usePantry() alakkal.
  • − Két CRUD-útvonal (két controller/service), de mindkettő triviális.
  • − A FoodItem-katalógus egyesítés később jön (receptek/étkezések szeletnél).

Modell B · egy egységes tábla

Egy pantry_item tábla kind diszkriminátorral; az étel- és supplement-mezők uniója nullable oszlopokban. A kártya egységes chrome + kind-badge.
  • + Egy CRUD-útvonal, egy controller/service/tábla.
  • + Közelebb a hosszú távú egységes FoodItem katalógushoz.
  • − Sok nullable oszlop (a supplementnek nincs makró/nova, az ételnek nincs dózis/protokoll/timing) → gyengébb DB-integritás.
  • − A FE úgyis kind szerint bontja → a felhasználó ugyanazt látja, mint A-nál.

Modell C · katalógus + leltár

Szétválik a „mi" (food_item: név, márka, makró, nova, source — újrahasználható, lehet publikus) és a „mennyim van" (inventory: készlet, lejárat, ár-snapshot). A receptek és étkezések is a katalógusra hivatkoznának.
  • + Architektúra-hű — ez a hosszú távú cél (FoodItem-katalógus + pgvector embedding később).
  • + Valódi új UI: „Amim van / Katalógus" váltó, készlet nélküli tételek újrahasználata.
  • − Minden read JOIN-t igényel a lapos FE-alakra (usePantry nem változhat).
  • − 2× annyi tábla + a „katalógus 142" felület jelentősen kinöveli az első szeletet (a scope: leltár + CRUD volt).

Hogyan döntsünk?

A felhasználói felület A és B között gyakorlatilag azonos — a különbség pusztán backend (külön táblák vs. egy tábla nullable mezőkkel). Mivel a FE úgyis kind szerint szétbontja, ez tisztán adatmodell-döntés: A feszesebb sémát és erősebb DB-integritást ad (kevesebb nullable oszlop, mindkét aggregátum saját CHECK-jeivel). C az egyetlen, ami valódi új UI-felületet hoz (katalógus/leltár váltó), és architektúra-hű — viszont jóval nagyobb első szelet, és minden read join-t kíván a változatlan lapos usePantry() alakra. A megbeszélt scope („leltár + valódi CRUD") leginkább A-val áll összhangban, a katalógus-egyesítést a receptek/étkezések szeletére halasztva.
