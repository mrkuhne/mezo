# Me Area — Feature Documentation

> The profile + personal-biometrics + relationships hub. **Mixed status:** `Súly` (weight) and `Alvás` (sleep) are ✅ Phase-2 backed (the `biometrics` backend feature); `Cél` (the long-term goal) is ✅ active-goal backed; `Emberek` (People) is ✅ backend-backed since Slice E (`mezo-t16y.2` — `person`/`mention` tables, dual-mode `usePeople`); `Profil` renders backend biometrics + progression (the `useProfile` identity statics are a recorded static-is-fine decision, §9); `Tudás` (Knowledge) facts are ✅ backend-backed since companion V1.2 (Insights-domain dual-mode `useKnowledge`; the graph/edges layer stays mock). Lives under the `/me` tab (`MeSection`, Hungarian sub-nav `Profil` / `Cél` / `Súly` / `Alvás` / `Emberek` / `Tudás` — `MeSubNav.tsx:4-11`).
>
> **Placeholder strip (`mezo-lfw`):** the Phase-3 "vision" UI was removed so every screen reflects only what actually works. The four backend-backed views (`Profil`/`Cél`/`Súly`/`Alvás`) were cut to **backend-data only** — all narrative/decorative surfaces gone (Mezo "mit látunk" insight cards, "Hatások" factor cards, every decorative `ToolChipRow`, the Profil identity hero / identity-goal quote / PERMA bars / quick-setting rows / version footer, the `SettingsSheet` notification overview, the goal hero "Vége" projection stat, the `TrendCell` delta/ON-TRACK legs, the sleep weekly cells + chart target overlay + hero "Target" comparison). The two no-backend views (`Emberek`/`Tudás`) were **trimmed** to a working mock shell (kept: People log/grid/feed, Knowledge fact list; removed: credit hero, ritual/attention/relation-pattern cards, IDENT-5 footer, the "Gráf nézet hamarosan" stub, tool chips). Net: ~1.2k lines deleted, 13 view-local components dropped, the `WeightTrends`/`SleepTrends`/`PeopleSummary` mock shapes pruned. See §2/§4/§9.

## 1. Summary

The **Me** area is the user's personal hub: who they are (`Profil`), their long-term body-weight goal & strategy (`Cél`), their daily weight log + trend (`Súly`), their sleep log (`Alvás`), the people they track (`Emberek`), and a knowledge facts surface (`Tudás`). It exists as the "self & relationships" counterpart to the activity-focused tabs (Today, Train, Fuel, Insights).

## 2. User-facing behavior

`MeSection` (`frontend/src/features/me/pages/MeSection.tsx`) renders the `MeSubNav` tab strip plus an `<Outlet>`, and owns the `SettingsSheet` open/close state, passing `openSettings` down via outlet context (`MeOutletContext = { openSettings: () => void }`).

### `Profil` (`pages/ProfilePage.tsx`) — ✅ backend-only (post-`mezo-lfw`)
**Stripped to backend data only.** Renders a static `Profil` header (the mock `user.name` title was retired — chrome only) + a gear chip and content cards: the **`Biometria` card**, (P6, `mezo-xje5`) the two **progression cards** (`AthleticRadarCard` + `MuscleLevelsCard`), and (Growth E2, `mezo-jzca`) the **`GrowthCard`** (`ProfilePage.tsx:38`), in a `col gap-md` below biometrics. The gear calls `onOpenSettings()` → `SettingsSheet` (**theme toggle via `useTheme`** — the notification overview was removed). Everything else the old dashboard showed — concentric avatar, name/handle, member/streak/mesocycle stats, the identity-goal quote, "Aktív területek" PERMA bars, the Tudás/Emberek `EntryCard` shortcuts, quick-setting rows and the version footer — was **deleted** (`mezo-lfw`) along with their components (`ConcentricAvatar`/`ProfileStat`/`EntryCard`/`SettingsListRow`) and the `data/me.ts` mock file. Navigation to the other tabs is via `MeSubNav`.

**Biometria card + editor (G6, `mezo-06n`):** the first-class biometric surface — set once, the engine always computes the base-TDEE from it (and it's the **hard gate** on goal creation — see the `Cél` "Új cél" gate below, `GoalGate.tsx`). `<BiometricCard>` (`components/BiometricCard.tsx`) is now the only content card on `Profil`; it reads `useBiometricProfile()` and renders the notch-clipped mockup card (`profile-biometria-v2`): a brand accent bar + a sex/height/age(derived from `birthDate`)/bodyFat/activity (× PAL) stat grid + the **derived base-TDEE line** from `profile.tdeeBootstrap` (`≈{tdee} kcal/nap · formula Katch|MSJ`, **omitted when `tdeeBootstrap == null`** — i.e. before the goal is evaluated). When no profile exists yet (real-mode 404) the card shows an **empty-state prompt** ("Állítsd be a biometriád"). Tapping `Szerkesztés ›` (or the empty prompt) opens **`BiometricSheet`** (`BiometricSheet.tsx`) — a `Sheet` (children-as-function) editing Nem (segmented M/F), Magasság (cm), Születési dátum (date), Testzsír% (optional), Aktivitási szint (5 options → the PAL enum, default `MODERATE`) — the field markup lifted from the deleted `GoalPlannerPage` Step2. **Mentés** calls `useBiometricActions().upsert(body)` (which invalidates `['biometricProfile']` + `['goals']` so the active goal recomputes server-side) then `close()`s on success (the `EditGoalSheet` "mutation inside the sheet" pattern).

**Progression cards (P6, `mezo-xje5` — the FE consumer of the P4 profile endpoint, completing the `mezo-8e4` epic):** below the Biometria card sit `<AthleticRadarCard>` + `<MuscleLevelsCard>` (`components/`), both reading the dual-mode `useProgressionProfile()` (§3) and **ghost-guarding on `profile.athleteLevel == null`** (the spec's ghost predicate — mirrors `BiometricCard`'s `if (!profile)` prompt; shown before any XP, or in real mode when the progression switch is off → the GET 404s → the hook's ghost `realEmpty`). **`AthleticRadarCard`** is a hand-rolled **SVG hexagon radar** (`radarGeometry.ts` pure point math — `cx=cy=124`, `R=88`, axis `i` at `-90°+60·i` clockwise; scaled to a fixed `radarMax = max(10, ceil(values))` level baseline) plotting the **6 server-computed `radarAxes`** (Erő/Robbanékonyság/Sebesség/Állóképesség/Mozgékonyság/Koordináció — consumed verbatim, NOT recomputed) + grid rings + axis labels (abbreviated) + a 3-cell stat row (athlete-level highlighted · best-athletic icon via `skillDisplay` · streak weeks). **`MuscleLevelsCard`** sorts `profile.muscle` by `(level desc, cumulativeXp desc)`, shows the top-4 (rank · `MUSCLE_LABELS` name · `Lv n` · a level/10 bar) + a reserve note naming the lowest (`{label} a legtöbb tartalék (Lv n)`). Both reuse the P5 `useReducedMotion` gate (radar scale-in / bar-grow collapse to final state) and ghost-prompt accessibly; the radar's `role=img` SVG is `aria-hidden` with an **`.sr-only` summary sentence** carrying the per-axis values to assistive tech (the P5 `lu-sr-only` lesson). The **"Teljes profil" / "Mind a 13" links are omitted in v1** (the full-profile route is deferred — spec §"Deferred to v1.1"); a non-navigating chip would be a finger-trap, so the headers show only the eyebrow until the route ships.

**Growth — LIFE card (Growth E2, `mezo-jzca`, `GrowthCard.tsx`):** below the muscle-levels card sits **`<GrowthCard>`** — the Me-side surface of the gamified-growth **LIFE band**. Reading the same `useProgressionProfile()` (`profile.life[]` + `profile.traits`), it renders a hand-rolled **SVG octagon radar** of the 8 LIFE skill levels in taxonomy order (emoji axis labels via the shared `radarGeometry` N-axis math + `LIFE_SKILLS`), the **top-3 LIFE skills** as `progress-mrow` meter rows, and two **computed** trait meters — **Fegyelem** (`traits.disciplinePct`, `–` when null) + **Következetesség** (`traits.consistencyWeeks` "hét", scaled /12) — under the caption „A számaid mondják ki — nem önbevallás." **Since Growth E3 (`mezo-6ng8`) a "Megtakarítás (30 nap)" row** sits below the trait meters showing `profile.savingsHuf30d` as `{amount} Ft` (brand-glow) — **rendered only when `savingsHuf30d` is a number > 0** (null = activity feature off, or ≤ 0 → hidden). **Ghost-guards** on all-zero LIFE XP (`life.every(cumulativeXp === 0)`) with a brand-tinted "Az élet is edzés." prompt pointing at the Today activity log. The domain itself (activity log, traits, savings, quest synergy) is documented in [growth.md](growth.md) — this card is only its Me consumer.

### `Cél` (`pages/GoalsPage.tsx`) — ✅ goal command-center (G4b; placeholder-stripped `mezo-lfw`)
The long-term goal & strategy surface — **no daily weight log here anymore** (that moved to `Súly`, G2). A tappable goal hero (trajectory + guard pills, kg progress track start→target, **two real stats** `Hátra` [weight-log derived] + `Tempó` [the real EWMA `last4w.weeklyRate`], identity frame) opens **`EditGoalSheet`** — now a real **manage sheet** (G4b): besides the read-only field rows it carries an **editable "Napi ritmus" day-planner section** (Fuel P5, `mezo-9ys`) + **Archiválás** (`archive(goalId)`) + **Törlés** (`remove(goalId)`, behind an inline two-step "Biztosan törlöd?" confirm; destructive color `var(--error)`) wired to `useGoalActions()`. **Napi ritmus** surfaces the goal's day-planner settings — an `Étkezés/nap` ±-stepper (clamped 3–6) + `Ébredés`/`Lefekvés` `<input type="time">` anchors, defaulting from the loaded goal (`goal.mealsPerDay ?? 4` / `wakeTime ?? '06:00'` / `bedTime ?? '23:00'`) — and its **Ritmus mentése** button calls `useGoalActions().savePlanner(goalId, goalResponse, { mealsPerDay, wakeTime, bedTime })`, which rebuilds a **full** `GoalUpsertRequest` from the raw `goalResponse` (via `goalResponseToUpsert`, so the window/weights/guards ride along unchanged) with the edited planner overrides and PUTs it (`goalApi.update`). This is the **first goal write-back from the sheet** — previously it was read-only. On success the sheet closes and the `['goals']` invalidation refetches `useGoal` — when no active goal remains, `GoalsPage` falls back to the empty state. **The hero reads the raw `GoalResponse` directly (G4b Decision C):** trajectory (→ `TRAJECTORY_LABEL`), `guards` (chip pills), the window (`huMonthDay(startDate/targetDate)`), and `startWeightKg`/`targetWeightKg` — these no longer pass through the `toGoal` back-compat mapper (which dropped its `startDate`/`targetDate`/`unit` fields). The latest-weight/progress numbers (`currentWeight`/`startWeight`) still come from `goal` (weight-log derived). In mock mode the hero renders the static `mockGoal` + `mockGoalResponse`. **Empty-state guard (`GoalsPage.tsx`, `mezo-72d`):** when `useGoal()` returns `goal === null`/`goalResponse === null` (real mode, no active goal), `GoalsPage` short-circuits to a `GhostState` "set up a goal" card with a `＋ Új cél` CTA (→ `/me/goals/new`) **before** any `goal.X`/`goalResponse.X` read. **The Mezo `InsightCard` section (`weightTrends.insights`) + the "Hatások" `FactorCard`s + the decorative `ToolChipRow` were removed (`mezo-lfw`)** — they were static narrative the engine never computes. Below the hero now: the **`<GoalTimeline>` lane view** (G4b) — gym/run plan bars positioned by start/end week + uncovered-window gap chips + the ambient volleyball band, replacing the old "Cél alatt fut" `LinkedMesoCard`s. Each plan bar's ✕ detaches the link via `useGoalActions().detachPlan(goalId, linkId)`. **Under the timeline sits the attach/detach hub** — `<GoalPlanSlots goalId>` (`components/GoalPlanSlots.tsx`): two always-present slots, **Mesociklus** + **Futóblokk**, each offering `＋ Tervezd` (launch the existing planner — meso `navigate('/train/mesocycles/new')`; run the create-then-navigate idiom `saveRunningBlock(null, newDraft(…), { onSuccess: b => navigate('/train/futas/'+b.id) })`) and `＋ Csatolj meglévőt` (open **`AttachPlanSheet`**, `AttachPlanSheet.tsx`). The attach sheet lists the user's owned plans of that type (`useTrain().mesocycles` / `useRunning().runningBlocks`) **excluding those already linked** (id in `timeline.links[].planId` for that type) + a start-week input (1..`timeline.weeks`), and confirms via `useGoalActions().attachPlan(goalId, { planType, planId, startWeek })` (POST `/api/goals/{goalId}/plans`). Volleyball is **not** a slot (ambient/read-only — it lives in the timeline band). `GoalsPage` calls `useGoal()` (goal + goalResponse + timeline + goalId), `useGoalActions()` (detach + evaluate), and `useWeight()` **only for `weightTrends.last4w.weeklyRate`** (the hero's `Tempó` stat). The `WeightChart`/`TrendCell`/`WeightLogSheet`/period-toggle it used to own are now in `WeightPage`.

**Recept — the G5 engine finale (`components/GoalRecept.tsx`, `mezo-g1u`):** below the timeline, `<GoalRecept prescription={goalResponse.prescription} onEvaluate={() => evaluate(goalId)} evaluating>` renders the engine's segmented prescription, **replacing G4b's static "G5 · hamarosan" placeholder** (the placeholder block was removed from `GoalTimeline.tsx`; the timeline is now purely time-axis lanes). The card has three parts: (1) a **feasibility verdict banner** — `feasible` → "Reális" (`var(--brand-glow)`), `feasible-with-warnings` → "Reális, figyelmeztetésekkel" (`var(--warning)`), `aggressive` → "Agresszív" (`var(--error)`), with `feasibility.notes` listed; (2) **per-segment recept cards** — each `prescription.segments[i]` shows the week range `W{fromWeek}–{toWeek}`, the `label`, and four metrics (`kcal`, `proteinG` g, `sleepTargetH` h, signed `projectedRateKgPerWk` kg/hét) + the HU `rationale`; (3) **guard-status pills** — strength (e1RM trend %, `var(--error)` + "· sérülve" when `breached`), muscle (`≥{minWeeklySetsPerMuscle} szett/izom`, `belowMaintenanceMuscles` count + rate-cap warning in `var(--warning)`), and protein shown as a muted "Fehérje: Fuel-re vár" pill since `proteinMonitored=false`; guard `notes` list below the pills. **Null prescription** (real mode, goal not yet evaluated) → an **"⚡ Értékeld a célt" CTA** that fires `useGoalActions().evaluate(goalId)` (POST `/api/goals/{id}/evaluate`); on success the `['goals']`/timeline invalidation refetches and the card renders the fresh prescription. Mock mode renders the static `mockGoalResponse.prescription` (a `feasible-with-warnings` verdict + 2 segments + guardStatus, `data/me/goals.ts`).

**Goal-creation hard gate (G6, `mezo-06n` Task 7, `GoalGate.tsx`):** both `+ Új cél` entries in `GoalsPage` (the header chip + the empty-state `GhostState` CTA) route through a shared `startNewGoal()` handler (`GoalsPage.tsx:57`) that reads `useBiometricProfile().isComplete`. **Complete** → `navigate('/me/goals/new')` (straight to the wizard). **Incomplete** (real-mode 404 → `isComplete=false`) → `setGateOpen(true)` instead of navigating, rendering **`<GoalGate>`** — a full-screen interstitial overlay (mockup `hard-gate.html`: notch glyph + "Előbb: a biometriád" + the explanatory copy + **warning chips for the missing fields** computed from `profile` (sex/heightCm/birthDate) + a "Biometria beállítása →" CTA). The CTA opens the same Profile **`BiometricSheet`**; the gate re-reads `useBiometricProfile()` and, once its upsert (which invalidates `['biometricProfile']`) flips `isComplete` to true, a `useEffect` calls `onComplete()` → continues into the wizard. The user is never dropped into `GoalPlannerPage` without a usable profile. **The gate is ALSO enforced at the route level (G6 review fix, `GoalPlannerPage.tsx:37-42`):** `GoalPlannerPage` is now a thin guard wrapper that reads `useBiometricProfile()` — it renders `null` while loading, redirects to `/me/goals` via `<Navigate to="/me/goals" replace />` when the profile is incomplete, and only renders the `GoalWizard` body once complete — so a direct URL / back-forward nav / bookmark to `/me/goals/new` can't bypass the gate.

**Goal creation (G4a, reshaped to 2 steps in G6 `mezo-06n`):** the `+ Új cél` chip in the `GoalsPage` header (`GoalsPage.tsx:107`, behind the gate above) reaches the sibling route `/me/goals/new` → **`GoalPlannerPage`** (`frontend/src/features/me/pages/GoalPlannerPage.tsx`), a **2-step** wizard (`STEP_COUNT = 2`):
1. **Trajektória + guards** — pick `cut`/`bulk`/`maintain` + optional `strength`/`muscle` guards.
2. **A cél — window + weights + identity** (`Step1`) — title, start/target dates, start/target weight (target hidden for `maintain`), optional identity frame. **No manual rate input** (G6 dropped it — the backend derives the weekly pace) and **no biometric fields** (G6 moved them to the Profile as a creation precondition, not a wizard payload). `canNext` requires a title + `targetDateIso > startDateIso`.

**Live feasibility preview (G6, `mezo-06n`, `FeasibilityPanel`):** on the cél step, as the user edits target weight / target date (and trajectory / start weight), `useFeasibilityPreview` (debounced ~400ms, real-mode `POST /api/goals/feasibility-preview`) returns the backend-derived `derivedRatePctPerWeek` + `verdict` + (when over the cap) a `suggestedTargetDate`, and the panel renders one of three states (mockup `goal-wizard-v2.html`): `withinSafeBand` → **brand-tinted** "X,Y %BW/hét · ✓ Reális" (`feasible-with-warnings` → "✓ Reális · figyelővel") + a kg/weeks summary (`≈Δkg · N hét`, computed from the inputs); **aggressive** (`!withinSafeBand`) → **warning-tinted** "X,Y %BW/hét · ⚠ Agresszív" + a **"↦ Reális dátum: <suggestedTargetDate> — Elfogadom"** button that sets `targetDateIso` to the suggestion (→ re-previews → flips to feasible). The CTA stays enabled throughout (soft nudge — the user MAY proceed). `maintain` (no target weight) shows a simple "≈ Tartás" note instead. Mock mode renders a static feasible preview (`feasibilityPreview` in `data/me/goals.ts`).

`save(activate)` builds a `GoalUpsertRequest` (NO `rateTargetPctPerWeek` — backend derives it; NO biometric profile) and calls `useGoalCreation().submit(input, { onSuccess: () => navigate('/me/goals') })` — two CTAs on the cél step: **"Cél létrehozása + aktiválás"** (`activate=true`) and **"Mentés tervezettként"** (`activate=false`). Step weights default from the latest `useWeight` entry. In **mock mode** the submit no-ops and just navigates back (Phase-1 parity with `MesocyclePlannerPage`).

- `GET /api/biometrics/weight` → `WeightLogResponse[]` (`{id, date, value, note}`, date-asc). Mapper renames `weightKg → value` (`WeightLogMapper`).
- `POST /api/biometrics/weight` — `LogWeightRequest {date, weightKg (>0, ≤999.99), note}` → 201.
- `GET /api/biometrics/sleep` → `SleepLogResponse[]` (`{id, date, bedtime, wakeup, duration, quality, awakenings, mealToSleep, notes}`). Mapper (`SleepLogMapper.java`) maps `durationH → duration` and **`mealToSleep` is a hardcoded `constant = "0"`** (the Fuel seam — §5.3).
- `POST /api/biometrics/sleep` — `LogSleepRequest {date, bedtime, wakeup, durationH, quality (1–10), awakenings, note}` → 201.

**Goal** (G1, `api/feature/goal/goal.yml`; tag `Goal` → `GoalApi`; `GoalController implements GoalApi`):
- `GET /api/goals` → `GoalResponse[]` — **active goal first** (`GoalService.listGoals` hoists `status=='active'`, then start-date desc).
- `POST /api/goals` — `GoalUpsertRequest` → 201; service forces `status='planned'`. `PUT /api/goals/{id}` updates fields **but never touches `status`** (lifecycle endpoints own it). `DELETE /api/goals/{id}` → 204 (soft delete).
- `POST /api/goals/{id}/activate` → 200 — enforces the **single-active-per-owner** invariant in the service (archives every other active goal in the same tx). `POST /api/goals/{id}/archive` → 200.
- `GoalUpsertRequest` uses `pattern` (not `enum`) on request `trajectory`/`guards` (a bad enum would 500 in Jackson, a `pattern` mismatch 400s — the house gotcha); `GoalResponse` uses `enum`. Ownership gate returns **404 for missing *or* foreign** (no existence leak).

**Biometric profile** (G1, `api/feature/biometrics-profile/biometrics-profile.yml`; tag `BiometricProfile` → `BiometricProfileApi`):
- `GET /api/biometrics/profile` → `BiometricProfileResponse {sex, heightCm, birthDate, bodyFatPct?, activityLevel?, tdeeBootstrap?}` — **404 when the owner has no profile yet** (`BiometricProfileService.getProfile`); `useBiometricProfile` catches the 404 → `null` (§3). The nullable **`tdeeBootstrap {bmr, tdee, pal, formula(MSJ|KATCH), computedAt}`** (G6, Task 3) is the derived base-TDEE snapshot the Biometria card's TDEE line reads.
- `PUT /api/biometrics/profile` — `BiometricProfileUpsertRequest {sex (M|F), heightCm (50–260), birthDate, bodyFatPct? (0–75), activityLevel? (SEDENTARY|LIGHT|MODERATE|VERY|EXTRA)}` → 200; **find-or-create by `createdBy`** (insert once, update thereafter — one row per owner). Sent by the Profile **`BiometricSheet`** (G6, via `useBiometricActions().upsert`) with `activityLevel` (default `MODERATE`).
- `GET /api/biometrics/weight/trend` → `WeightTrendResponse {ewmaSeries[], latestTrendKg, weeklyRateKgPerWeek, weeklyRatePctPerWeek, last4wRateKgPerWeek, dataSufficiency}` — the real EWMA weight trend (G5); `useWeight` folds it into `weightTrends` (§4).
**People** (Slice E, `api/feature/people/people.yml`; tag `People` → `PeopleApi`; `PeopleController implements PeopleApi`):
- `GET /api/people` → `PeopleResponse { persons: PersonResponse[], mentions: MentionResponse[] }` — one bootstrap read (the knowledge pattern). Persons ordered **mention-count desc then name**; `mentionCount`/`mentionsThisWeek` (rolling 7d)/`lastMentionedAt` are **computed from live mention rows in `PeopleService`, never persisted/seeded**. Mentions ts-desc, capped at 50.
- `POST /api/people/{personId}/mentions` — `LogMentionRequest { tone (pattern), text? ≤500 }` → 201 `MentionResponse`; the server stamps `ts=now`, `source='chip'`, `flagged=false`. **404 for missing *or* foreign person** (no existence leak); bad tone → 400 field error.
- 401 on all → `SystemMessageList`. The `goal` write/lifecycle endpoints exist but the FE does not yet POST to them (read-only consumption — §6).

## 6. How to use it (consume)

Import the hook from `@/data/hooks` (never deeper) and destructure. Always ghost-guard real mode (empty arrays / `lastNight === undefined` on first paint).

```tsx
import { useGoal, useWeight, useSleep, usePeople } from '@/data/hooks'

function Example() {
  const { goal, goalResponse, linkedMesocycles, timeline, goalId } = useGoal()  // active goal + raw contract + plan-link timeline
  const { archive, remove, detachPlan } = useGoalActions()     // goal mutations (G4b; dual-mode)
  const { weightLog, weightTrends, logWeight } = useWeight()    // weight log + real EWMA rates
  const { sleepLog, lastNight, logSleep } = useSleep()          // sleepTrends was removed (mezo-lfw)
  const { people, mentions, logMention } = usePeople()          // summary/patterns were removed (mezo-lfw)

  if (!lastNight) return <p>Még nincs alvásadat.</p>   // canonical real-mode guard

  logWeight({ date: '2026-06-14', weightKg: 82.4, note: null })  // WeightLogInput
  logSleep({ date: '2026-06-14', bedtime: '23:10', wakeup: '06:40',
             durationH: 7.5, quality: 8, awakenings: 1, note: null })  // SleepLogInput
  // ...
}
```

Notes:
- **`useGoals()` no longer exists** — it was split into `useGoal()` (goal + `linkedMesocycles`) and `useWeight()` (`weightLog`/`weightTrends`/`logWeight`). A view that needs both calls both (e.g. `WeightPage` calls `useWeight` for the log/trend and `useGoal` for both `goal` [hero start/target/kind] and `goalResponse` [the chart's plan trajectory], `mezo-l82h`; `GoalsPage` calls `useGoal` for the hero and `useWeight` only for `weightTrends`). `useGoal()` is **read-only**; goal **creation** is a separate hook — **`useGoalCreation()`** (G4a, `{ submit, pending }`, drives `GoalPlannerPage`). Editing an *existing* goal in place is still G4b.
- `useProfile()` returns just `{ user }` (static, shared with Today + Fuel) since the Profil strip (`mezo-lfw`). The **biometric profile is a separate hook** — `useBiometricProfile()` (read, 404→null, `isComplete`) + `useBiometricActions()` (upsert), both from `@/data/hooks` (G6, §3) — consumed by `ProfilePage`'s Biometria card/sheet (and reused by the goal-creation hard gate).
- Mutations are **fire-and-forget** — `logWeight`/`logSleep`/`logMention` call `mutation.mutate(input)` and surface no promise or loading flag to the caller today.
- `WeightPage` tolerates an empty `weightLog` because `WeightTrendChart` shows the soft "Kevés mérés…" hint when `<2` points fall in the window (and renders actual-only with no goal), no `WeeklyWeightCard`s render (`groupByWeek([]) === []`), and the hero falls back to `goal?.currentWeight ?? 0` (goal may be `null` since `mezo-72d`); `SleepPage` must guard `lastNight` explicitly. `GoalsPage` guards `goal === null` up front (empty-state CTA).
- Run modes: mock (no backend) — `VITE_USE_MOCK=true pnpm dev`; real (default) — needs the backend on `:8090` with the `demodata` profile (owner seed makes login work; **weight/sleep start empty — they are NOT seeded**; the 5 People persons ARE seeded, with an honest empty mention feed — the demo feed is opt-in `demofixtures`).

Key decisions: **bubble-up local state** (mirrors `useCheckins`, no store); **no date picker** in sheets (`date` is stamped to "today" by the mutator but is part of the durable contract because Phase 2 persists it); **voice CTA is decorative**; **EditGoal display-only**; **PersonDetail → PersonLog nesting**.
- **Backend was issue `mezo-v67`** — auth + weight + sleep + checkin shipped together as biometrics Slice A (changeset filenames carry `mezo-v67`).
- **Goal + biometric-profile backend was issue `mezo-2hp` (goal-system slice G1)** — spec [`docs/superpowers/specs/2026-06-18-goal-system-design.md`](../superpowers/specs/2026-06-18-goal-system-design.md). **G1 is foundation only:** the `goal`/`biometric_profile` aggregates + CRUD/lifecycle endpoints + the `useGoals → useGoal/useWeight` split, but **no** timeline/plan-links (G3), **no** TDEE/prescription engine (G4/G5). The `toGoal` adapter (`goalHooks.ts:12`) keeps `GoalsPage` on the legacy `Goal` shape; the writable `EditGoalSheet` + native `GoalResponse` rendering is G4.
- **Súly-tab IA split was issue `mezo-9sv` (goal-system slice G2)** — the daily weight log + trend chart + trend cells moved out of `GoalsPage` into a dedicated `WeightPage` at `/me/weight` (`router.tsx:95`, `MeSubNav.tsx:7`), leaving `Cél` goal/strategy-only. Pure frontend IA — **no backend or contract change** (both views reuse the G1 `useWeight`/`useGoal` hooks). `WeightPage`'s `useGoal()` call now drives the chart's plan trajectory + tolerance band via `goalResponse` (not just start/target reference lines); the `mezo-4nu` goal-coupling note is superseded.
- **Date-less owned aggregates** (`goal`, `biometric_profile`) extend `JpaRepository`, **not** `OwnedRepository` — the latter's `findAllOwned` needs an `e.date` field these tables lack. Bespoke `findBy…CreatedByAndDeletedFalse` finders instead.
- **Single-active goal invariant** is enforced **in the service** (`GoalService.activateGoal` archives every other active goal in the activate transaction), not by a DB constraint — by design, so reactivation/history stays flexible.
- **`mealToSleep` is a stub `0`** (`SleepLogMapper.java` constant + contract note) — the live Fuel→Sleep seam (§5.3).
- **Phase 3 (companion) CONSUMES this domain, not extends it — ✅ wired at V0.3, deepened at V0.5:** the `ContextSnapshotAssembler` ([`companion.md`](companion.md) §5.5) reads the goal + prescription + day-planner fields, `biometric_profile`, `WeightTrendService.computeTrend` and the latest `sleep_log` read-only into every chat turn's system prompt; V0.3 added the `SleepLogRepository.findFirstByCreatedByAndDeletedFalseOrderByDateDesc` finder for it. **V0.5 (chat tools)** added the `SleepLogRepository` since-date finder (for `get_sleep`) and extracted the prescription-segment picker to a shared static `GoalPrescriptionJson.currentSegment` (used by the snapshot AND `get_goal_progress`); `get_weight_trend`/`get_goal_progress` reuse `computeTrend` unchanged. **Proactive B1.2 (`mezo-h4wp.2`)** added a third read-only `SleepLogRepository` finder — an `existsBy…DateGreaterThanEqualAndCreatedAtAfter` staleness probe the morning-briefing read path uses to decide whether last night's sleep arrived after the briefing was generated ([`proactive.md`](proactive.md)). **Proactive P1 (`mezo-h4wp.7`)** added a read-only window finder on BOTH repos — `WeightLogRepository.findByCreatedByAndDeletedFalseAndDateGreaterThanEqualOrderByDateDesc` (the weight sibling of the existing sleep one), consumed by the shared `MetricWindowEvaluator` (extracted at **P2 `mezo-h4wp.8`** so P1 validation + P2 experiment outcome share the same weight/sleep/training window-vs-baseline reads). Coupling stays one-directional (companion/proactive → me), no Me-side behavior change. The `mezo-lfw`-removed decorative `ToolChipRow`s were the mock preview of exactly this — real tool chips landed via the companion chat surface at V0.5, not via Me views.
- **`SleepEntry` type over-promises nullability:** the domain type claims `bedtime`/`duration`/… are always present, but the contract/DB allow null; `sleepApi.list`/`log` use an explicit `as Promise<SleepEntry[]>` cast (`biometricsApi.ts:28-38`) flagged "until normalized (see mezo bd issue)." A latent FE↔BE type mismatch to fix.
- **`OwnedRepository.findAllOwned` requires a `date` field** (JPQL `order by e.date asc`); both weight/sleep have one. `useSleep.lastNight` silently depends on this ordering — a date-less owned entity would need its own finder.
- **Belt-and-braces soft delete:** `findAllOwned` filters `deleted=false` AND each entity carries `@SQLRestriction` — intentional, keep both.
- **People backend was issue `mezo-t16y.2` (Phase-2 Slice E)** — plan [`2026-07-04-people-slice-e-plan.md`](../superpowers/plans/2026-07-04-people-slice-e-plan.md). Key decisions: **(1) persons seed under plain `demodata`** (deliberate exception to "demo content → demofixtures": the IDENT-5 five are the owner's real inner circle, and v1 has no person-create UI — an unseeded deploy would leave the tab unusable; the mention **feed** fixture IS `demofixtures`-only, with now-relative timestamps); **(2) honest derived stats** — `mentionCount`/`mentionsThisWeek`/`lastMentionedAt` are computed from live mention rows, never seeded (the mock's `84` would be a lie), while the narrative person fields (`knownFacts`/`ties`/`affectTrend`/notes/cadence) ARE seeded owner-curated data — a *computed* affect trend belongs to the pattern-engine epic; **(3) enums as DB CHECKs** (house style); **(4) mentions do NOT feed the companion snapshot** (proactive epic); **(5) display labels FE-derived** (contract carries `ts`/`lastMentionedAt` only); **(6) v1 write path is the chip sheet** — server stamps `source='chip'`/`ts`/`flagged=false`; the `flagged` filter shows only fixture rows until a flagging surface ships.
- **`useProfile` decision (Slice E, parked since 2026-06-10 §5):** `user: UserMeta` **stays a static const — recorded, not wired.** No real-mode surface renders the identity statics (Today overrides only the meso fields from `useTrain()` — Slice T; `FuelStackPage` reads seed consts — `mezo-4nu`; the Profil identity hero died in `mezo-lfw`), and the legacy `user_profiles` table lacks `name`. Revisit when a Profile identity surface returns.
- **Mock-only, no backend:** `Tudás`/Knowledge edges/graph (trimmed to a working mock shell in `mezo-lfw`; facts are real). **The placeholder strip (`mezo-lfw`) deleted** the weight/sleep-trend factor/insight/pattern narrative data, every decorative `ToolChipRow` from Me, the Profil dashboard mock (`data/me.ts`), and the `WeightTrends`/`SleepTrends`/`PeopleSummary`/`RelationPattern` mock shapes — so the four backend-backed views (`Profil`/`Cél`/`Súly`/`Alvás`) now render only real API data. (The legacy `user_profiles` table is still unexposed. The active **goal** is backed — not in this list.)
- **Goal creation was issue `mezo-pqt` (goal-system slice G4a)** — the `GoalPlannerPage` 3-step wizard at `/me/goals/new` (`router.tsx:102`) reached from the `+ Új cél` chip in `GoalsPage`, saving via the new `useGoalCreation` hook (PUT `biometric_profile` → POST `goal` → optional `activate`). **Pure FE write-path slice** — no backend/contract change (the G1 endpoints already existed). Deliberately scoped: no command-center timeline UI, no attach/detach hub, no `toGoal` retirement, no existing-goal editor (all G4b). Mock mode no-ops + navigates back (Phase-1 parity).
- **Goal command-center was issue `mezo-tji` (goal-system slice G4b)** — the `<GoalTimeline>` lane view replaced the "Cél alatt fut" `LinkedMesoCard`s; `EditGoalSheet` became a manage sheet (Archiválás + Törlés via `useGoalActions`); the hero reads the raw `GoalResponse` (trajectory/guards/window/weights), retiring `startDate`/`targetDate`/`unit` from `toGoal` (Decision C); and the **attach/detach hub** landed — `GoalPlanSlots` (Mesociklus + Futóblokk slots: ＋ Tervezd launches the planner, ＋ Csatolj meglévőt opens `AttachPlanSheet`) over `attachPlan`, plus the per-bar ✕ `detachPlan`. FE-only (reuses the G1 endpoints + G3 timeline). The `ambient` volleyball-band prop is intentionally omitted (the band renders constant). The recept strip was a static G5 placeholder inside `GoalTimeline` — **G5 (`mezo-g1u`) removed it and replaced it with the real `<GoalRecept>` card** rendered by `GoalsPage` from the engine prescription (see the `Cél` §).
- **Goal engine surface was issue `mezo-g1u` (goal-system slice G5)** — the FE side of the TDEE/recept engine: `<GoalRecept>` (`components/GoalRecept.tsx`) renders the segmented `prescription` (per-block kcal/protein/sleep/rate + rationale) + the feasibility verdict + the guard-status pills, replacing G4b's static placeholder; `useGoalActions().evaluate(goalId)` (POST `/api/goals/{id}/evaluate`) runs the engine when a goal has no prescription yet (the "Értékeld a célt" CTA); `useWeight` now reads the real EWMA trend in real mode (`GET /api/biometrics/weight/trend`, folded into `weightTrends`); and `GoalPlannerPage` Step 2 gained the `activityLevel` PAL picker. Mock mode supplies a static `prescription` + `weightTrends`.
- **Deferred to later goal-system slices (G5b/Phase-3):** an existing-goal **field editor** (the sheet's field rows are still read-only) + a fuller native `GoalResponse` render (more `toGoal` retirement as consumers migrate); the **adaptive** TDEE leg (back-calc from Fuel intake + the EWMA blend — needs Fuel Slice C) + the protein guard going live (`proteinMonitored`). (The Profile biometric card + editor + `useBiometricProfile`/`useBiometricActions` shipped in **G6**; goal *creation* shipped in G4a; the **attach/detach hub** + timeline lane shipped in G4b; the Train→`Cél` mesocycle plan-links that fill `Goal.mesocycles` shipped in G3; the Súly-tab UI move shipped in G2.)
- **Deferred to later epics** (see [`docs/milestones/roadmap.md`](../milestones/roadmap.md)): the pattern engine legs the factors/relation-patterns *described* but nothing computes (person-mention pattern mining included); the Fuel→Sleep `mealToSleep` join; People CRUD + real mention↔domain joins + mention→companion-snapshot wiring.
