# Me Area — Feature Documentation

> The profile + personal-biometrics + relationships hub. **Mixed status:** `Súly` (weight) and `Alvás` (sleep) are ✅ Phase-2 backed (the `biometrics` backend feature); `Cél` (the long-term goal) is ✅ active-goal backed; `Emberek` (People) is ✅ backend-backed since Slice E (`mezo-t16y.2` — `person`/`mention` tables, dual-mode `usePeople`); `Profil` renders backend biometrics + progression (the `useProfile` identity statics are a recorded static-is-fine decision, §9); `Tudás` (Knowledge) facts are ✅ backend-backed since companion V1.2 (Insights-domain dual-mode `useKnowledge`; the graph/edges layer stays mock). Lives under the `/me` tab (`MeSection`, Hungarian sub-nav `Profil` / `Growth` / `Cél` / `Súly` / `Alvás` / `Emberek` / `Tudás` — `ME_TABS` in `pages/tabs.ts`, rendered via the shared `SubNavDropdown` inside `AppHero`, compact-header redesign `mezo-ugqb`). **`Growth` (`/me/growth`, `mezo-rmhr`) is the dedicated home of the whole XP universe** — all-band skills + a 30-day quest/activity journal + computed badges/perks — with the Profil tab reduced to a single `GrowthSummaryCard` that taps through to it (§2; the growth **domain** — quests, activity log, XP economy — lives in [growth.md](growth.md)).
>
> **Placeholder strip (`mezo-lfw`):** the Phase-3 "vision" UI was removed so every screen reflects only what actually works. The four backend-backed views (`Profil`/`Cél`/`Súly`/`Alvás`) were cut to **backend-data only** — all narrative/decorative surfaces gone (Mezo "mit látunk" insight cards, "Hatások" factor cards, every decorative `ToolChipRow`, the Profil identity hero / identity-goal quote / PERMA bars / quick-setting rows / version footer, the `SettingsSheet` notification overview, the goal hero "Vége" projection stat, the `TrendCell` delta/ON-TRACK legs, the sleep weekly cells + chart target overlay + hero "Target" comparison). The two no-backend views (`Emberek`/`Tudás`) were **trimmed** to a working mock shell (kept: People log/grid/feed, Knowledge fact list; removed: credit hero, ritual/attention/relation-pattern cards, IDENT-5 footer, the "Gráf nézet hamarosan" stub, tool chips). Net: ~1.2k lines deleted, 13 view-local components dropped, the `WeightTrends`/`SleepTrends`/`PeopleSummary` mock shapes pruned. See §2/§4/§9.

## 1. Summary

The **Me** area is the user's personal hub: who they are (`Profil`), their long-term body-weight goal & strategy (`Cél`), their daily weight log + trend (`Súly`), their sleep log (`Alvás`), the people they track (`Emberek`), and a knowledge facts surface (`Tudás`). It exists as the "self & relationships" counterpart to the activity-focused tabs (Today, Train, Fuel, Insights).

## 2. User-facing behavior

`MeSection` (`frontend/src/features/me/pages/MeSection.tsx`) renders the shared **`AppHero`** (`@/features/progression/components/AppHero`, gamified-header slice `mezo-k7rn` — superseded `MeHead`) with a **`SubNavDropdown`** (`@/shared/ui/SubNavDropdown`, `items={ME_TABS}`, `accent="var(--lav-deep)"`, compact-header redesign `mezo-ugqb`) as its `utilities` prop, then an `<Outlet>`, and owns the `SettingsSheet` open/close state — the gear is the dropdown's `extraAction` (`{ label: 'Beállítások', icon: <Icon name="settings"/>, onSelect: () => setSettingsOpen(true) }`), a menu row after a separator inside the popover rather than a standalone icon-button in the row. **The old `MeOutletContext` (`{ openSettings: () => void }`) is retired (S7 Task 2)** — no page reads settings-open state via outlet context anymore, and **`MeSubNav.tsx`/`.np-pills` are deleted** (`mezo-ugqb`) — Me's sub-nav is the `ME_TABS` array (`pages/tabs.ts`) passed to the shared dropdown, not its own component/CSS. `AppHero` is domain-free platform chrome (avatar + XP ring/level badge, equipped title, label-less 🔥/⚡/🪙 counters) shared by all 5 sections — see [growth.md](growth.md) for the account-progression domain and [`_platform-design-system.md` §1a](_platform-design-system.md) for the `.apphero`/`SubNavDropdown` CSS families.

### `Profil` (`pages/ProfilePage.tsx`) — ✅ backend-only (post-`mezo-lfw`)
**Stripped to backend data only.** The old in-page "Profil" header + gear chip moved up to `MeSection`/`AppHero` (Napiv S7; `AppHero` since `mezo-k7rn`, see §3) — `ProfilePage` itself is now just **four content cards** in a `col gap-md`: **`MeBioRow`** (one-line biometrics — age · height · latest weight · body-fat%, `mezo-k7rn` — was `MeHead`'s `.t2`, now lives in page content since the uniform `AppHero` header carries no biometrics), **`GoalMiniCard`** (the goal mini-track, Task 3), the **`Biometria` card**, and the **`GrowthSummaryCard`** (`ProfilePage.tsx:25-28`). The gear (the `SubNavDropdown` `extraAction` menu row in the header popover, since `mezo-ugqb`) opens `SettingsSheet` (**a 3-option „Téma" selector via `useTheme`** — **Világos** / **Sötét** / **Cirkadián**, one `aria-pressed` card each, driving `setMode(mode)`; the single dark/light `toggle` was replaced by this in sleep slice C-éj, `mezo-d71m`, when the theme gained the `'auto'` circadian mode — default **Cirkadián** (`auto`), [`_platform-design-system.md` §2](_platform-design-system.md); the notification overview was already removed). Everything the old dashboard showed beyond this — concentric avatar, name/handle, member/streak/mesocycle stats, the identity-goal quote, "Aktív területek" PERMA bars, the Tudás/Emberek `EntryCard` shortcuts, quick-setting rows and the version footer — was **deleted** (`mezo-lfw`) along with their components (`ConcentricAvatar`/`ProfileStat`/`EntryCard`/`SettingsListRow`) and the `data/me.ts` mock file. Navigation to the other tabs is via the header's `SubNavDropdown` popover.

**Goal mini-track (`GoalMiniCard`, `components/GoalMiniCard.tsx`, Napiv S7 Task 3, `mezo-8141`) — first card on `Profil`.** A compact `.card.goalmini` button (whole surface taps through to `/me/goals`) reading `useGoal()`: a `🎯 {TRAJECTORY_LABEL} · {title}` row + a `{pct}% · {kg} hátra` readout (**or a plain `tartás` caption when `total === 0`** — a `maintain` goal whose start weight equals its target; `mezo-mifi` S8, replacing the misleading `0% · X kg hátra`), then the shared **`.track`/`.fill`/`.dot`/`.track-l`** progress rail (start→current→target, same idiom + `total !== 0` guard as the `Cél` hero below — a `maintain` goal with no target weight hides the track). Renders **`null`** while `pending` or when there is no active goal (no ghost — an absent card, not an empty one), so `Profil` silently drops to two cards until a goal exists.

**Biometria card + editor (G6, `mezo-06n`; re-skinned to Napiv `.biocard`, `mezo-8141` Task 4):** the first-class biometric surface — set once, the engine always computes the base-TDEE from it (and it's the **hard gate** on goal creation — see the `Cél` "Új cél" gate below, `GoalGate.tsx`). `<BiometricCard>` (`components/BiometricCard.tsx`) reads `useBiometricProfile()` and renders the `.card.biocard` idiom (spec §4.6): a `.bhd` header row (`Biometria` + a `szerkesztés ›` link in `--lav-deep`) over a `.biogrid` sex/height/age(derived from `birthDate`)/bodyFat/activity (× PAL) stat grid (Aktivitás spans both columns) + the **derived `.tdee` row** from `profile.tdeeBootstrap` (`≈{tdee} kcal/nap`, value in `--sage-deep`, **omitted when `tdeeBootstrap == null`** — i.e. before the goal is evaluated). When no profile exists yet (real-mode 404) the card shows an **empty-state prompt** ("Állítsd be a biometriád", `--wash-lav` surface + `--lav-deep` sparkle icon). Tapping `szerkesztés ›` (or the empty prompt) opens **`BiometricSheet`** (`BiometricSheet.tsx`) — a `Sheet` (children-as-function) editing Nem (segmented M/F), Magasság (cm), Születési dátum (date), Testzsír% (optional), Aktivitási szint (5 options → the PAL enum, default `MODERATE`) — the field markup lifted from the deleted `GoalPlannerPage` Step2, its selection/accent styling now `--lav-deep` (was the teal brand token). **Mentés** calls `useBiometricActions().upsert(body)` (which invalidates `['biometricProfile']` + `['goals']` so the active goal recomputes server-side) then `close()`s on success (the `EditGoalSheet` "mutation inside the sheet" pattern).

**`GrowthSummaryCard` (`mezo-rmhr`, `components/GrowthSummaryCard.tsx`) — the one progression card on Profil; re-skinned to Napiv `.growth2` (`mezo-8141` Task 4).** It **replaced** the three former progression cards (athletic radar + muscle levels + the LIFE `GrowthCard`) as part of the Growth-page split; **radar diagrams retired app-wide** (user decision — "engedjük el a diagramot"). It reads the dual-mode `useProgressionProfile()` (§3) and, when total XP > 0, renders a compact summary — a `.hd` row with a `{total} XP · {streakWeeks} hét sorozat` `.xp` chip (FE-summed `cumulativeXp` across all three bands, folding the streak text in), the **top-3 skills across all bands** as `.skl` progress rows (icon + HU name, a `.bar` fill driven by `SkillLevel.progressPct`, `Lv n` via `ATHLETIC_META`/`LIFE_SKILLS`/`MUSCLE_LABELS`), and a footer meta row (Atléta-szint `athleteLevel` · Fegyelem `disciplinePct`, values in `--lav-deep`, + "Megtakarítás (30 nap)" in `--sage-deep` when `savingsHuf30d > 0`). **The whole card is a `button`** → `navigate('/me/growth')`. When total XP is 0 it renders the **ghost** — the retired LIFE card's "Az élet is edzés." markup/copy verbatim (now on a `--wash-lav` surface), still a button opening `/me/growth` (an honest tap target before any XP). The growth **domain** (activity log, traits, savings, quest synergy) is documented in [growth.md](growth.md) — this card and the Growth page are only its Me consumers.

- `POST /api/sleep/screenshot` — **multipart** `photo` (jpeg/png/webp, ≤5 MB — `mezo.sleep-shot` → `SleepShotProperties`) → `SleepShotDraftResponse` (all extraction fields **nullable** — honest partial; only `confidence`+`needsReview` required). `SleepShotService.extract` (`SleepShotService.java:47-68`) runs **ONE** multimodal vision call over the screenshot, brace-window-parses the JSON (the meal/pantry pipeline idiom), zero-pads clock times (Sleep Cycle renders `0:42`), and sets `durationH = asleepMin/60`. **Deterministic `SleepShotDraftValidator`** (`SleepShotDraftValidator.java`, the LLM never grades itself) scores consistency — asleep ≤ in-bed, phase-sum within ±10% of in-bed, span ≈ in-bed ±15 min, HH:mm parse — as `confidence = passed/applicable`; `needsReview` when `confidence ≤ threshold` (boundary-inclusive, config `0.6`) **or** a key field (bedtime/wakeup/asleep) is missing.
- **Errors:** missing/oversized/unsupported photo → **400** `VALIDATION_INVALID_VALUE`/`photo`; unparseable model answer → **502** `SLEEP_SHOT_EXTRACT_FAILED`; no LLM port (companion off) → **503** `SLEEP_SHOT_LLM_UNAVAILABLE`. It reaches the model through the **sleep-owned `SleepShotLlm` port** (ADR 0012; companion supplies the `SleepShotLlmAdapter` — see [`companion.md`](companion.md) §5.3), injected via `ObjectProvider` so companion-off degrades to the clean 503. **Nothing persists** — the FE confirms the draft down the existing `POST /api/biometrics/sleep` (`source:'screenshot'`).

## 6. How to use it (consume)

Import the hook from `@/data/hooks` (never deeper) and destructure. Always ghost-guard real mode (empty arrays / `lastNight === undefined` on first paint).

```tsx
import { useGoal, useWeight, useSleep, usePeople } from '@/data/hooks'

function Example() {
  const { goal, goalResponse, linkedMesocycles, timeline, goalId } = useGoal()  // active goal + raw contract + plan-link timeline
  const { archive, remove, detachPlan } = useGoalActions()     // goal mutations (G4b; dual-mode)
  const { weightLog, weightTrends, logWeight } = useWeight()    // weight log + real EWMA rates
  const { sleepLog, lastNight, logSleep } = useSleep()          // sleepTrends was removed (mezo-lfw)
  const { people, mentions, logMention } = usePeople()          // summary/patterns were removed (mezo-lfw)

  if (!lastNight) return <p>Még nincs alvásadat.</p>   // canonical real-mode guard

  logWeight({ date: '2026-06-14', weightKg: 82.4, note: null })  // WeightLogInput
  logSleep({ date: '2026-06-14', bedtime: '23:10', wakeup: '06:40',
             durationH: 7.5, quality: 8, awakenings: 1, note: null })  // SleepLogInput
  // ...
}
```

Notes:
- **`useGoals()` no longer exists** — it was split into `useGoal()` (goal + `linkedMesocycles`) and `useWeight()` (`weightLog`/`weightTrends`/`logWeight`). A view that needs both calls both (e.g. `WeightPage` calls `useWeight` for the log/trend and `useGoal` for both `goal` [hero start/target/kind] and `goalResponse` [the chart's plan trajectory], `mezo-l82h`; `GoalsPage` calls `useGoal` for the hero and `useWeight` only for `weightTrends`). `useGoal()` is **read-only**; goal **creation** is a separate hook — **`useGoalCreation()`** (G4a, `{ submit, pending }`, drives `GoalPlannerPage`). Editing an *existing* goal in place is still G4b.
- `useProfile()` returns just `{ user }` (static, shared with Today + Fuel) since the Profil strip (`mezo-lfw`). The **biometric profile is a separate hook** — `useBiometricProfile()` (read, 404→null, `isComplete`) + `useBiometricActions()` (upsert), both from `@/data/hooks` (G6, §3) — consumed by `ProfilePage`'s Biometria card/sheet (and reused by the goal-creation hard gate).
- Mutations are **fire-and-forget** — `logWeight`/`logSleep`/`logMention` call `mutation.mutate(input)` and surface no promise or loading flag to the caller today.
- `WeightPage` tolerates an empty `weightLog` because `WeightTrendChart` shows the soft "Kevés mérés…" hint when `<2` points fall in the window (and renders actual-only with no goal), no `WeeklyWeightCard`s render (`groupByWeek([]) === []`), and the hero falls back to `goal?.currentWeight ?? 0` (goal may be `null` since `mezo-72d`); `SleepPage` must guard `lastNight` explicitly. `GoalsPage` guards `goal === null` up front (empty-state CTA).
- Run modes: mock (no backend) — `VITE_USE_MOCK=true pnpm dev`; real (default) — needs the backend on `:8090` with the `demodata` profile (owner seed makes login work; **weight/sleep start empty — they are NOT seeded**; the 5 People persons ARE seeded, with an honest empty mention feed — the demo feed is opt-in `demofixtures`).

## 9. Decisions, gotchas & deferred

- **Spec of record:** [`docs/superpowers/specs/2026-06-08-me-domain-sheets-design.md`](../superpowers/specs/2026-06-08-me-domain-sheets-design.md) (`mezo-k0i`). Key decisions: **bubble-up local state** (mirrors `useCheckins`, no store); **no date picker** in sheets (`date` is stamped to "today" by the mutator but is part of the durable contract because Phase 2 persists it); **voice CTA is decorative**; **EditGoal display-only**; **PersonDetail → PersonLog nesting**.
- **Backend was issue `mezo-v67`** — auth + weight + sleep + checkin shipped together as biometrics Slice A (changeset filenames carry `mezo-v67`).
- **Goal + biometric-profile backend was issue `mezo-2hp` (goal-system slice G1)** — spec [`docs/superpowers/specs/2026-06-18-goal-system-design.md`](../superpowers/specs/2026-06-18-goal-system-design.md). **G1 is foundation only:** the `goal`/`biometric_profile` aggregates + CRUD/lifecycle endpoints + the `useGoals → useGoal/useWeight` split, but **no** timeline/plan-links (G3), **no** TDEE/prescription engine (G4/G5). The `toGoal` adapter (`goalHooks.ts:12`) keeps `GoalsPage` on the legacy `Goal` shape; the writable `EditGoalSheet` + native `GoalResponse` rendering is G4.
- **Súly-tab IA split was issue `mezo-9sv` (goal-system slice G2)** — the daily weight log + trend chart + trend cells moved out of `GoalsPage` into a dedicated `WeightPage` at `/me/weight` (`router.tsx:95`, `ME_TABS` in `pages/tabs.ts`), leaving `Cél` goal/strategy-only. Pure frontend IA — **no backend or contract change** (both views reuse the G1 `useWeight`/`useGoal` hooks). `WeightPage`'s `useGoal()` call now drives the chart's plan trajectory + tolerance band via `goalResponse` (not just start/target reference lines); the `mezo-4nu` goal-coupling note is superseded.
- **Date-less owned aggregates** (`goal`, `biometric_profile`) extend `JpaRepository`, **not** `OwnedRepository` — the latter's `findAllOwned` needs an `e.date` field these tables lack. Bespoke `findBy…CreatedByAndDeletedFalse` finders instead.
- **Single-active goal invariant** is enforced **in the service** (`GoalService.activateGoal` archives every other active goal in the activate transaction), not by a DB constraint — by design, so reactivation/history stays flexible.
- **`mealToSleep` is a stub `0`** (`SleepLogMapper.java` constant + contract note) — the live Fuel→Sleep seam (§5.3).
- **Sleep goal + day-anchor was issue `mezo-dbsr`** — spec [`docs/superpowers/specs/2026-07-23-sleep-anchor-design.md`](../superpowers/specs/2026-07-23-sleep-anchor-design.md) (D1–D8; Matthew Walker "regularity is king"). Locked decisions, each stated once: **D1/D4 — the sleep goal is the single wake/bed source**; the derivation (WAKE→bed = wake − target; BED→wake = bed + target, mod 24h) lives in exactly one place per layer — backend `SleepAnchorResolver.derive`, FE `deriveSleepTimes` — and the **ungated `SleepAnchorPort`** lets Habit/Fuel resolve it even with the `/api/sleep/goal` surface off. **D2 — regularity is a ±band (default 15 min) circular window over the last 14 nights, anchored to the latest log, not "today"**; strict ±15 is only the score, while the `wake_on_time`/`bed_on_time` habits keep their 45-min config window (`mezo.habit.wake-window-min`). **D5 — `sleep_log` is enriched additively** (nullable in-bed + phase minutes + source) so manual rows stay sparse and slice B (Sleep Cycle screenshot) lands into the same model + the same POST path. **D6 — efficiency = asleep ÷ in-bed, target ≥ 85%** (prefers `inBedMin`, falls back to the bedtime→wakeup span). **GET never 404s** — a config-default ghost is the honest pre-save anchor (never a mock leak in real mode). The two scores (`regularityScore`/`efficiencyPct`) are **FE-pure** (`features/me/logic/sleepStats.ts`) — the backend has no consumer. Gated on `mezo.feature.sleep-goal.enabled`; config in `mezo.sleep.*` (`SleepGoalProperties`).
- **Phase 3 (companion) CONSUMES this domain, not extends it — ✅ wired at V0.3, deepened at V0.5:** the `ContextSnapshotAssembler` ([`companion.md`](companion.md) §5.5) reads the goal + prescription + day-planner fields, `biometric_profile`, `WeightTrendService.computeTrend` and the latest `sleep_log` read-only into every chat turn's system prompt; V0.3 added the `SleepLogRepository.findFirstByCreatedByAndDeletedFalseOrderByDateDesc` finder for it. **V0.5 (chat tools)** added the `SleepLogRepository` since-date finder (for `get_sleep`) and extracted the prescription-segment picker to a shared static `GoalPrescriptionJson.currentSegment` (used by the snapshot AND `get_goal_progress`); `get_weight_trend`/`get_goal_progress` reuse `computeTrend` unchanged. **Proactive B1.2 (`mezo-h4wp.2`)** added a third read-only `SleepLogRepository` finder — an `existsBy…DateGreaterThanEqualAndCreatedAtAfter` staleness probe the morning-briefing read path uses to decide whether last night's sleep arrived after the briefing was generated ([`proactive.md`](proactive.md)). **Proactive P1 (`mezo-h4wp.7`)** added a read-only window finder on BOTH repos — `WeightLogRepository.findByCreatedByAndDeletedFalseAndDateGreaterThanEqualOrderByDateDesc` (the weight sibling of the existing sleep one), consumed by the shared `MetricWindowEvaluator` (extracted at **P2 `mezo-h4wp.8`** so P1 validation + P2 experiment outcome share the same weight/sleep/training window-vs-baseline reads). Coupling stays one-directional (companion/proactive → me), no Me-side behavior change. The `mezo-lfw`-removed decorative `ToolChipRow`s were the mock preview of exactly this — real tool chips landed via the companion chat surface at V0.5, not via Me views.
- **`SleepEntry` type over-promises nullability:** the domain type claims `bedtime`/`duration`/… are always present, but the contract/DB allow null; `sleepApi.list`/`log` use an explicit `as Promise<SleepEntry[]>` cast (`biometricsApi.ts:28-38`) flagged "until normalized (see mezo bd issue)." A latent FE↔BE type mismatch to fix.
- **`OwnedRepository.findAllOwned` requires a `date` field** (JPQL `order by e.date asc`); both weight/sleep have one. `useSleep.lastNight` silently depends on this ordering — a date-less owned entity would need its own finder.
- **Belt-and-braces soft delete:** `findAllOwned` filters `deleted=false` AND each entity carries `@SQLRestriction` — intentional, keep both.
- **People backend was issue `mezo-t16y.2` (Phase-2 Slice E)** — plan [`2026-07-04-people-slice-e-plan.md`](../superpowers/plans/2026-07-04-people-slice-e-plan.md). Key decisions: **(1) persons seed under plain `demodata`** (deliberate exception to "demo content → demofixtures": the IDENT-5 five are the owner's real inner circle, and v1 has no person-create UI — an unseeded deploy would leave the tab unusable; the mention **feed** fixture IS `demofixtures`-only, with now-relative timestamps); **(2) honest derived stats** — `mentionCount`/`mentionsThisWeek`/`lastMentionedAt` are computed from live mention rows, never seeded (the mock's `84` would be a lie), while the narrative person fields (`knownFacts`/`ties`/`affectTrend`/notes/cadence) ARE seeded owner-curated data — a *computed* affect trend belongs to the pattern-engine epic; **(3) enums as DB CHECKs** (house style); **(4) mentions do NOT feed the companion snapshot** (proactive epic); **(5) display labels FE-derived** (contract carries `ts`/`lastMentionedAt` only); **(6) v1 write path is the chip sheet** — server stamps `source='chip'`/`ts`/`flagged=false`; the `flagged` filter shows only fixture rows until a flagging surface ships.
