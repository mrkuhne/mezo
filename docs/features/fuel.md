# Fuel (Nutrition) — Feature Documentation

> The `/fuel` tab — meal pacing, supplement stack/protocol, pantry (Kamra), recipes, and a weekly fuel rhythm. **Status: 🔶 mostly mock** (Phase 1 FE done; Phase 2 backend **Slice C · Fuel — Pantry (Kamra), Recipes (Receptek), Meal-logging (Mai), Medication (Gyógyszer) AND Stack/Protocol sub-slices now backend-backed**, the rest of Fuel still mock-only; AI scoring/replan/import is 🟣 Phase-3-planned, simulated client-side today). The Pantry inventory (`usePantry`/`usePantryActions`, mezo-9xu) is real dual-mode CRUD against `/api/pantry`; the scrape-feed (`imports`) and `suggestions` stay mock/deferred. **Recipes (`useRecipes`/`useRecipeActions`, mezo-lns)** are real dual-mode CRUD against `/api/recipe` — a `recipe` + `recipe_ingredient` aggregate whose lines reference `pantry_item`, with per-line `name`/`contribution` macros computed at read time. **Meal-logging (`useFuelDay`/`useMealActions`/`useRecipeLogs`, mezo-arb)** is real dual-mode against `/api/meal` + `/api/fuel/day/{date}` — a polymorphic `meal` + `meal_item` aggregate (items reference a `recipe` or a `pantry_item`, each with a per-line macro snapshot); the 4-dimension meal **score** (`meal.breakdown` jsonb) still ships NULL behind the pending-sparkle (Phase-3). The owner's prod pantry is seeded with a **147-item imported catalog** (`seed/pantry-catalog.json` → idempotent `@Profile("demodata")` `PantryCatalogLoader`, mezo-zza); that slice also extended `pantry_item` with preserved nutrition facts (`fiber_g/sugar_g/salt_g/saturated_fat_g`), a **`category` enum** (18 values: vegetables…supplement/other) and additional **`source`** vendors (lidl/nutriversum/herbahaz/nutrifit/decathlon).

## 1. Summary

Fuel is mezo's nutrition domain: five sub-views under the bottom-nav route `/fuel` covering today's meal/supplement **pacing** (`"Mai"`), the weekly **plan/rhythm** (`"Terv"`), an AI supplement-**protocol** builder (`"Stack"`), a **recipe** library (`"Receptek"`), and a **pantry/shelf** (`"Kamra"`). It exists to turn training + medication context (notably the **Retatrutide** appetite-cycle) into appetite-aware, time-boxed nutrition guidance.

**Status per layer:**
- **FE mock:** ✅ complete — all five views, all sheets return static mock data synchronously; the Fuel hooks in `frontend/src/data/hooks.ts` are mock-only **except** the Pantry, Recipes and Meal-logging hooks (`usePantry`/`usePantryActions`, `useRecipes`/`useRecipeActions`, `useFuelDay`/`useMealActions`/`useRecipeLogs`), which are dual-mode (mock returns the static seed synchronously via `initialData`).
- **FE real:** 🔶 **Pantry + Recipes + Meal-logging + Medication + Stack/Protocol** — `usePantry`/`usePantryActions` (`data/fuel/pantryApi.ts`, `/api/pantry`), `useRecipes`/`useRecipeActions` (`data/fuel/recipeApi.ts`, `/api/recipe`, mezo-lns), `useFuelDay`/`useMealActions`/`useWaterActions`/`useRecipeLogs` (`data/fuel/mealApi.ts`, `/api/fuel/day/{date}` + `/api/meal` + `/api/water-log` — the last is the `+víz` quick-log, mezo-0z5), `useMedication`/`useMedicationActions` (`data/fuel/medicationApi.ts`, `/api/medication`, mezo-d94) and **`useStack`/`useProtocol` + `useStackActions`/`useProtocolActions`** (`data/fuel/fuelApi.ts` + `data/fuel/stackHooks.ts`, `/api/fuel/protocol` + `/api/fuel/intake`, mezo-09g) all branch on `isMockMode()` and hit a real API in real mode (the `useWeight`/`useDualQuery` dual-mode pattern). The **Mai timeline** is now dual-mode too — **`useFuelTimeline`** moved to `data/fuel/timelineHooks.ts` (Fuel P5, mezo-9ys): mock serves the hand-authored `fuelPlan.today` seed unchanged, real **composes** the live day via the pure `buildDayPlan` (`features/fuel/logic/buildDayPlan.ts`) — goal day-planner settings + the current-week prescription budget, the day's logged meals + recipe suggestions, the supplement protocol (anchor-aware `buildProtocol`) + intakes, and today's gym/sport/run blocks; ALL composed hooks are called unconditionally and only the return branches on `isMockMode()`. **The Terv weekly plan is dual-mode too — `useFuelWeek` moved to `data/fuel/fuelWeekHooks.ts` (Fuel P4, mezo-kpo):** mock serves the `fuelWeek.ts` seeds unchanged; real composes the live week (gym days from `useTrain()`'s derived schedule, volleyball from Train's sport schedule, the Reta strip from `useMedication().cycle.week`, weekly stats from the new `GET /api/fuel/week/{start}` rollup) and returns honest-empty for the surfaces with no real source yet (`patterns`/`weeklySupplements` `[]`, `weeklyNote`/`supplementsAdherence` null); gym-time edits **write through to Train** via `useFuelWeekActions().saveGymSchedule`. Only `useReplanScenarios` still has no real path (mock-only const spread in `data/fuel/fuelReadHooks.ts`); `useStackRecommendations` is mode-aware (real mode returns `[]` — recommendations defer to Phase-3/P8, mezo-09g).
- **Backend:** 🔶 **Pantry + Recipes + Meal-logging + Medication + Stack/Protocol** — `backend/.../feature/pantry` (single-table `pantry_item`, Model B, `kind` discriminator; `pantry.yml`, mezo-9xu), `backend/.../feature/recipe` (`recipe` + `recipe_ingredient`, `recipe.yml`, mezo-lns), `backend/.../feature/meal` (`meal` + `meal_item` + **`water_log`**, `meal.yml`, `/api/fuel/day/{date}` + `/api/meal` + `/api/recipe/{id}/logs` + `POST/DELETE /api/water-log` — `FuelDayService.consumed.water` is now a real per-day Σ, no longer a fake `= targets.water` 100% ring, mezo-arb/mezo-0z5), `backend/.../feature/medication` (`medication` + `medication_dose`, `medication.yml`, `/api/medication` + `…/{id}/dose`, **derived cycle** via `MedicationCycleService`, mezo-d94) and **`backend/.../feature/fuel`** (the first Fuel-owned package: `protocol` + `protocol_item` + `supplement_intake` tables, `ProtocolService`/`IntakeService`, `FuelController` implementing the new generated `FuelApi`; `fuel.yml`, `/api/fuel/protocol` + `/api/fuel/intake`, mezo-09g) are all live. The rest of Fuel backend is ❌ none; `train`/`biometrics`/`auth`/`pantry`/`recipe`/`meal`/`medication`/`fuel` are the backend features so far. Some nutrition references remain forward-looking (`ProvenanceEnvelope.java:10`: "Fuel reuses this pattern for meal score" — meal `score`/`breakdown` jsonb still ships NULL behind the pending-sparkle, Phase-3).

The offset is the browser's local UTC offset for that wall-clock moment, appended rather than `.toISOString()`'d on purpose: Jackson's `OffsetDateTime` deserializer rejects a **zone-less** datetime (a naive `{date}T{time}:00` → 400 in real mode), and an offset matching local keeps `administeredAt.toLocalDate()` (the backend's day key, `MedicationService.logDose`) equal to the **chosen** `date` — a `.toISOString()` would shift local-midnight to UTC and roll the date back a day in a `+offset` timezone, corrupting the cycle's day math. ## 4. Data model & API

**Most Fuel types are still FE-only TypeScript in `frontend/src/data/types.ts`** (no tables/DTOs/endpoints). **Backend-backed exceptions:** Pantry (`/api/pantry`, `pantry_item`, mezo-9xu), **Recipes (`/api/recipe`, `recipe` + `recipe_ingredient`, mezo-lns)**, **Meal logging (`/api/fuel/day/{date}` + `/api/meal[/{id}]` + `GET /api/recipe/{id}/logs` + `POST/DELETE /api/water-log`, `meal` + `meal_item` + `water_log`, mezo-arb + mezo-0z5)**, **Medication (`/api/medication` + `…/{id}/dose`, `medication` + `medication_dose`, mezo-d94)**, and **Stack/Protocol (`GET/POST /api/fuel/protocol` + `GET/POST /api/fuel/intake` + `DELETE /api/fuel/intake/entry/{id}`, `protocol` + `protocol_item` + `supplement_intake`, mezo-09g)**.

**Stack / Protocol contract (mezo-09g) — the FIRST Fuel-owned `fuel.yml` fragment.** The contract lives in **`api/feature/fuel/fuel.yml`** → merged `api/openapi.yml` → FE types in `frontend/src/data/_client/api.gen.ts` (`Protocol{Response,ViewResponse,ActivateRequest,HistoryEntry}`, `Intake{Request,Response,ListResponse}`) + BE `FuelApi`/`api.dto`. Endpoints (controller `FuelController` implements the generated `FuelApi`):
- **`GET /api/fuel/protocol`** (`getProtocol`) → `ProtocolViewResponse` (`{ active?, history[] }`) — `active` is **absent when no protocol exists** (honest-empty; the FE maps that to the version-0 ghost).
- **`POST /api/fuel/protocol`** (`activateProtocol`, 201) — activates a new version from `ProtocolActivateRequest {selectedPantryItemIds[≥1], reason?}`; the previous active row is **superseded** and the version bumped `v+1` (`ProtocolService`; confidence from `mezo.fuel.protocol.default-confidence` config).
- **`GET /api/fuel/intake/{date}`** (`listIntakes`) → `IntakeListResponse` — the day's supplement intakes (feeds `useStack`'s `taken` derivation; the P5 timeline reuses this read).
- **`POST /api/fuel/intake`** (`logIntake`, 201) — logs one intake (`IntakeRequest {pantryItemId, takenAt?, dose?, slotKey?, note?}`; the dose snapshot defaults from the pantry item. The **FE always sends an offset-bearing `takenAt`** (browser wall-clock + local UTC offset, via `nowOffsetIso`/`offsetIso` in `shared/lib/dates.ts` — same rationale as `LogDoseSheet`'s `toOffsetIso` for medication doses) so the server's day key `takenDate = takenAt.toLocalDate()` matches the browser's calendar day; the server's UTC-now default is a fallback for API-direct callers only).
- **`DELETE /api/fuel/intake/entry/{id}`** (`deleteIntake`, 204) — soft-deletes an intake entry (the `entry` path segment **avoids the `/{date}` path-param collision** on `/api/fuel/intake`).

Three **owned** tables (`202607021300_mezo-09g_create_protocol_supplement_intake.sql`): **`protocol`** (versioned selection + metadata ONLY — `version`, `built_at`, `status active|superseded`, `confidence`, `last_replan_reason`; the timing **slots stay FE-computed** by the deterministic `buildProtocol`, per the fuel.yml note — activating supersedes the prior active row and writes `v+1`); **`protocol_item`** (the join → `pantry_item` — one row per selected supplement, `protocol_id` FK + `pantry_item_id` FK, so a protocol's selection is a real relation, not a jsonb blob); **`supplement_intake`** (an **append-only taken-ledger** FK-ing `pantry_item` — `pantry_item_id`, `taken_at timestamptz` + a derived `taken_date date` for the day-key index, a `dose` snapshot, optional `slot_key`/`note`; the `medication_dose` ledger precedent). All three carry `created_by` (app-level ownership filter) + `is_deleted` soft-delete.

**Medication contract (mezo-d94).** The contract lives in `api/feature/medication/medication.yml` → merged `api/openapi.yml` → FE types in `frontend/src/data/_client/api.gen.ts` (`Medication{Request,Response,DayResponse,DoseRequest,DoseResponse}`, `MedicationCycle{Config,Response,Cell}`, `MedicationPhase`) + BE `MedicationApi`/`api.dto`. Endpoints (controller `MedicationController` implements the generated `MedicationApi`):
- **`GET /api/medication`** (`getMedicationDay`) → `MedicationDayResponse` (`{ medication, cycle, recentDoses }`) — feeds the whole Gyógyszer tab in one round-trip; `cycle`/`recentDoses` are **derived/queried**, never client-supplied.
- **`PUT /api/medication/{id}`** (`updateMedication`) → edits the owned definition + cycle config (`MedicationRequest` → `MedicationResponse`).
- **`POST /api/medication/{id}/dose`** (`logDose`, 201) — logs one actual injection (`MedicationDoseRequest` `{ administeredAt?, dose, note? }` → `MedicationDoseResponse`); a null `administeredAt` defaults server-side to now.
- **`DELETE /api/medication/{id}/dose/{doseId}`** (`deleteDose`, 204) — soft-deletes a mis-entered dose.

Two **owned** tables (`202606261200_mezo-d94_create_medication.sql`): **`medication`** (the drug master — `name`, `active_ingredient`, `route`, `cadence`, `default_dose`, `dose_unit`, `is_active`, + a `cycle jsonb NOT NULL` envelope = `cycleLengthDays` + a `phases[]` of `{key:peak|stable|trough, fromDay, toDay, label}`, mapped via `@JdbcTypeCode(SqlTypes.JSON)` onto the typed `MedicationCycleJson` record — the Train provenance-envelope pattern); **`medication_dose`** (an append-only injection ledger — `medication_id` FK `ON DELETE RESTRICT`, `administered_at timestamptz` + a derived `administered_date date` for the cycle lookup index, `dose`, `note`). Both carry `created_by` (app-level ownership filter) + `is_deleted` soft-delete. The **cycle is derived, not stored**: `MedicationCycleService.derive(userId, med, onDate)` finds the newest dose at-or-before `onDate` and projects it — `retaDay = days-since-last-dose + 1` (1-based, **clamped** to `cycleLengthDays`), phase from the config `phases[]`, a `week[]` of `MedicationCycleCell`s with the current cell flagged; **no dose → honest-zero** (`retaDay 0`, null `lastDoseAt`, ghost week, never a fabricated day). The Meal contract lives in `api/feature/meal/meal.yml` → merged → FE types in `frontend/src/data/_client/api.gen.ts` (`Meal{Request,Response}`, `MealItem{Request,Response}`, `FuelDayResponse`, `Macros`, `MacroSet`, `MealScore`, `RecipeLog{Response,ListResponse}`, `WaterLog{Request,Response}`) + BE `MealApi`/`api.dto`. `meal_item` is **polymorphic** — a plain-UUID FK to `recipe` OR `pantry_item` (`ON DELETE RESTRICT`) + a `source` discriminator + an exactly-one-of CHECK + a frozen per-basis macro snapshot; the meal `score`/`breakdown` jsonb ships **NULL** behind the pending-sparkle placeholder (4-dimension scoring is Phase-3). Macros are computed at read (`factor = amount/snapshot_per`, whole-number HALF_UP round; meal = Σ line contributions; day `consumed` = Σ meal macros vs config `nutrition_targets`). **Water logging (mezo-0z5)** rides the same `meal.yml` fragment + `MealController`: **`POST /api/water-log`** (`WaterLogRequest {date?, amountMl}` → 201, `date` defaults to server-today) appends one owned `water_log` row and **`DELETE /api/water-log/{id}`** soft-deletes it (the weight/sleep-log precedent — smallest owned single-row aggregate: `created_by`, `is_deleted`, `log_date`, `amount_ml`; migration `202607021200_mezo-0z5_create_water_log.sql`). `FuelDayService.consumed.water` = the day's Σ `amount_ml` (was hardcoded `= targets.water`), surfaced through the unchanged `FuelDayResponse.consumed.water` — goal/Reta-aware water **targets** stay a config constant (deferred). **The 7-day rollup (Fuel P4, mezo-kpo) rides the same `meal.yml` fragment: `GET /api/fuel/week/{start}`** → `FuelWeekResponse { start, days[7] }`, each `FuelDayRollup { date, targets, consumed }` (no meal bodies — `FuelDayService.getWeek` loops the per-day consumed/water rollup) — feeds the Terv weekly stats and is the designated server aggregate for the Insights Weekly review (Phase-2 roadmap D′). The Recipe contract lives in `api/feature/recipe/recipe.yml` → merged `api/openapi.yml` → FE types in `frontend/src/data/_client/api.gen.ts` (`Recipe{Request,Response,ListResponse,IngredientRequest,IngredientResponse,Macros,Contribution,MezoFit}`) + BE `RecipeApi`/`api.dto`. Endpoints: `GET /api/recipe` (list), `GET /api/recipe/{id}`, `POST /api/recipe`, `PUT /api/recipe/{id}` (full-replace), `DELETE /api/recipe/{id}` (soft-delete cascade). `recipe_ingredient` references `pantry_item` by a plain-UUID FK (`ON DELETE RESTRICT`) plus a per-basis macro snapshot; macros are computed at read time (contribution = `amount/snapshot_per × snapshot macros`; whole-recipe = Σ contributions). Key FE shapes:

- **`FuelSlot`** (`types.ts:19-30`): one timeline entry — `kind: FuelKind` (`'wake'|'meal'|'midday'|'snack'|'preworkout'|'workout'|'sport'|'evening'`), `state: 'done'|'now'|'pending'`, optional `mealName`/`mezoNote`/`windowTip`, macros, `items: SlotItem[]` (`SlotItem` = `{type:'supplement', refId, label, done, primary?, note?}`).
- **`FuelPlanToday`** (`types.ts:31-36`): `workout`/`volleyball` blocks, `bedtime`/`kitchenClose`/`caffeineCutoff`, `slots[]`.
- **`MealDimension`** discriminated union (`types.ts:39-45`) — the four weighted scoring dimensions: `MacroDimension` (macroRatio/macroTargets/kcalShareOfDay), `MicroDimension` (`micros[]` with `MicroStatus 'good'|'ok'|'low'`), `NovaDimension` (food-processing class `NovaGroup 1|2|3|4`), `ContextDimension`. All share `MealDimensionBase` (label/weight/score/color/detail).
- **`MealBreakdown`** (`types.ts:46-52`): `confidence`, `summary`, `dimensions[]`, `improve[]`, `tools[]` (`ToolType 'read'|'compute'|'write'` — the transparency list). **This is the natural jsonb envelope** when Fuel lands (the `MealBreakdown` ⇒ meal `score` typed-jsonb mapping reuses Train's `ProvenanceEnvelope` pattern).
- **`FuelMeal`** (`types.ts`), **`FuelDay`**: `FuelDay` = `targets`/`consumed` (`MacroSet` incl. `water`), `meals[]`, `pacing`, `micronutrients`, `supplements` (`FuelSummary`). Since mezo-arb `FuelMeal` carries structured **`mealItems: MealItemLine[]`** (polymorphic logged lines — `{ source: 'recipe'|'pantry', refId, amount, unit, name, contribution{kcal,p,c,f}, nova? }`; `refId === recipeId|pantryItemId`, mirrors `RecipeIngredientLine`) plus real `loggedAt`/`mealDate`; the legacy free-text `items: string[]` is **kept** (the mock score-sheet seeds still read it). **`MealInput`** (`{ slot: MealSlot, loggedAt?, title?, items: MealInputItem[] }`) is the capture-sheet save payload → `mealApi.toRequest` → `MealRequest`. `MealSlot = 'breakfast'|'lunch'|'dinner'|'snack'`.
- **`SupplementStashItem`** (`types.ts`): `type 'supplement'|'stimulant'|'medication'`, category, dose, form, stock, protocol, timing, taken, `caffeine?`. Since **mezo-1za9** it also carries the same **nutrition + commerce** facts as `Ingredient` (`source?`, `per?`, `unit?`, `macros?`, `price?`, `priceUnit?`, `pkg?`, `micros?`, `nova?`, `fiberG?/sugarG?/saltG?/saturatedFatG?`) — all optional because pure dose/protocol items (many stim/med) have none. This closes the gap where the backend `SupplementStashResponse` dropped a supplement's macros/price so the Kamra card/detail showed nothing for it.
- **`Protocol`** (`types.ts:75-79`): `version`/`builtAt`/`source`/`status`/`itemCount`/`confidence`/`lastReplanReason`/`history[]`.
- **Pantry:** `Ingredient` (`stashRefId?`, `nova`, `micros`, `source: PantrySourceKey`), `PantryImport`/`PantrySuggestion`, unified **`PantryItem`** (`kind 'food'|'supplement'|'stim'|'med'`).
- **Recipe** (`types.ts`): `Recipe` with `ingredients: RecipeIngredientLine[]` (`{refId, amount, unit, note?, name?, contribution?}` — `refId === pantryItemId`, `name`/`contribution` server-computed), `macros`, `novaDominant`, `mezoFit.score: number | null`, `starred`, derived `recentLogs?`/`templateBreakdown?` (mock-only). **`RecipeInput`** is the editor save payload (`ingredients[].pantryItemId` lines) → `recipeApi.toRequest` → `RecipeRequest`. `RecipeLog` is the per-meal log shape on `recentLogs`.
- **Weekly:** `RetaPhase`/`RetaDayCell`, `GymScheduleDay`, `WeeklySupplementRow`, `RecurringPattern`, `ReplanScenario`/`ReplanCascade` (cascade `system: 'Fuel'|'Train'|'Sleep'|'Insights'`, `types.ts:304`), `StackRecommendation`.
- **Stack builder:** `ProtocolSlotItem`/`ProtocolSlotData`, `Reasoning` (`kind 'physiology'|'timing'|'interaction'|'sleep'`), `MealMatch`, `BuiltProtocol`.
- Supporting: `data/nova.ts` (`NovaGroup`, `NOVA_META`, `STATUS_COLOR`), `data/pantrySources.ts` (`PantrySourceKey = 'kifli.hu'|'myprotein.hu'|'tesco.hu'|'auchan.hu'|'manual'`).

**Where the rest of the backend plugs in (contract-first):** the `api/feature/fuel/fuel.yml` fragment **exists** (mezo-09g: `GET/POST /api/fuel/protocol` + `/api/fuel/intake`); the day/week reads landed in `api/feature/meal/meal.yml` (`/api/fuel/day/{date}` mezo-arb, `/api/fuel/week/{start}` mezo-kpo) and the timeline stayed client-composed (P5) — all merged `api/openapi.yml` → FE types in `frontend/src/data/_client/api.gen.ts` + BE `io.mrkuhne.mezo.api` interfaces/DTOs, per **[`docs/references/api_contract_conventions.md`](../references/api_contract_conventions.md)**. Remaining tables per the design spec: a fuel-timeline view, UUID PKs, `OwnedEntity` + `created_by`, soft delete, meal `score` as typed jsonb. (`pantry_item`/`recipe`/`meal`/`medication(_dose)`/`protocol`/`protocol_item`/`supplement_intake` are **already shipped**; `nutrition_targets` shipped as `@ConfigurationProperties`, not a table; the spec's `food_item` was superseded by `pantry_item`, Model B.)

**Goal day-planner settings (Fuel P5, mezo-9ys) ride the GOAL contract, not a Fuel table.** The Mai timeline's rhythm + budget inputs are three additive, nullable **goal** fields — `mealsPerDay` (3–6), `wakeTime`/`bedTime` (`HH:mm`) — on `GoalRequest`/`GoalResponse` (`api/feature/goal/goal.yml`) + `goal` columns (migration `202607021500_mezo-9ys_goal_planner_settings.sql`, CHECK `meals_per_day between 3 and 6`); FE defaults `4 / 06:00 / 23:00` when unset (`PLANNER_DEFAULTS`, `data/fuel/fuelConfig.ts`). They are **contract passthrough** — no goal-engine involvement (see [`goal-engine.md`](goal-engine.md) §4) — read by `useFuelTimeline` and written back by `EditGoalSheet`'s "Napi ritmus" via `useGoalActions().savePlanner` (the FE's first goal write-back; see [`me.md`](me.md) §2). The daily **budget** comes from the goal's `prescription` current-week segment (`kcal`/`proteinG`), carbs/fat derived by the fixed `FAT_KCAL_SHARE` split, with the config `FuelDay.targets` as the fallback (`deriveDailyBudget`, `features/fuel/logic/buildDayPlan.ts`). **This slice adds NO new Fuel endpoint** — the timeline is a client-side composition of already-real reads; the only backend touch is the three goal columns/contract fields.

## 5. Integrations

Fuel is the most cross-coupled mock domain. Each seam below names the **contract** (the type/shape that crosses):

- **Today → Fuel preview (Today consumes Fuel).** `useFuelPreview` (`data/today/todayHooks.ts`) now **composes `useFuelTimeline()`** (mezo-9ys) — it slices that same dual-mode plan's slots (the next 3 from `state==='now'`) + finds the next pending supplement stack for the Today screen's fuel card, so Today and the Fuel "Mai" view render the **same plan** in both modes (never diverge). Crossing type: `FuelSlot[]`; the `{ visible, nextStack }` shape is unchanged. (`useToday()` still also exposes the static `fuelToday` seed = `fuelPlan.today` for other Today reads.)
- **Today scenario → Fuel Mai (Mai consumes Today).** `FuelMaiPage` calls `useTodayScenario()` for `retaDay` and `useToday()` for `today.workoutType`. Crossing types: `TodayScenario`, `TodayMeta`. Since mezo-d94, `useTodayScenario` (`hooks.ts`) derives the `retaDay` **base** from the real medication cycle (`useMedication().cycle.retaDay`, falling back to `today.retaDay` when the cycle is 0/no-dose); mock mode keeps `today.retaDay`. The `?retaDay=` URL override (clamped 1..7) stays **top priority in both modes**, and `?day=` still drives `dayState` — so both knobs still change the Mai view's Reta bar.
- **Me/Goals → Fuel Stack (Stack consumes Me + Goals).** `FuelStackPage` reads `useProfile().user.weekInMeso` and `useGoal().linkedMesocycles` (the active mesocycle's `shortTitle`) for the `"Meso W{n}"` context cell — the consumer was updated from `useGoals` to `useGoal` in the G1 split (`mezo-2hp`). Crossing types: `UserMeta`, the linked-mesocycle map.
- **Train (gym/sport/run) → Fuel — two REAL paths since Fuel P4.** **(1) Terv weekly plan · Train's schedule, read AND written (Fuel P4, mezo-kpo):** in real mode `useFuelWeek` renders Train's derived gym week (`useTrain().gymSchedule.weeklyTimes` — the meso template's WHAT joined with the standalone slots' WHEN) + Train's sport-schedule volleyball sessions; `GymScheduleSheet` saves **write through** to Train via `useFuelWeekActions().saveGymSchedule` → `gymDaysToSlots` → `useTrain().saveGymSchedule` (`PUT /api/train/gym-schedule`, per the P0a ADR: Train owns the schedule, Fuel is a secondary editor). The old private mock copy (`fuelWeek.ts` `gymSchedule` + `today.ts` `volleyballSessions`) survives only as the mock seed. Crossing types: `GymScheduleDay` (out), `GymScheduleSlotInput[]` (in). **(2) Mai timeline · REAL Train blocks (Fuel P5, mezo-9ys):** `useFuelTimeline` composes today's gym/sport/run blocks from the live `useTrain().gymSchedule`/`sport.schedule` + `useRunning().activeRunningBlock` (reusing the Train views' own today-derivation — `runSessionsForDay`/`todayIdx`), feeding them as `PlannerBlock[]` into `buildDayPlan`, where meal windows snap around each block (pre −75m / post +45m) and the supplement pre-workout anchor derives from the first block (−40m). Crossing types: `GymSchedule`/`SportSchedule`/`RunningBlockResponse` → `PlannerBlock`.
- **Goal → Fuel timeline (budget + rhythm, Fuel P5, mezo-9ys).** `useFuelTimeline` reads the active goal's day-planner settings (`mealsPerDay`/`wakeTime`/`bedTime` — the goal-contract passthrough fields, §4) for the meal-window rhythm, and the goal's `prescription` **current-week segment** (`kcal`/`proteinG`, selected via `currentWeekOf` over the goal window) for the daily budget (`deriveDailyBudget`; config `FuelDay.targets` fallback; carbs/fat via the `FAT_KCAL_SHARE` split). Crossing type: `GoalResponse` (settings + `prescription.segments[]`). This wires the Goal→Fuel bridge the goal engine emitted-but-narrated-as-deferred ([`goal-engine.md`](goal-engine.md) §5) — the prescription's `kcal`/`proteinG` now feed a real consumer.
- **Recipes → Fuel timeline (slot suggestions, Fuel P5, mezo-9ys).** For each un-logged meal/snack window `buildDayPlan` ranks the recipe catalog (`useRecipes().recipes`): candidates of the window's slot category with per-serving kcal within **±20%** of the slot budget, ranked `|Δkcal| → starred → |Δprotein|`; the top pick becomes the slot's `suggestedRecipeId` + `mealName` (the `ajánlott` chip, tap → prefilled `LogMealSheet`). No fit → a budget-only slot. Crossing type: `Recipe[]` → `FuelSlot.suggestedRecipeId`.
- **Reta (medication) → cross-cutting.** `retaWeek`/`retaDay` is a Fuel-owned concept (the 7-day Retatrutide kinetic cycle: D1-2 Peak → D3-5 Stable → D6-7 Trough) but is surfaced on Today and across all Fuel sub-views. `retaDay` arrives via `useTodayScenario`, whose real-mode base is now **derived from the medication cycle** (`useMedication().cycle.retaDay`, mezo-d94) — the single FE broadcast point every Reta surface (Today, Mai `RetaPhaseBar`, Terv `RetaWeekStrip`) reads. Because `useTodayScenario` now issues a `['medication']` query, any test that renders one of those surfaces needs a `QueryClientProvider`.
- **Replan cascade → Sleep / Insights / Train (Fuel exposes the ripple model).** `ReplanScenario.cascades[].system` is typed `'Fuel'|'Train'|'Sleep'|'Insights'` (`types.ts:304`); `ReplanSheet` color-codes per system. This is the explicit "context change ripples across domains" model — all simulated prose in `data/fuel/fuelWeek.ts`.
- **Supplements ↔ Pantry (internal seam, now a real read composition — mezo-09g).** `useStack` no longer reads the `supplementsStash` const directly — it **composes `usePantry().stash`** (`stackHooks.ts:80-85`), the single source of truth, and only overlays each item's `taken` from the day's intakes. So in mock mode the stash is the seed via `pantryHooks.ts`, and in real mode it comes from `pantryApi.list()` — one source, no divergence (`supplementsStash` survives only as the mock seed behind `usePantry` + the intake-seed derivation). The Kamra still merges the same stash into pantry items, de-duping via `Ingredient.stashRefId` (`kamraItems.ts:17-18`).
- **Recipes ↔ FuelDay (internal seam).** `data/fuel/pantry.ts` imports `fuelDay` to derive recipe `recentLogs` from logged meals — so scoring a meal and viewing a recipe's log history share one source (`pantry.ts`). Note: since mezo-arb the mock `fuelDay.meals[].score` ships **NULL** (pending-sparkle), so the recipe seed's historical `recentLogs` scores are now sourced from a decoupled `recentLogScore` map in `data/fuel/pantry.ts` (nulling the day-view display score must not erase the recipe log history).
- **Meal logging → FuelDay + Recipes + Pantry (the real write seam, mezo-arb).** `useMealActions` (`data/fuel/fuelHooks.ts`) is the write path for logging/editing/deleting a meal. In real mode each write calls `mealApi` then invalidates **`['fuelDay']` + `['recipes']` + `['pantry']`** — logging a meal shifts a recipe's `recentLogs` (`RecipeLogsList`) and pantry usage, so all three caches must refetch. In mock mode it mutates the `['fuelDay', date]` cache in place and recomputes `consumed`. Crossing types: `MealInput` (in), `FuelMeal`/`FuelDayData` (out).
- **Supplement intake → Pantry + Stack (the real intake write seam, mezo-09g).** `useStackActions` (`data/fuel/stackHooks.ts`) is the write path for logging/undoing a supplement/stim intake (from the Stack timing slots' tap-to-log). The intake ledger FK-s `pantry_item` — an intake references a pantry supplement row by `pantryItemId`, and `useStack` reads that same pantry stash back to overlay `taken`, so **one intake log updates the Stack's taken-state directly** (no second store). In real mode a log/undo hits `POST /api/fuel/intake` / `DELETE /api/fuel/intake/entry/{id}` then invalidates `['fuelIntake', date]` → `useStack` refetch; mock mutates that cache in place. Crossing types: `pantryItemId` (in), `Intake[]` (out, → `taken`). The **protocol** write seam is `useProtocolActions().applyProtocol` (Stack "Bekapcsolás · ma") → `protocol_item` rows FK the same `pantry_item` selection. (The P5 fuel-timeline **now** reuses this `GET /api/fuel/intake/{date}` read + the anchor-aware `buildProtocol` slots to light up the supplement half of the merged agenda — mezo-9ys.)
- **Shared design primitives.** Fuel reuses `frontend/src/shared/ui/` heavily: `Sheet`, `StatCell`, `ProgressBar`, `Eyebrow`, `PageTitle`, `Chip`, `Icon`, `ScoreRing`, `RetaPhaseBar`, `ToolChipRow`, `SourceBadge`, `NovaDot`, `MacroRow`, `SafeMarkdown`, `Toggle` — all "Deep Current v2" tokens from `frontend/src/styles/prototype.css`.

## 6. How to use it (consume)

Import any of the nine Fuel hooks from the single boundary `@/data/hooks`:

```ts
import {
  useFuelDay, useMealActions, useRecipeLogs,
  useFuelTimeline, useStack, useProtocol,
  usePantry, useRecipes, useFuelWeek,
  useReplanScenarios, useStackRecommendations,
} from '@/data/hooks'

const { fuel } = useFuelDay()                       // FuelDay (composed, dual-mode): targets/consumed/meals/pacing/micronutrients/supplements
const { logMeal, updateMeal, deleteMeal } = useMealActions()  // write path: MealInput → POST/PUT/DELETE /api/meal (+ mock cache mutate)
const { logs } = useRecipeLogs(recipeId)            // RecipeLog[] for RecipeLogsList
const { plan, getScoredMeal } = useFuelTimeline()   // plan: FuelPlanToday; getScoredMeal(slot) → FuelMeal|null
const { stash } = useStack()                        // SupplementStashItem[]
const { protocol } = useProtocol()                  // Protocol
const { ingredients, stash, sources, categoryMeta, imports, suggestions } = usePantry()
const { recipes, sources, categoryMeta } = useRecipes() // ingredients come from usePantry() (mezo-yew)
const { title, retaWeek, gymSchedule, weeklySupplements, patterns, weeklyStats, volleyball, weeklyNote } = useFuelWeek() // FuelWeekView (dual-mode, P4)
const { saveGymSchedule } = useFuelWeekActions()    // GymScheduleDay[] → write-through to Train (P0a)
const { scenarios } = useReplanScenarios()          // ReplanScenario[]
const { recommendations } = useStackRecommendations()
```

Pure helpers to reuse: `buildProtocol(selectedIds, stash)` (`features/fuel/logic/buildProtocol.ts`), `buildKamraItems(ingredients, stash)` (`features/fuel/logic/kamraItems.ts`), `getScoredMeal(slot, meals)` (`data/fuel/fuel.ts`). ## 7. How to extend it

**To make Fuel real (Phase-2 Slice C), use the biometrics slice as the template** (`useWeight` + `frontend/src/data/me/biometricsApi.ts` are the proven pattern). Recipe:

1. **Contract-first** (**[`api_contract_conventions.md`](../references/api_contract_conventions.md)**): add `api/feature/fuel/fuel.yml` (endpoints + DTO schemas: FuelDay, MealBreakdown, PantryItem, Recipe, Protocol…). Merge: `cd api/generate && npm run generate:api`; regenerate FE types: `cd frontend && pnpm generate:api`; backend DTOs regenerate in `./mvnw generate-sources`.
2. **Backend** under `io.mrkuhne.mezo.feature.fuel/{controller,service,repository,entity,dto,mapper}` per **[`java_package_structure.md`](../references/java_package_structure.md)** + **[`spring_patterns.md`](../references/spring_patterns.md)**. Entities: `food_item`/`meal`/`meal_item`/`recipe`/`supplement_intake`/`medication(_dose)`/`nutrition_targets` + a fuel-timeline view. UUID PKs, `OwnedEntity` + `created_by`, `@SQLDelete`/`@SQLRestriction`. **Meal `score` → typed jsonb** via `@JdbcTypeCode(SqlTypes.JSON)` onto a typed record — reuse the Train `ProvenanceEnvelope` pattern (`MealBreakdown` is the natural envelope).
3. **Liquibase** (**[`liquibase_conventions.md`](../references/liquibase_conventions.md)**): changesets `{YYYYMMDDHHMM}_{bd-id}_{desc}.sql`, explicit constraint names, **seed data in Java `@Profile("demodata")`** (never SQL). Add new tables to the `ResetDatabase` TRUNCATE list and add a populator per aggregate (**[`integration_test_framework.md`](../references/integration_test_framework.md)**).
4. **FE client:** add `frontend/src/data/fuel/fuelApi.ts` mirroring `biometricsApi.ts` (`apiFetch`, `satisfies` on request bodies, contract types from `api.gen.ts`).
5. **Hooks:** rewrite the nine Fuel hooks in `hooks.ts` to branch `isMockMode() ? mockConst : fuelApi.x()` with `useQuery`/`useMutation` + `initialData` for synchronous mock parity — **copy `useWeight` (`weightHooks.ts:11`) exactly. Do not touch view/component signatures.**
6. **Config** (**[`configuration_conventions.md`](../references/configuration_conventions.md)**): scoring dimension weights, the kcal floor (hardcoded `2500` in `FuelMaiPage`), and `weeklyStats` factors must become `mezo.*` `@Validated` properties — never `@Value`, never literals.
7. **Tests** (**[`testing_standards.md`](../references/testing_standards.md)** + **[`integration_test_framework.md`](../references/integration_test_framework.md)**): integration-first (`@SpringBootTest` + Testcontainers PG, AssertJ, populators), an ownership-isolation test per owned entity. FE: add MSW handlers for real-mode hook tests; keep **both** mock and real modes green.
8. **Docs:** an **ADR** in `docs/decisions/` for the scoring/jsonb-envelope decision; update **[`docs/milestones/roadmap.md`](../milestones/roadmap.md)** when the slice lands.

**Within the mock layer** (no backend), the established recipes are: add a logged metric → extend the type in `data/types.ts` + the const in `data/fuel/fuel.ts`/`fuelWeek.ts`/`pantry.ts`; add a Stack reasoning branch → extend `buildProtocol.ts` (keep the empty-selection-yields-zero-slots invariant); add a pantry source → extend `PantrySourceKey` + `pantrySources` (`data/pantrySources.ts`); add a sub-tab → add a route child in `router.tsx`, a label in `FuelSubNav.tsx`, and a view in `pages/`.

## 9. Decisions, gotchas & deferred

- **Everything is mock; the "AI" is theater.** `buildProtocol` is deterministic; meal scores, NOVA classifications, improve-suggestions, replan cascades, and scrape previews are pre-written prose/fixtures. Real intelligence is **🟣 Phase 3** (Spring AI / pgvector / RAG), not in scope for Slice C — now mapped: companion epic `mezo-fnnq` ([roadmap](../superpowers/plans/2026-07-03-companion-roadmap.md), [design spec](../superpowers/specs/2026-07-03-phase3-companion-chat-design.md)); the Fuel-side AI items stay under the P8 umbrella `mezo-0h6w`, layered on the companion slices (LLM port V0.1, embeddings V2.x, patterns V3.x).
- **Inert mutations (shrinking).** Replan apply and Import `"Polcra"` still show UI feedback (phase change / local state) but **never persist**; **`"Mentés protokollként"` is deferred** (`· hamarosan`, disabled). **No longer inert:** (a) **Stack `"Bekapcsolás · ma"` now persists (mezo-09g)** — `useProtocolActions().applyProtocol` really activates a versioned `protocol` (mock `['protocol']` cache `v+1`, real `POST /api/fuel/protocol`), and the toast shows the **real** returned version (the old "read-only `version + 1` display" is gone); the Stack timing slots' **tap-to-log** intakes persist too (`useStackActions.logIntake`/`undoIntake` → `supplement_intake`). (b) Meal **logging** persists since mezo-arb — the `+ Mai étkezéshez` / `＋ Logolás` CTAs, the Mai ＋Log and the Today preview ＋Log all open `LogMealSheet` → `useMealActions().logMeal` (real `POST /api/meal` + dual-cache invalidate, or the mock `['fuelDay']` cache mutate). (c) Recipe-detail Csillag/Szerkesztés/Törlés have persisted since mezo-lns.
- **Hardcoded literals (deliberate prototype fidelity).** kcal floor `2500` (`FuelMaiPage`), `"conf 0.86"`, the import preview fixture `"Görög joghurt 10%"`. These become config/data in Slice C. (`RetaWeekStrip currentDay` derives from `useTodayScenario().retaDay` since mezo-d94; the week title `"Máj 18 – 24"` is mock-seed-only since P4 — real mode date-derives it.)
- **`recipes` is module-load-derived, not static** — couples `data/fuel/pantry.ts` → `data/fuel/fuel.ts` (`recentLogs`/`templateBreakdown` from `fuelDay.meals` + `recipeTemplateBreakdowns`, `pantry.ts:548-573`). The backend must reproduce this recipe↔log linkage server-side.
- **The Terv schedule copy is RECONCILED to Train (Fuel P4, mezo-kpo)** — real mode reads Train's derived gym week + sport schedule and writes gym times through to `PUT /api/train/gym-schedule` (P0a: Train owns the schedule, Fuel is a secondary editor); `fuelWeek.ts` `gymSchedule` + `today.ts` `volleyballSessions` survive as mock seeds only. **Known v1 limits (documented, accepted):** only day+time persist through the sheet — `type`/`active` are mesocycle-owned (an edit shows optimistically via the page's local override but meso truth wins on reload; toggling OFF persists correctly by dropping the slot); `duration` has no DB home → `DEFAULT_BLOCK_MIN` renders the grid block; the sheet copies the schedule on mount, so opening it before the Train queries resolve shows an empty list (resolves on next open).
- **Reta (Retatrutide) is now FIRST-CLASS (mezo-d94)** — a real `medication` + `medication_dose` aggregate, **NOT** a `pantry_item` row (the design spec's earlier "Reta in pantry" idea was superseded; see `docs/superpowers/specs/2026-06-26-fuel-medication-design.md`). The 7-day appetite cycle (D1-2 Peak → D3-5 Stable → D6-7 Trough) is **derived from the dose log** by `MedicationCycleService` (real) / the mock re-derivation in `medicationHooks.ts` — `retaDay = days-since-newest-dose + 1`, clamped to `cycleLengthDays`; a dose today re-anchors `retaDay` to 1. `retaDay` **broadcasts** through `useTodayScenario().retaDay` (real-mode base = `useMedication().cycle.retaDay`), the single FE point every Reta surface (Today, Mai `RetaPhaseBar`, Terv `RetaWeekStrip`) reads. The `?retaDay=` URL override **remains** top-priority in both modes (demos/tests); since P4 the weekly-plan `retaWeek` cells also derive from the medication cycle in real mode (`toRetaCells(cycle.week)` — the `fuelWeek.ts` cells are the mock seed).
- **`DAYS_HU` quirk** (`data/fuel/fuelWeek.ts:22`): `['H','K','Sz','Cs','P','Sz','V']` has a duplicate `'Sz'` (Szerda + Szombat) — index-based, matches the prototype.
- **`getScoredMeal` matches by `mealName` string, not id** — fragile if titles diverge between `fuelPlan.slots.mealName` and `fuelDay.meals.title`.
