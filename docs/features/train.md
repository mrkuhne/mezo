## 1. Summary

Train is the largest mezo domain: a six-tab area for planning and executing strength work and conditioning. It covers periodized **mesocycles** (`Mesociklus`), the active week's **gym split**, full-screen **workout execution** with per-set logging, an **exercise catalog + per-exercise records** explorer, **sport** (`R├Âplabda` ÔÇö volleyball, cross, and TRX, 0..n scheduled slots per day, `mezo-05v6`) scheduling/logging, and structured **interval running** (`Fut├ís`) plans. A `Mai` tab aggregates all three rhythms (gym + sport + running) onto one weekly agenda.

**Status per layer ÔÇö all Ôťů done:**

- **FE mock** (Phase 1): synchronous static fixtures, every tab renders without a backend.
- **FE real** (Phase 2): each read/write swapped to `/api/train/*` behind the same hooks; no signature change.
- **Backend** (Phase 2): full `feature/train/**` Spring Boot slice over Postgres ÔÇö mesocycles, workout execution, sport, exercise catalog/records, running.

**Phase-3-planned (­ččú, flagged throughout, do NOT assume live):** `niggleWarning`, PR detection (the current toast is a scripted demo), the live cross-load Ôćĺ volume recompute engine (both Sport and Running cross-load are static), `volumePerMuscle`/`volumeRecompute` provenance (seed-only), `shoulderLoadTrend`, sport `jumpCount`/`intensity`/`team`/`season` capture, gym-schedule **`duration`** (the gym-time **schedule is now live** ÔÇö see ┬ž4 `GymScheduleSlot`; only per-day duration stays out of scope), copy-week, and the active-workout **Voice** tool (still no UI at all, Phase 3). **The old inert `90s` timer chip is gone, not deferred** ÔÇö Napiv S5 (`mezo-8141`) replaced it with a real rest Live-Activity countdown island (┬ž2/┬ž9), whose 150s/90s duration is a **hardcoded** per-exercise-type heuristic (`restTimer.ts`), not a data field; a configurable per-exercise rest duration would need a new backend field (Phase 2 backlog candidate, not Phase 3).

## 2. User-facing behavior

The sub-nav (`TrainSubNav`, `frontend/src/features/train/pages/tabs.ts`) has six tabs: **`Mai`** (`/train`), **`Gym`** (`/train/gym`), **`Sport`** (`/train/sport`), **`Fut├ís`** (`/train/futas`), **`Gyakorlatok`** (`/train/exercises`), **`Mesociklusok`** (`/train/mesocycles`). Four full-screen sibling routes have no sub-nav: active workout (`/train/session`), the done-day review (`/train/review/:workoutId`, `WorkoutReviewPage`), mesocycle planner/builder (`/train/mesocycles/new`, `/train/mesocycles/:id`), and the running-block builder (`/train/futas/:id`). **Since Napiv S4 (`mezo-8141`) the tab label reads `Gym` (was `GYM`)** ÔÇö a copy-only change (the route/id are untouched) ÔÇö and `TrainSubNav` itself renders the new coral `.np-pills`/`.np-pill` pill nav rather than the generic `.subnav-item`/`.active` idiom other domains used to use; `.np-pill`'s active tint is parametric via the `--pill-accent` custom property (defaults to `var(--coral)`), reserved for Fuel (`--sage`, S6) and Me (`--lav`, S7) to adopt the same markup later (`docs/features/_platform-design-system.md`). **Every Train page now also opens with a `.pghead-np` page-head** (an `.over` eyebrow line + `h1`, optionally a `.pgact-np` pill action chip) in place of the generic `.page-header`/`Eyebrow`/`PageTitle` idiom described in the design-system doc.

#### Set recommendation engine ÔÇö hypertrophy drive (`mezo-dhdr`)
`SetRecommendationService` (`feature/train/service/SetRecommendationService.java`) computes the per-set targets that `getToday` attaches as `prescribedSets`. It applies **double progression** on the exercise's **top working set** of the most-recent completed instance: reps `Ôëą rep_max` Ôćĺ bump the load by the type increment; reps `< rep_min` Ôćĺ drop by the increment; reps **in range** Ôćĺ hold the load (the athlete earns reps up toward `rep_max` first). On the **first session** (no history) it seeds from `anchor_weight_kg`, or leaves the target weight **null** when no anchor is set. **Plyometric / bodyweight exercises are weightless** ÔÇö the engine returns null target weights (reps/RIR only), which the FE renders as a reps-only stepper. Warmup sets are generated by ramping the working target down through the configured `warmupRamp` ÔÇö and are **emitted even when the base weight is unknown** (first session, no anchor; `mezo-eerq`): their target weight is simply null, keeping the prescribed list aligned with the recipe's set count so the FE never loses the `B{n}` labels / RIR suppression on a first session. The method returns a `Prescription(sets, rationale)` record (`service/Prescription.java`) ÔÇö the set list plus the human-readable `rationale` shown in the active workout (replacing the old static hint). It reads **completed-instance history only**, so a live (open) session never feeds back into its own prescription Ôćĺ the recommendation is **stable across a mid-workout resume** (┬ž9).
- **Config** (`mezo.hypertrophy.*` Ôćĺ `HypertrophyProperties`, `feature/train/config/HypertrophyProperties.java`): `plateStep` (2.5 ÔÇö rounding granularity for computed kg), `defaultIncrement` (2.5 ÔÇö fallback), a per-type `increment` map (compound 5.0 / isolation 2.5), and the `warmupRamp` (`List<Ramp(pct, repsFactor)>`). `@Validated` record, wired in `application.yml` under `mezo.hypertrophy:` ÔÇö never `@Value`.
- **Switch:** the whole engine is gated by `mezo.feature.hypertrophy-drive.enabled` via the `HypertrophyDriveGate` `@ConditionalOnProperty` marker bean (`feature/train/HypertrophyDriveGate.java`; key constant `FeaturesConfiguration.HYPERTROPHY_DRIVE_SWITCH`). Off Ôćĺ `WorkoutService` skips the engine and returns null `prescribedSets`/`rationale`, so the FE logs ad-hoc.

#### Closing block ÔÇö fix z├ír├ís (`mezo-z2ul`)
`ClosingBlockService` (`feature/train/service/ClosingBlockService.java`) **lazily ensures the configured closing exercises at the END of every non-empty template day of the active meso** ÔÇö by default **Dead Hang** (2 ├Ś 45ÔÇô60 mp; weightless `plyo` type Ôćĺ reps-are-seconds, kg-stepper hidden on the FE) and **45┬░ Back Extension** (2 ├Ś 12ÔÇô15). #### Lazy auto-close of stale instances (`mezo-cd8s`)
`WorkoutAutoCloseService` (`feature/train/service/WorkoutAutoCloseService.java`) **settles abandoned workout instances** so a session left open on a past day never lingers as `active`. Invoked as the **first statement** of `WorkoutService.getToday` (before the closing-block ensure and the meso lookup, so stale instances settle even when there is no active meso). A SEPARATE `@Transactional` bean because `getToday` is a plain read and a same-class `@Transactional` self-invocation would bypass the proxy (mirrors `ClosingBlockService`). For each of the owner's `active` instances dated **strictly before today** (`WorkoutSessionRepository.findByCreatedByAndStatusAndDateBeforeAndTemplateSessionIdIsNotNull`): it becomes **`completed`** when it carries **Ôëą1 non-skipped logged set** (the work counts ÔÇö feeds `findDoneInstanceDates`, `lastWeekRefs` and the prescription engine), else **`skipped`** (a started-but-empty session never counts as done). **Idempotent** (a settled instance is no longer `active`, so a re-run is a no-op) and **always on** ÔÇö no feature gate, **no DDL/contract change**. After it runs, only a **today-dated** instance can still be `active`, so the open-instance resume lookup below can never resurrect last week's abandoned session. IT: `WorkoutAutoCloseIT` (past-active-with-set Ôćĺ completed / past-empty Ôćĺ skipped / today-active Ôćĺ untouched).

#### Done-state = completed status + workout-detail review (`mezo-cd8s`)
The gym "done that day" signal changed from **Ôëą1 non-skipped logged set** to **`status = 'completed'`** (an explicitly finished ÔÇö or lazily auto-closed ÔÇö instance): `WorkoutSessionRepository.findDoneInstanceDates` (Ôćĺ `weekDoneDates`) and `findDoneInstancesBetween` (Ôćĺ Insights `listWorkouts` + the companion `get_recent_workouts` tool, ┬ž5) both switched their predicate to `s.status = 'completed'`, so a started-but-unclosed instance with logged sets is **no longer** counted (the auto-close settles stale ones, above). `lastWeekRefs` and `SetRecommendationService` already read completed-only history ÔÇö unchanged. The same shift flows to every consumer of `findDoneInstanceDates` (QuestEvaluator, `TrainingCommitmentCalculator`/discipline trait, Insights weekly counts, companion tools) ÔÇö the agreed semantic root of the four fixes.
- **`completedWorkout` on `getToday`** ÔÇö after the auto-close, `WorkoutService.getToday` resolves the most recent `completed` instance of today's template day **dated today** (`findFirstByCreatedByAndTemplateSessionIdAndStatusAndDateOrderByCreatedAtDesc`) onto `WorkoutTodayResponse.completedWorkout` (null when none). - **`GET /api/train/workouts/{id}` Ôćĺ `WorkoutDetailResponse`** (`WorkoutService.getWorkoutDetail`, a pure read, no `@Transactional`): the done-day review data source (weekly rows + Mai "Megn├ęzem"). Reads additionally **heal** a stale/invalid stored value (null, `<1`, or `>weeks`) via `RunningService.toResponse`, so a block written before this (e.g. the legacy `currentWeek 0` that rendered "az aktu├ílis h├ęt (0) nincs a tervben" for a plan starting today) self-corrects on the next fetch. Weeks are 1-indexed; the `RunWeekView` `weeks.find(w => w.weekNumber === currentWeek)` therefore always resolves for an in-range plan. Mock mode mirrors the derivation via `currentWeekOf` (`shared/lib/dates.ts`). _Known limitation:_ like mesocycles, the stored value only re-derives on write/activate; the read-time heal only fires for out-of-range values, so a valid block does not auto-advance week-to-week without a write.
- **DTOs:** `RunningBlockResponse`/`RunningBlockUpsertRequest`, `RunningBlockStructureDto`, `RunWeek`, `RunPrescribedSession`, `RunSegment`, `RpeTarget`, `RunSessionLogResponse` (now also carries an optional `levelUp` Ôćĺ `LevelUpResult`, the same schema as the gym finish)/`RunSessionLogRequest`.
- **Seed:** `RunningSeedData` is `@Profile("demofixtures")` (same as `TrainSeedData`, since `mezo-uuv`) ÔÇö a plain `demodata` (prod) app starts **clean** with no running blocks; the 3 demo blocks (active/planned/archived) require the opt-in `demodata,demofixtures` profile combo.

## 6. How to use it (consume)

Import from the single boundary `@/data/hooks`; never reach into `trainHooks.ts`/`runningHooks.ts` directly.

```tsx
import { useTrain, useRunning } from '@/data/hooks'

function MyWidget() {
  const {
    mesocycles, activeMeso, workout, gymSchedule, gymSlots, sport,
    exerciseLibrary, exerciseRecords, workoutPending, todaySession,
    // imperative writes:
    createMesocycle, activateMesocycle, closeMesocycle, saveDayExercises,
    startWorkout, logSet, saveWorkoutFeedback, finishWorkout,
    logSportSession, saveSportSchedule, saveGymSchedule,
    createCatalogExercise, updateCatalogExercise, deleteCatalogExercise, setExerciseVideo,
  } = useTrain()

  const {
    runningBlocks, activeRunningBlock, runSessions, runningPending,
    saveRunningBlock, activateRunningBlock, closeRunningBlock,
    deleteRunningBlock, logRunSession,
  } = useRunning()

  // ALWAYS ghost-guard real mode (no static fallback) and check *Pending before redirecting:
  if (workoutPending) return null
  if (!activeMeso) return <GhostState />
  // ...
}
```

- Reads return `null`/`[]` in real mode when the backend is empty ÔÇö render a ghost, never assume data.
- Writes are imperative callbacks that `mutate` + invalidate the relevant query key; most accept an optional `{ onSuccess }`. In mock mode `useTrain` writes **no-op** while `useRunning` writes **emulate the server** via `queryClient.setQueryData` (so the `Fut├ís` flows are interactive without a backend).
- **Backend consumer:** inject the relevant `*Service`; all methods take `UUID createdBy` first (from `CurrentUserId`). Use generated `api.dto` types only.

## 7. How to extend it

Adding a Train endpoint/field/sub-tab is **contract-first + dual-mode + both-test-modes**. The obligations are non-negotiable.

**Add or change an endpoint/field:**
1. Edit `api/feature/train/train.yml` (path + schema) FIRST. Follow `docs/references/api_contract_conventions.md`.
2. Merge + regenerate: `cd api/generate && npm run generate:api`; then `cd frontend && pnpm generate:api`. Backend types regenerate during `./mvnw generate-sources`.
3. Backend: implement the method on `TrainController` (delegate to a `*Service`). Follow `docs/references/java_package_structure.md` (`feature/train/{controller,service,repository,entity,dto,mapper}`) and `spring_patterns.md` (constructor DI via `@RequiredArgsConstructor`, method-level `@Transactional` on writes only, derivedÔćĺJPQLÔćĺnative repositories, MapStruct).
4. New table Ôćĺ Liquibase changeset `{YYYYMMDDHHMM}_{bd-id}_{desc}.sql` per `docs/references/liquibase_conventions.md` (UUID PK `gen_random_uuid()`, explicit `pk_/fk_/uq_/ck_/idx_` names, soft-delete columns). Seed data in Java `@Profile("demodata"|"demofixtures")` ÔÇö never SQL. Add the table to `ResetDatabase`'s TRUNCATE list and add/extend a `*Populator` (`docs/references/integration_test_framework.md`).
5. New configurable value Ôćĺ `application.yml` under the `mezo:` root (`docs/references/configuration_conventions.md`), never `@Value`.
6. Errors Ôćĺ `SystemRuntimeErrorException` + a `SystemMessage` code (`docs/references/error_handling.md`).
7. Frontend: add the client method to `data/train/trainApi.ts`/`data/train/runningApi.ts`, wire a query/mutation in `trainHooks.ts`/`runningHooks.ts` **with both branches** ÔÇö mock (no-op for `useTrain`, or `setQueryData` for `useRunning`) and real. Keep the boundary cast idiom (`as Mesocycle` style) consistent and use `satisfies` on request bodies. Ghost-guard the new read in every view.

**Add a Train sub-tab:** add an entry to `TRAIN_TABS` (`frontend/src/features/train/pages/tabs.ts`), add the route + a `pages/<X>View.tsx`, and surface its data through `useTrain`/`useRunning` (or a new hook re-exported from `data/hooks.ts`).

**Swap a mock-only feature to real** (e.g. a future cross-load engine): produce the data server-side, replace the static fixture branch in the hook with the API client call, and drop the `crossLoad: null` ghost ÔÇö keeping the mock branch byte-identical to Phase 1.

## 9. Decisions, gotchas & deferred

**Key decisions** (see linked specs): mock parity vs clean-slate real mode (`2026-06-11-train-write-clean-slate-design.md`); running as a mesocycle-mirroring slice with typed-jsonb plan structure (`2026-06-14-train-running-slice-design.md`); on-the-fly records aggregation (no materialized table) over soft-deleted rows (`2026-06-12-exercise-records-design.md`); catalog as master content loaded in every profile (`2026-06-12-exercise-catalog-design.md`).

**Gotchas:**
- **Mock parity contract is the single most important rule:** mock returns byte-identical Phase-1 statics (synchronous `initialData`); real has no fallback Ôćĺ views MUST ghost-guard.
- **`useTrain` mocks no-op; `useRunning` mocks emulate** (`setQueryData`). Asymmetric on purpose.
- **Real-mode run-block create inserts synchronously into the cache** (`runningHooks.ts:52-63`, **mezo-11m**): on a `create` success the new block is written into `['running','blocks']` directly (no async invalidate that would race the read-after-write), so navigating straight to the builder no longer crashes on an `undefined` structure. Updates + mock mode still invalidate. The builder also guards `!draft.structure` with a `Bet├Âlt├ęsÔÇŽ` placeholder (`RunningBlockBuilderPage.tsx:97-99`).
- **Run-builder edits auto-save (no Save button):** a 600 ms debounce (`RunningBlockBuilderPage.tsx:65-69`) plus a flush-on-back (`backToList`, `:71-74`); `dirty` is a `JSON.stringify` diff of the draft vs the loaded block (`:57-60`). The re-seed effect keys on `block?.id` only so a refetch never clobbers in-progress edits.
- **Template vs instance** both live in `workout_session`: template = `template_session_id IS NULL`; instance carries date/status/back-link. `listMesocycles`/`getToday` filter to templates; `logSet`/`finish` operate on instances.
- **Single-active invariant** for both mesocycles and running blocks (enforced server-side on activate AND on meso create-as-active).
- **Records resolve over soft-deleted exercises** via native query ÔÇö day-edits soft-delete template rows but must not erase history.
- **Warmup sets (`kind='warmup'`) are excluded from every "real work" reader** (hypertrophy drive, `mezo-dhdr`): `ExerciseRecordService` (e1RM / best-set / session-volume), `WorkoutService.lastWeekRefs`, and the `GymSignalCalculator` progression XP all count **working sets only**, so a warmup ramp never inflates a PR, a "M├║lt h├ęt" reference, or earned XP. Correspondingly the recommendation engine reads **completed-instance history only** ÔÇö the open live session is invisible to its own prescription, which is what makes the prescription **stable across a mid-workout hard-reload resume** (┬ž4 engine subsection).
- **Ownership errors:** foreign/missing Ôćĺ **404** (indistinguishable); `logSet` on a completed instance Ôćĺ **409**.
- **`phase_curve text[]` is `List<String>`** (not `String[]`) for Hibernate dirty-checking.
- Day-edit save (`MesoExercises.tsx`) fires `saveDayExercises` only when the day carries a real row `id`; mock fixtures have no id Ôćĺ local-only. **Exercise reorder is real** (the `SortableList` primitive, drag + Ôľ▓Ôľ╝): the builder remaps the day's exercises and persists via the same `saveDayExercises` full-list PUT (array order = the contract); the planner (`MesocyclePlannerPage` step 3) reorders its local draft `program` only (saved wholesale at the end).
- **Gym time is a *standalone* schedule, not a meso-day field** (Option Y, `2026-06-15-gym-schedule-times-design.md`): `GymScheduleSlot` persists across mesocycles; the meso decides *whether* you train that weekday, the slot decides *when*. `deriveGymSchedule` joins them by weekday ÔÇö a slot with no meso gym day shows nothing; a meso gym day with no slot renders with `time: null` (graceful). PUT is **replace-all** (delete-then-insert the whole week).
- **`Mai` orders by time-of-day, not modality** ÔÇö `daySessions` (`features/train/logic/agenda.ts`) is the single ordering helper shared by `WeeklyDayRow` and `TrainTodayPage` heroes; untimed (null/`''`) sort last (`'99:99'` sentinel), ties stable by modality. `TrainTodayPage` builds today's hero from a `daySessions` pass, with run-only-today handled via a **synthetic** agenda day (no gym/sport but a running session) so the run still gets a hero; a fully empty today shows "Ma pihen┼Ĺnap".
- **`GymPage`'s "Id┼Ĺpontok" entry is mock-gated** (`GymPage.tsx:70`): hidden in mock mode because `saveGymSchedule` no-ops there, mirroring `SportPage`. Mock mode still *renders* gym times from the static `gymScheduleMock` slots (`data/train/train.ts`), it just can't edit them.
- **`Mai` done-state matching is asymmetric** (`TrainTodayPage.tsx`) ÔÇö there is **no scheduleÔćölog link** for sport, so "logged today" is matched by **today's date AND sport, per slot** (`mezo-05v6` generalized this from a volleyball-hardcoded match): `loggedSportToday(k) = sport.sessions.find(s => s.sport === k && s.date === huMonthDayDow(localDateString()))` for the hero, `sportDoneOn(date, k)` for the weekly rows (the mapped `SportSession.date` is the HU display string, so we compare against today formatted the same way) ÔÇö a mixed day (e.g. a TRX slot + a volleyball slot) flips each slot independently since each carries its own `k`. Running **does** carry the prescribed tuple back, so it matches on `blockId + weekNumber + sessionKey` (`runLoggedFor(key)`), ignoring date. **Gym** has no client-readable per-day completion, so the backend computes it: `GET /api/train/workouts/today` returns `weekDoneDates` ÔÇö the ISO dates this MonÔÇôSun week whose gym instance is **`status = 'completed'`** (`WorkoutService.doneDatesThisWeek` Ôćĺ `WorkoutSessionRepository.findDoneInstanceDates`; **completed-only since `mezo-cd8s`**, not the old Ôëą1-logged-set ÔÇö a started-but-unclosed session is settled by the lazy auto-close, ┬ž4). `TrainTodayPage` derives `loggedGym = weekDoneDates.includes(today)` for the hero, and per-row `gymLogged`/`isSportLogged(slot)` (matched against each weekly row's computed calendar `date`, the latter also against the slot's sport) so **past** completed days are marked, not just today. **The done-state stores nothing new ÔÇö it is a pure read-derivation over already-persisted sessions/sets.** In real mode the log itself is persisted to Postgres (`POST /api/train/sport-sessions` Ôćĺ `SportService.create` Ôćĺ `sport_session` table, ┬ž4); the flip then happens after `logSportMutation`/`logRunSession` **invalidateÔćĺrefetch** the session query from the backend (the DB is the source of truth ÔÇö the same data the Sport `Napl├│` reads). The `setQueryData` cache-append is **mock-only** (the `mock ? ÔÇŽ : ÔÇŽ` arm in `trainHooks.ts`): it exists so the UI flips without a backend in tests/prototype, was a silent no-op before, and **never** substitutes for the DB in real mode ÔÇö mock `sport.week` stats are also not recomputed on append (real mode re-derives them). A timezone edge exists: the backend stamps the log date server-side, so the flip relies on the server "today" matching the browser's local date.
- **Napiv S4 tab rename + pill sub-nav (`mezo-8141`).** The `GYM` tab is now labelled **`Gym`** (`TRAIN_TABS`, `pages/tabs.ts`) ÔÇö copy-only, no route/id change. `TrainSubNav` switched from the generic `.subnav-item`/`.active` (`--brand-glow` underline) idiom to the coral `.np-pills`/`.np-pill` pill nav (`.on` active state, tinted via the `--pill-accent` custom property, defaulting to `var(--coral)`); Train is the first domain off the old idiom, Fuel (`--sage`) and Me (`--lav`) are slated to adopt the same pill markup in S6/S7 (`docs/features/_platform-design-system.md`).
