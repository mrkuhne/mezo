# Train — Feature Documentation

> The six-tab strength + conditioning area (plan, execute, log, analyze). **Status: ✅ done (FE mock + FE real + Spring Boot backend)** — the only gaps are clearly-flagged Phase-3 AI/analysis features. Lives under the **`Edzés`/Train** tab at route `/train` (sub-nav defined in `frontend/src/features/train/pages/tabs.ts`).

---

## 1. Summary

Train is the largest mezo domain: a six-tab area for planning and executing strength work and conditioning. It covers periodized **mesocycles** (`Mesociklus`), the active week's **gym split**, full-screen **workout execution** with per-set logging, an **exercise catalog + per-exercise records** explorer, **sport** (`Röplabda` — volleyball, cross, and TRX, 0..n scheduled slots per day, `mezo-05v6`) scheduling/logging, and structured **interval running** (`Futás`) plans. A `Mai` tab aggregates all three rhythms (gym + sport + running) onto one weekly agenda.

**Status per layer — all ✅ done:**

- **FE mock** (Phase 1): synchronous static fixtures, every tab renders without a backend.
- **FE real** (Phase 2): each read/write swapped to `/api/train/*` behind the same hooks; no signature change.
- **Backend** (Phase 2): full `feature/train/**` Spring Boot slice over Postgres — mesocycles, workout execution, sport, exercise catalog/records, running.

**Phase-3-planned (🟣, flagged throughout, do NOT assume live):** `niggleWarning`, PR detection (the current toast is a scripted demo), the live cross-load → volume recompute engine (both Sport and Running cross-load are static), `volumePerMuscle`/`volumeRecompute` provenance (seed-only), `shoulderLoadTrend`, sport `jumpCount`/`intensity`/`team`/`season` capture, gym-schedule **`duration`** (the gym-time **schedule is now live** — see §4 `GymScheduleSlot`; only per-day duration stays out of scope), copy-week, and the active-workout **Voice** tool (still no UI at all, Phase 3). **The old inert `90s` timer chip is gone, not deferred** — Napiv S5 (`mezo-8141`) replaced it with a real rest Live-Activity countdown island (§2/§9), whose 150s/90s duration is a **hardcoded** per-exercise-type heuristic (`restTimer.ts`), not a data field; a configurable per-exercise rest duration would need a new backend field (Phase 2 backlog candidate, not Phase 3).

## 2. User-facing behavior

**Since the gamified-header slice (`mezo-k7rn`, 2026-07-18), `TrainSection` renders the shared `AppHero` identity/progression header** — every Train sub-page inherits it exactly like the pre-existing `.pghead-np` per-page headers below it. **Since the compact-header redesign the same afternoon (`mezo-ugqb`), `TrainSection` also passes a `SubNavDropdown` as `AppHero`'s `utilities` prop** (`TrainSection.tsx` — `items={TRAIN_TABS}`, `accent="var(--coral-deep)"`); Train's sub-nav is no longer its own row/component, just this one prop. `AppHero` itself is domain-free platform chrome (avatar + XP ring/level badge, equipped title, label-less 🔥/⚡/🪙 counters). See [growth.md](growth.md) for the account-progression domain and [`_platform-design-system.md` §1a](_platform-design-system.md) for the `.apphero`/`SubNavDropdown` CSS families.

The sub-nav (`TRAIN_TABS`, `frontend/src/features/train/pages/tabs.ts`, rendered via the shared `SubNavDropdown`) has six tabs: **`Mai`** (`/train`), **`Gym`** (`/train/gym`), **`Sport`** (`/train/sport`), **`Futás`** (`/train/futas`), **`Gyakorlatok`** (`/train/exercises`), **`Mesociklusok`** (`/train/mesocycles`). Four full-screen sibling routes have no sub-nav: active workout (`/train/session`), the done-day review (`/train/review/:workoutId`, `WorkoutReviewPage`), mesocycle planner/builder (`/train/mesocycles/new`, `/train/mesocycles/:id`), and the running-block builder (`/train/futas/:id`). **Since Napiv S4 (`mezo-8141`) the tab label reads `Gym` (was `GYM`)** — a copy-only change (the route/id are untouched). **`TrainSubNav.tsx` and the `.np-pills`/`.np-pill` pill-nav CSS it rendered are deleted (compact-header redesign, `mezo-ugqb`)** — the chip shows the active tab's label (`"Gym ▾"`) and tapping it opens a popover menu instead of a pill row; see [`_platform-design-system.md`](_platform-design-system.md) §3/§5 for the primitive. **Every Train page still opens with a `.pghead-np` page-head** (an `.over` eyebrow line + `h1`, optionally a `.pgact-np` pill action chip) in place of the generic `.page-header`/`Eyebrow`/`PageTitle` idiom described in the design-system doc.

### `Mai` — the weekly cross-domain agenda (`pages/TrainTodayPage.tsx`)
**Visual composition (Napiv S4, `mezo-8141`; multi-sport since `mezo-05v6`):** `.pghead-np` header (`Edzés · {day} · W{n}`, or just `Edzés · W{n}` on a run-only/no-today row) → today's hero card(s) in time-of-day order — a coral `.trainhero` gym hero, or `.np-eventrow` sport/run heroes tagged `.typetag-sport`/`.typetag-run` (**one hero per scheduled sport slot** — a mixed day renders a hero for each, e.g. a noon TRX hero + an evening volleyball hero) — → a dashed rest-day card when nothing is scheduled today → the `LoadTiles` weekly-load summary (`logic/weeklyLoad.ts` + `components/LoadTiles.tsx`, one `.loadtile` per active modality, renders nothing on an empty week) → the `Heti terv` `.secthead-np` section of `WeeklyDayRow` `.dayrow` day cards (each slot tagged via `.stag-gym`/`.stag-sport`/`.stag-run`, one row per sport slot). Where gym, sport (volleyball/cross/TRX), and running converge. Both **today's hero cards** and the **weekly timeline rows** (`WeeklyDayRow`) are now ordered **by time-of-day** rather than by a fixed gym→sport→running modality order — a morning run renders above an evening gym; untimed sessions sort last. Both surfaces order through the same pure helper `daySessions(day)` (`features/train/logic/agenda.ts`), which flattens a day's `{gym, sport[], running[]}` — `WeeklyAgendaDay.sport: VolleyballSession[]` (`components/WeeklyDayRow.tsx:16`) holds **0..n slots per day**, each carrying its own `sport: volleyball|cross|trx` (resolved via `sportOf(slot)`, `logic/sportKinds.ts`) — into typed `AgendaItem`s carrying `timeOfDay` and sorts ascending (null/`''` → `'99:99'` last, then stable by modality). Gym time comes from the standalone gym schedule (§4 `GymScheduleSlot`); running time from each session's `timeOfDay` (now shown next to the weekday in the weekly rows). You can log a sport (volleyball/cross/TRX) or running session inline via the shared `SportLogSheet`/`RunLogSheet` — the sport CTA **pre-selects the tapped slot's sport** (`SportLogSheet`'s `initialSport` prop, driven by `TrainTodayPage`'s `setSportLogSport(k)`). **Once a session is logged, its hero and weekly row flip to a `done` state — matched by date AND sport**, so a mixed day (e.g. a noon TRX slot + an evening volleyball match) flips each slot independently (`loggedSportToday`/`sportDoneOn`, `TrainTodayPage.tsx`): the `MA` chip becomes a `✓ Kész` (success-coloured) chip and the log CTA is replaced by a muted summary of the logged effort (volleyball: `Logolva · RPE · perc · váll`; cross/TRX: `Logolva · RPE · perc` — no `váll`, shoulder strain stays volleyball-only; running: `Logolva · RPE · kör`; **gym**: a **three-state gate** — see the gym-hero gating paragraph below). **All three modalities now flip — including gym** — and **past days this week are marked too**, not only today: in the weekly rows a gym day shows `kész` when its ISO date is in `weekDoneDates` (server-computed, §4), a sport slot when a session of **that slot's sport** is logged on that date, a running day when its prescribed session is logged. Tapping a sport/run *today* summary re-opens the sheet (an in-place edit needs a backend `PUT` — `mezo-0p3` follow-up); the **completed gym day is now interactive** (weekly kész gym rows and the today hero both open the read-only review — see the gating paragraph below). Ghosts entirely when there is no active mesocycle ("+ Tervezz mesociklust"). Run-only-today (a running session today with no gym/sport) renders its hero via a synthetic agenda day; a true rest day shows the "Ma pihenőnap" card.

**Gym-hero done-day gating (`mezo-cd8s`).** The gym hero is a **three-state gate** driven by today's instance, not by `weekDoneDates`: (1) **no instance** → the `Indítsuk →` start CTA (`openSession` → `/train/session`); (2) an **open** (started, unfinished) instance (`todaySession.openWorkout`) → a `● Folyamatban` eyebrow chip + a `Folytassuk → · {n} szett kész` resume CTA, `n` = the non-skipped set count; (3) a **completed** instance (`completedTodayWorkout` — since the cross-day start (`mezo-p7rp`) the resolved day's most recent finished instance **of the current Mon–Sun week**, not just today's) → a success-tinted `Kész · {n} szett — Megnézem →` button that opens the read-only review `/train/review/{completedTodayWorkout.id}` (`WorkoutReviewPage`, renders the shared `WorkoutSummary` in `closed` mode). Both `todaySession`/`completedTodayWorkout` come from `/today` and are **real-mode only** (null in mock → the hero is always `Indítsuk →`, byte-identical to Phase 1). **Since `mezo-p7rp` the `/today` day resolution is `open instance > ?day= param > weekday label`** (§4) and the session route accepts **`/train/session?day={templateDayId}`** (`ActiveWorkoutPage` threads `useSearchParams` → `useTrain({ workoutDay })`) — this is how a non-today `GymDaySheet` start pins its template day; the open instance winning resolution means the hero's `Folytassuk` and a hard reload resume a cross-day session too. In the **weekly rows**, a completed (`kész`) gym day — today OR a past day this week — is now **tappable → opens its review**: `WeeklyDayRow`'s gym branch routes its click to `onReviewGym` when `gymLogged`, else `onStartGym` when today, else inert; `TrainTodayPage` resolves the per-date instance id from `useWeekWorkouts()` (this Mon–Sun week's completed summaries mapped date→id, `workoutIdByDate`) and passes `onReviewGym` only for dates present in that map. Today's not-yet-completed gym row also shows a `folyamatban` chip while an instance is open (`gymInProgress = a.isToday && todaySession.openWorkout`). The `/train/session` route itself is **gated**: when today is completed with nothing open, `ActiveWorkoutPage` redirects to the review (`<Navigate to={/train/review/{id}} replace />`, after the pending/no-workout guards) so the prep screen (challenges included) is unreachable until next week; the same rule flips `GymDaySheet`'s footer CTA to `Kész · Megnézem →` (`completedWorkoutId` prop from `GymPage`).

### Active workout (`ActiveWorkoutPage.tsx`, `/train/session`)
Full-screen takeover, **four phases** (`mezo-cd8s` replaced the old prep→active→auto-finish flow): **`prep`** (niggle pre-flag, **live challenges carousel** — see below, warmup, exercise list, "Kezdjük el") → **`active`** (free-navigation per-set weight/reps/RIR logging — **the viewed exercise is the logging target**, see the navigation paragraph below — "Múlt hét" comparison, set dots, set history, PR toast, RP feedback debrief) → **`summary`** (the explicit-finish **`WorkoutSummary`** in **`closing`** mode: day stats + challenge outcomes + per-exercise recap + optional note, with **"Edzés lezárása ✓"** as the ONLY finish trigger) → **`complete`** (the same `WorkoutSummary` read-only in **`closed`** mode, rendered after the finish POST resolves). A mid-workout hard reload **resumes** straight back into `active`. **Done-day gate (`mezo-cd8s`):** when today's instance is already **completed** and nothing is open, the outer `ActiveWorkoutPage` redirects to the review (`<Navigate to={/train/review/{completedTodayWorkout.id}} replace />`, placed **after** the `workoutPending`/no-workout guards) — the prep/session screen (challenges included) is unreachable until next week; mock mode has no `completedTodayWorkout`, so it never fires there. **Explicit finish (`mezo-cd8s`) + gamified level-up (`mezo-te8k`, P5).** The workout no longer auto-finishes — the summary's **"Edzés lezárása ✓"** CTA is the ONLY completion trigger. It calls `finishAndCelebrate` → `finishWorkout(workoutId ?? 'mock', { onSuccess, onSettled })` (a `finishPending` flag disables the CTA while the POST is in flight; **`onSettled` re-enables it so a failed finish can be retried** — never stuck-disabled); on the response's `levelUp` it calls `showLevelUp(r.levelUp)` (the shared `LevelUpProvider` host, §5) and flips the phase to `complete`. The full-bleed `LevelUpScreen` overlay portals **over** the **closed `WorkoutSummary`** and its single `Tovább` CTA dismisses it to reveal the closed summary underneath. The summary's **"· PR ✨"** title suffix now derives from a real `max_strength` level-up in the signal (`hadPrFromSignal`), replacing the old 105 kg demo scan; the mid-workout `showPR` toast is unchanged. On a successful finish real mode also **invalidates the day's challenges** (`['challenges', templateSessionId, today]`) so the server-resolved outcomes land on the closed summary. When the progression switch is off the response carries no `levelUp` → no overlay, the closed summary shows directly. In **mock mode** `workoutId` is null (no instance), so the `?? 'mock'` sentinel still drives the no-op finish mutation, which returns a seeded `LevelUpResult` fixture so the prototype shows the overlay. **Since Napiv S5 (`mezo-8141`) weight/reps use the giant `SetStepper`** (`features/train/components/SetStepper.tsx`) instead — a `.stepper` tile per field with ± buttons clamped to the contract bounds (weight 0–999 step 2.5, reps 1–100 step 1, integer) on every tap; **the center value is tap-to-edit again (`mezo-o7ds`)** — at rest the `.n` value renders as a reset-styled `<button>` (`aria-label` `"{label} pontos megadása"`, visually unchanged), tapping it swaps in an `<input>` backed by the same `useEditableNumber` hook as `NumberStep` (free keyboard entry, HU decimal comma, commit + clamp on blur/Enter), so exact non-±2.5 weights (microplates, odd dumbbells — `weightKg` has no `multipleOf` in the contract) are loggable again, restoring the CompactStepper capability S5 dropped. **The tile layout deviates from the S5 mockup (`mezo-eerq`):** the value line sits ABOVE a centered ± button row instead of the mockup's side-flanking buttons — at real phone widths (two tiles in a ~330px card) a flanking layout cannot fit a `107,5 kg` value without squashing the 40 px round buttons into ovals. Its CSS is **scoped as `.steprow .stepper`** (`prototype.css`) because the bare `.stepper` class is the older generic grid stepper (RecipeStepper et al.), whose `display:grid` used to bleed in and overflow the excard (the `CompactStepper` component that provided logging here pre-S5 is gone from this screen — it survives only in the run-block builder's `RunWeekEditor`, where its `useEditableNumber` hook still backs `NumberStep`, same as in the sport/run log sheets — but its tap-to-edit UX is back on this screen via `SetStepper`, above). See the S5 layout paragraph below for the full `.wk-top`/`.excard` structure this stepper sits in. The active-phase session state — logged sets, the per-exercise cursor, and per-exercise set counts — lives in a pure, **exerciseId-keyed** model (`workoutState.ts`: `Session{order,logged,extra,skipped,planned,prescribed}` + `makeSession`/`currentExerciseId`/`nextSetIdx`/`nextUnfinishedAfter`/`effectiveSetCount`/`completeSet(s,exerciseId,set)`/`addExtraSet`/`skipExercise`/`seedFromOpen`/`mergePlan`), so the screen is a thin view over it (resume rebuilds the model from the open instance via `seedFromOpen`). **The cursor is per-exercise (`mezo-cd8s`):** there is no scalar `setIdx` — an exercise's next-set index derives from `nextSetIdx(s,id) = logged[id].length`, `completeSet` appends to a **given** exercise (not a linear cursor), and the page tracks a `viewedId` (the on-screen exercise = the logging target) so navigation can be free; `nextUnfinishedAfter(s,id)` picks the auto-advance target (next non-skipped, not-fully-logged exercise after `id`, wrapping) after a debrief/skip. The default flow is behavior-identical — `viewedId` seeds from and advances through the linear `currentExerciseId` order — but the athlete can now **navigate freely** (`mezo-cd8s`): the two-way **`.pagerbar`** below the card (‹ Előző / Következő › — plain order-neighbours of the viewed exercise, disabled at the list edges), **tappable `.exdots`**, a **horizontal swipe** on the `.excard` (a ≥60 px drag; left = next / right = previous, so taps on the inner controls never trigger it), and the header counter (now a `▾` button, `aria-label="Gyakorlatlista"`) opening **`ExerciseOverviewSheet`** (`sheets/ExerciseOverviewSheet.tsx`) — a jump-to list of every exercise with its live state (`✓ kész` / `● folyamatban n/m` / `○ hátravan` / `⊘ kihagyva`, current row tinted). Every jump routes through `jumpTo(id)`, which **no-ops while a debrief is open** (`feedbackEx` pins the on-screen exercise). **Mid-session plan growth is reconciled via `mergePlan`** (`mezo-ohvm`): the session is seeded once at mount, but the server can append template exercises while a workout is open (the fix-zárás closing block, §4) — an id missing from `session.planned` would read as 0 planned sets and count as done, silently auto-finishing the workout. A `useEffect` on `W.exercises` folds new plan exercises into `order`/`planned`/`prescribed` (appended at the end, progress untouched; identity-stable when nothing is new). This non-linear, id-keyed cursor is the foundation for the in-session reorder / add-set / skip / note features. The active-phase header carries a **`⋯` "Gyakorlat műveletek"** button that opens `ExerciseActionSheet` (`sheets/ExerciseActionSheet.tsx`) — a **five-row** menu (↕ Áthelyezés · ⊘ Kihagyás · ＋ Szett · ✎ Jegyzet · ✓ **Edzés befejezése…**) built on the shared `Sheet`; the last row (`onFinishWorkout`) is the **early exit to the summary** — it lands the session on the `WorkoutSummary` closing screen *before* every exercise is resolved (`mezo-cd8s`). **Áthelyezés (reorder)** switches to a sub-view hosting the shared `SortableList` over the **remaining (future) exercises only** — the done + current exercises (`session.order.slice(0, ci+1)`) stay fixed, and the reorder is **client-only / ephemeral** (it just replaces `session.order` via a functional `setSession` updater, no persistence). Because `nextUnfinishedAfter` (the post-debrief auto-advance) derives "next" from `order` rather than an array index, reordering the tail directly changes which exercise comes up after the current one. **＋ Szett (add-set, F2)** is wired via `onAddSet`: it bumps `session.extra[id]` (`addExtraSet`), growing `effectiveSetCount` (planned + extra) so the header set count and the set dots gain one more. **The added set carries a distinct dashed marker again**, restored in the final-review fix (`mezo-8141`): a set-dot at/past the exercise's `session.planned` baseline (`i >= session.planned[current.id]`) renders `.setdots .sd.extra` — a dashed coral outline (`prototype.css`) that only applies while the dot is still pending, yielding to the normal `.don`/`.cur` styling once it's logged/current. ## 3. Architecture & data flow

The whole area flows through **two hooks**, both re-exported from the single data boundary `frontend/src/data/hooks.ts:202-203`:

```
View (features/train/**)
  → useTrain()      (data/train/trainHooks.ts)   |   useRunning()  (data/train/runningHooks.ts)
    → isMockMode()  (@/data/_client/mode → VITE_USE_MOCK !== 'false')
       ├─ MOCK: TanStack Query initialData = static fixtures (data/train/train.ts, data/train/running.ts)
       │        writes no-op (useTrain) OR emulate the server via queryClient.setQueryData (useRunning)
       └─ REAL: api client (data/train/trainApi.ts, data/train/runningApi.ts) over apiFetch → /api/train/*
            → TrainController.java (implements generated TrainApi)
              → {Train,Workout,Sport,ExerciseCatalog,ExerciseRecord,Running}Service
                → repositories → Postgres
```

**Invariants of the seam:**
- **Mock mode is synchronous** (`initialData` set) so the first render equals the Phase-1 static return — preserved for the visual baselines and component tests. **Real mode has NO static fallback**: an empty backend surfaces as `null`/`[]` and every view ghost-guards (the "T0 clean slate" rule, `2026-06-11-train-write-clean-slate-design.md`). The full rationale is in the comment block at `frontend/src/data/train/trainHooks.ts:166-173`.
- **Mode is read inside hook bodies, never at module scope** (`frontend/src/data/_client/mode.ts`), so tests stub per-case with `vi.stubEnv`.
- Default mode is **mock** (`VITE_USE_MOCK !== 'false'`), but per `CLAUDE.md` the project runs FE in REAL mode by default in dev/test.
- Contract types are generated: `api/feature/train/train.yml` → merged `api/openapi.yml` → FE `frontend/src/data/_client/api.gen.ts` (re-exported as named types in `trainApi.ts`/`runningApi.ts`) and BE `io.mrkuhne.mezo.api.dto.*` + the `TrainApi` interface that `TrainController` implements.

**Real-mode mapping notes** (`trainHooks.ts`): `toWorkoutPlan` (`:47-62`) collapses the API response to the Phase-1 `WorkoutPlan` shape and sets `challenges: []` (live challenges are proactive-owned and sourced separately via `useChallenges` on `ActiveWorkoutPage`, not through `WorkoutPlan` — see §Active workout); `deriveGymSchedule(meso, slots)` (`:67-83`) builds the gym week by joining the active meso's template days (WHAT) with the standalone weekly gym slots (WHEN) — `DAY_ORDER` index `==` `slot.dayOfWeek`, a gym day with no matching slot keeps `time: null`; `duration` stays `null` (no DB home, out of scope). The active-meso real fallback is `realActiveMeso ?? (mock ? activeMeso : null)` (`:393`).

#### Set recommendation engine — hypertrophy drive (`mezo-dhdr`)
`SetRecommendationService` (`feature/train/service/SetRecommendationService.java`) computes the per-set targets that `getToday` attaches as `prescribedSets`. It applies **double progression** on the exercise's **top working set** of the most-recent completed instance: reps `≥ rep_max` → bump the load by the type increment; reps `< rep_min` → drop by the increment; reps **in range** → hold the load (the athlete earns reps up toward `rep_max` first). On the **first session** (no history) it seeds from `anchor_weight_kg`, or leaves the target weight **null** when no anchor is set. **Plyometric / bodyweight exercises are weightless** — the engine returns null target weights (reps/RIR only), which the FE renders as a reps-only stepper. Warmup sets are generated by ramping the working target down through the configured `warmupRamp` — and are **emitted even when the base weight is unknown** (first session, no anchor; `mezo-eerq`): their target weight is simply null, keeping the prescribed list aligned with the recipe's set count so the FE never loses the `B{n}` labels / RIR suppression on a first session. The method returns a `Prescription(sets, rationale)` record (`service/Prescription.java`) — the set list plus the human-readable `rationale` shown in the active workout (replacing the old static hint). It reads **completed-instance history only**, so a live (open) session never feeds back into its own prescription → the recommendation is **stable across a mid-workout resume** (§9).
- **Config** (`mezo.hypertrophy.*` → `HypertrophyProperties`, `feature/train/config/HypertrophyProperties.java`): `plateStep` (2.5 — rounding granularity for computed kg), `defaultIncrement` (2.5 — fallback), a per-type `increment` map (compound 5.0 / isolation 2.5), and the `warmupRamp` (`List<Ramp(pct, repsFactor)>`). `@Validated` record, wired in `application.yml` under `mezo.hypertrophy:` — never `@Value`.
- **Switch:** the whole engine is gated by `mezo.feature.hypertrophy-drive.enabled` via the `HypertrophyDriveGate` `@ConditionalOnProperty` marker bean (`feature/train/HypertrophyDriveGate.java`; key constant `FeaturesConfiguration.HYPERTROPHY_DRIVE_SWITCH`). Off → `WorkoutService` skips the engine and returns null `prescribedSets`/`rationale`, so the FE logs ad-hoc.

#### Closing block — fix zárás (`mezo-z2ul`)
`ClosingBlockService` (`feature/train/service/ClosingBlockService.java`) **lazily ensures the configured closing exercises at the END of every non-empty template day of the active meso** — by default **Dead Hang** (2 × 45–60 mp; weightless `plyo` type → reps-are-seconds, kg-stepper hidden on the FE) and **45° Back Extension** (2 × 12–15). Triggered from `WorkoutService.getToday` behind the `ClosingBlockGate` `@ConditionalOnProperty` marker bean (`mezo.feature.closing-block.enabled`; key constant `FeaturesConfiguration.CLOSING_BLOCK_SWITCH`; mirrors `HypertrophyDriveGate`). **Idempotent** by catalog id OR case-insensitive name (a manually added copy counts, nothing duplicates); **empty (rest) template days are skipped** so the block never turns a rest day into a workout; a day-edit that strips the rows self-heals on the next `getToday`. Config: `mezo.closing-block.exercises` — `ClosingBlockProperties` (`feature/train/config/`), a `@Validated` record list of `{slug, workingSets, repMin, repMax, targetRir}` where `slug` references a **master catalog row** (a missing slug logs a warning and is skipped, never a 500). The appended rows are **ordinary catalog-linked template exercises** (`warmupSets 0`, server-side `createdBy`), so logging, records, history and prescriptions work unchanged — **no API contract change, no DDL** (the new `dead-hang` master entry ships via `content/exercise-catalog.json`; `back-extension-45` already existed). ITs: `ClosingBlockIT` (append/idempotency/multi-day/rest-day/manual-duplicate) + `ClosingBlockSwitchOffIT`.

#### Lazy auto-close of stale instances (`mezo-cd8s`)
`WorkoutAutoCloseService` (`feature/train/service/WorkoutAutoCloseService.java`) **settles abandoned workout instances** so a session left open on a past day never lingers as `active`. Invoked as the **first statement** of `WorkoutService.getToday` (before the closing-block ensure and the meso lookup, so stale instances settle even when there is no active meso). A SEPARATE `@Transactional` bean because `getToday` is a plain read and a same-class `@Transactional` self-invocation would bypass the proxy (mirrors `ClosingBlockService`). For each of the owner's `active` instances dated **strictly before today** (`WorkoutSessionRepository.findByCreatedByAndStatusAndDateBeforeAndTemplateSessionIdIsNotNull`): it becomes **`completed`** when it carries **≥1 non-skipped logged set** (the work counts — feeds `findDoneInstanceDates`, `lastWeekRefs` and the prescription engine), else **`skipped`** (a started-but-empty session never counts as done). **Idempotent** (a settled instance is no longer `active`, so a re-run is a no-op) and **always on** — no feature gate, **no DDL/contract change**. After it runs, only a **today-dated** instance can still be `active`, so the open-instance resume lookup below can never resurrect last week's abandoned session. IT: `WorkoutAutoCloseIT` (past-active-with-set → completed / past-empty → skipped / today-active → untouched).

#### Done-state = completed status + workout-detail review (`mezo-cd8s`)
The gym "done that day" signal changed from **≥1 non-skipped logged set** to **`status = 'completed'`** (an explicitly finished — or lazily auto-closed — instance): `WorkoutSessionRepository.findDoneInstanceDates` (→ `weekDoneDates`) and `findDoneInstancesBetween` (→ Insights `listWorkouts` + the companion `get_recent_workouts` tool, §5) both switched their predicate to `s.status = 'completed'`, so a started-but-unclosed instance with logged sets is **no longer** counted (the auto-close settles stale ones, above). `lastWeekRefs` and `SetRecommendationService` already read completed-only history — unchanged. The same shift flows to every consumer of `findDoneInstanceDates` (QuestEvaluator, `TrainingCommitmentCalculator`/discipline trait, Insights weekly counts, companion tools) — the agreed semantic root of the four fixes.
- **`completedWorkout` on `getToday`** — after the auto-close, `WorkoutService.getToday` resolves the most recent `completed` instance of today's template day **dated today** (`findFirstByCreatedByAndTemplateSessionIdAndStatusAndDateOrderByCreatedAtDesc`) onto `WorkoutTodayResponse.completedWorkout` (null when none). One extra read, no second endpoint — drives the Mai Kész/Megnézem hero, the `GymDaySheet` gating and the `/train/session` → review redirect (§2).
- **`GET /api/train/workouts/{id}` → `WorkoutDetailResponse`** (`WorkoutService.getWorkoutDetail`, a pure read, no `@Transactional`): the done-day review data source (weekly rows + Mai "Megnézem"). It looks up the owned **instance** (`ownedInstanceOrThrow`, instance-only) → joins the template day's exercises (`ExerciseRepository.findByCreatedByAndWorkoutSessionIdInOrderByOrderIndexAsc` on the instance's `templateSessionId`) with this instance's logged sets grouped per exercise; a skip-marker set flips the exercise's `skipped` flag and is excluded from `sets` (non-skipped, `setIndex` order). A foreign/missing id **or a template row** (`templateSessionId == null`) → **404 `TRAIN_WORKOUT_NOT_FOUND`**. ITs: `WorkoutDoneSemanticsIT` (completed counts, active-with-sets does not), `WorkoutDetailContractIT` (happy / 404 foreign / 404 template / skipped flag).

Mock mode mirrors the derivation via `currentWeekOf` (`shared/lib/dates.ts`). _Known limitation:_ like mesocycles, the stored value only re-derives on write/activate; the read-time heal only fires for out-of-range values, so a valid block does not auto-advance week-to-week without a write.
- **DTOs:** `RunningBlockResponse`/`RunningBlockUpsertRequest`, `RunningBlockStructureDto`, `RunWeek`, `RunPrescribedSession`, `RunSegment`, `RpeTarget`, `RunSessionLogResponse` (now also carries an optional `levelUp` → `LevelUpResult`, the same schema as the gym finish)/`RunSessionLogRequest`.
- **Seed:** `RunningSeedData` is `@Profile("demofixtures")` (same as `TrainSeedData`, since `mezo-uuv`) — a plain `demodata` (prod) app starts **clean** with no running blocks; the 3 demo blocks (active/planned/archived) require the opt-in `demodata,demofixtures` profile combo.

## 6. How to use it (consume)

Import from the single boundary `@/data/hooks`; never reach into `trainHooks.ts`/`runningHooks.ts` directly.

```tsx
import { useTrain, useRunning } from '@/data/hooks'

function MyWidget() {
  const {
    mesocycles, activeMeso, workout, gymSchedule, gymSlots, sport,
    exerciseLibrary, exerciseRecords, workoutPending, todaySession,
    // imperative writes:
    createMesocycle, activateMesocycle, closeMesocycle, saveDayExercises,
    startWorkout, logSet, saveWorkoutFeedback, finishWorkout,
    logSportSession, saveSportSchedule, saveGymSchedule,
    createCatalogExercise, updateCatalogExercise, deleteCatalogExercise, setExerciseVideo,
  } = useTrain()

  const {
    runningBlocks, activeRunningBlock, runSessions, runningPending,
    saveRunningBlock, activateRunningBlock, closeRunningBlock,
    deleteRunningBlock, logRunSession,
  } = useRunning()

  // ALWAYS ghost-guard real mode (no static fallback) and check *Pending before redirecting:
  if (workoutPending) return null
  if (!activeMeso) return <GhostState />
  // ...
}
```

- Reads return `null`/`[]` in real mode when the backend is empty — render a ghost, never assume data.
- Writes are imperative callbacks that `mutate` + invalidate the relevant query key; most accept an optional `{ onSuccess }`. In mock mode `useTrain` writes **no-op** while `useRunning` writes **emulate the server** via `queryClient.setQueryData` (so the `Futás` flows are interactive without a backend).
- **Backend consumer:** inject the relevant `*Service`; all methods take `UUID createdBy` first (from `CurrentUserId`). Use generated `api.dto` types only.

## 7. How to extend it

Adding a Train endpoint/field/sub-tab is **contract-first + dual-mode + both-test-modes**. The obligations are non-negotiable.

**Add or change an endpoint/field:**
1. Edit `api/feature/train/train.yml` (path + schema) FIRST. Follow `docs/references/api_contract_conventions.md`.
2. Merge + regenerate: `cd api/generate && npm run generate:api`; then `cd frontend && pnpm generate:api`. Backend types regenerate during `./mvnw generate-sources`.
3. Backend: implement the method on `TrainController` (delegate to a `*Service`). Follow `docs/references/java_package_structure.md` (`feature/train/{controller,service,repository,entity,dto,mapper}`) and `spring_patterns.md` (constructor DI via `@RequiredArgsConstructor`, method-level `@Transactional` on writes only, derived→JPQL→native repositories, MapStruct).
4. New table → Liquibase changeset `{YYYYMMDDHHMM}_{bd-id}_{desc}.sql` per `docs/references/liquibase_conventions.md` (UUID PK `gen_random_uuid()`, explicit `pk_/fk_/uq_/ck_/idx_` names, soft-delete columns). Seed data in Java `@Profile("demodata"|"demofixtures")` — never SQL. Add the table to `ResetDatabase`'s TRUNCATE list and add/extend a `*Populator` (`docs/references/integration_test_framework.md`).
5. New configurable value → `application.yml` under the `mezo:` root (`docs/references/configuration_conventions.md`), never `@Value`.
6. Errors → `SystemRuntimeErrorException` + a `SystemMessage` code (`docs/references/error_handling.md`).
7. Frontend: add the client method to `data/train/trainApi.ts`/`data/train/runningApi.ts`, wire a query/mutation in `trainHooks.ts`/`runningHooks.ts` **with both branches** — mock (no-op for `useTrain`, or `setQueryData` for `useRunning`) and real. Keep the boundary cast idiom (`as Mesocycle` style) consistent and use `satisfies` on request bodies. Ghost-guard the new read in every view.

**Add a Train sub-tab:** add an entry to `TRAIN_TABS` (`frontend/src/features/train/pages/tabs.ts`), add the route + a `pages/<X>View.tsx`, and surface its data through `useTrain`/`useRunning` (or a new hook re-exported from `data/hooks.ts`).

**Swap a mock-only feature to real** (e.g. a future cross-load engine): produce the data server-side, replace the static fixture branch in the hook with the API client call, and drop the `crossLoad: null` ghost — keeping the mock branch byte-identical to Phase 1.

## 9. Decisions, gotchas & deferred

**Key decisions** (see linked specs): mock parity vs clean-slate real mode (`2026-06-11-train-write-clean-slate-design.md`); running as a mesocycle-mirroring slice with typed-jsonb plan structure (`2026-06-14-train-running-slice-design.md`); on-the-fly records aggregation (no materialized table) over soft-deleted rows (`2026-06-12-exercise-records-design.md`); catalog as master content loaded in every profile (`2026-06-12-exercise-catalog-design.md`).

**Gotchas:**
- **Mock parity contract is the single most important rule:** mock returns byte-identical Phase-1 statics (synchronous `initialData`); real has no fallback → views MUST ghost-guard.
- **`useTrain` mocks no-op; `useRunning` mocks emulate** (`setQueryData`). Asymmetric on purpose.
- **Real-mode run-block create inserts synchronously into the cache** (`runningHooks.ts:52-63`, **mezo-11m**): on a `create` success the new block is written into `['running','blocks']` directly (no async invalidate that would race the read-after-write), so navigating straight to the builder no longer crashes on an `undefined` structure. Updates + mock mode still invalidate. The builder also guards `!draft.structure` with a `Betöltés…` placeholder (`RunningBlockBuilderPage.tsx:97-99`).
- **Run-builder edits auto-save (no Save button):** a 600 ms debounce (`RunningBlockBuilderPage.tsx:65-69`) plus a flush-on-back (`backToList`, `:71-74`); `dirty` is a `JSON.stringify` diff of the draft vs the loaded block (`:57-60`). The re-seed effect keys on `block?.id` only so a refetch never clobbers in-progress edits.
- **Template vs instance** both live in `workout_session`: template = `template_session_id IS NULL`; instance carries date/status/back-link. `listMesocycles`/`getToday` filter to templates; `logSet`/`finish` operate on instances.
- **Single-active invariant** for both mesocycles and running blocks (enforced server-side on activate AND on meso create-as-active).
- **Records resolve over soft-deleted exercises** via native query — day-edits soft-delete template rows but must not erase history.
- **Warmup sets (`kind='warmup'`) are excluded from every "real work" reader** (hypertrophy drive, `mezo-dhdr`): `ExerciseRecordService` (e1RM / best-set / session-volume), `WorkoutService.lastWeekRefs`, and the `GymSignalCalculator` progression XP all count **working sets only**, so a warmup ramp never inflates a PR, a "Múlt hét" reference, or earned XP. Correspondingly the recommendation engine reads **completed-instance history only** — the open live session is invisible to its own prescription, which is what makes the prescription **stable across a mid-workout hard-reload resume** (§4 engine subsection).
- **Ownership errors:** foreign/missing → **404** (indistinguishable); `logSet` on a completed instance → **409**.
- **`phase_curve text[]` is `List<String>`** (not `String[]`) for Hibernate dirty-checking.
- Day-edit save (`MesoExercises.tsx`) fires `saveDayExercises` only when the day carries a real row `id`; mock fixtures have no id → local-only. **Exercise reorder is real** (the `SortableList` primitive, drag + ▲▼): the builder remaps the day's exercises and persists via the same `saveDayExercises` full-list PUT (array order = the contract); the planner (`MesocyclePlannerPage` steps `Gyakorlatok`/`Set & rep`) reorders its local draft `program` only (saved wholesale at the end).
- **Planner draft lives at page level behind a regeneration guard (`mezo-n46i`).** `program` state, program *generation*, and every draft mutation (`addExercise`/`removeExercise`/`reorderExercises`/`renameDay` + the new recipe patcher `updateExercise`) sit in `MesocyclePlannerPage`, not in `Step3Program`. Generation runs inside a `useEffect` gated by an input **signature** (`generatedFor` ref keyed on `goal|split|days|selectedDays`): it only (re)generates when that signature actually changes, so **program edits survive a step round-trip** (e.g. `Gyakorlatok`↔`Split + napok` or the `Gyakorlatok`↔`Set & rep` back-and-forth) while a real input change still regenerates. `Step3Program` (the `Gyakorlatok` review) is presentational over `{goal,name,weeks,days,program,onAdd,onRemove,onReorder,onRename}`; it keeps only `expandedDay`/`pickerDay` local and auto-expands the first training day **exactly once per generation** via a `hadProgram` ref (an edit's array-identity change never re-opens a collapsed day — the `mezo-xnq` guard). The `Set & rep` step (`Step4Recipe`) consumes the same page-level draft mutations — `updateExercise` as `MesoDayTabsEditor`'s `onChange` recipe patcher, plus add/remove/reorder — over the day-tabbed editor. Added-exercise ids use `crypto.randomUUID()` (never `Date.now()`).
- **Gym time is a *standalone* schedule, not a meso-day field** (Option Y, `2026-06-15-gym-schedule-times-design.md`): `GymScheduleSlot` persists across mesocycles; the meso decides *whether* you train that weekday, the slot decides *when*. `deriveGymSchedule` joins them by weekday — a slot with no meso gym day shows nothing; a meso gym day with no slot renders with `time: null` (graceful). PUT is **replace-all** (delete-then-insert the whole week).
- **`Mai` orders by time-of-day, not modality** — `daySessions` (`features/train/logic/agenda.ts`) is the single ordering helper shared by `WeeklyDayRow` and `TrainTodayPage` heroes; untimed (null/`''`) sort last (`'99:99'` sentinel), ties stable by modality. `TrainTodayPage` builds today's hero from a `daySessions` pass, with run-only-today handled via a **synthetic** agenda day (no gym/sport but a running session) so the run still gets a hero; a fully empty today shows "Ma pihenőnap".
- **`GymPage`'s "Időpontok" entry is mock-gated** (`GymPage.tsx:70`): hidden in mock mode because `saveGymSchedule` no-ops there, mirroring `SportPage`. Mock mode still *renders* gym times from the static `gymScheduleMock` slots (`data/train/train.ts`), it just can't edit them.
- **`Mai` done-state matching is asymmetric** (`TrainTodayPage.tsx`) — there is **no schedule↔log link** for sport, so "logged today" is matched by **today's date AND sport, per slot** (`mezo-05v6` generalized this from a volleyball-hardcoded match): `loggedSportToday(k) = sport.sessions.find(s => s.sport === k && s.date === huMonthDayDow(localDateString()))` for the hero, `sportDoneOn(date, k)` for the weekly rows (the mapped `SportSession.date` is the HU display string, so we compare against today formatted the same way) — a mixed day (e.g. a TRX slot + a volleyball slot) flips each slot independently since each carries its own `k`. Running **does** carry the prescribed tuple back, so it matches on `blockId + weekNumber + sessionKey` (`runLoggedFor(key)`), ignoring date. **Gym** has no client-readable per-day completion, so the backend computes it: `GET /api/train/workouts/today` returns `weekDoneDates` — the ISO dates this Mon–Sun week whose gym instance is **`status = 'completed'`** (`WorkoutService.doneDatesThisWeek` → `WorkoutSessionRepository.findDoneInstanceDates`; **completed-only since `mezo-cd8s`**, not the old ≥1-logged-set — a started-but-unclosed session is settled by the lazy auto-close, §4). `TrainTodayPage` derives `loggedGym = weekDoneDates.includes(today)` for the hero, and per-row `gymLogged`/`isSportLogged(slot)` (matched against each weekly row's computed calendar `date`, the latter also against the slot's sport) so **past** completed days are marked, not just today. **The done-state stores nothing new — it is a pure read-derivation over already-persisted sessions/sets.** In real mode the log itself is persisted to Postgres (`POST /api/train/sport-sessions` → `SportService.create` → `sport_session` table, §4); the flip then happens after `logSportMutation`/`logRunSession` **invalidate→refetch** the session query from the backend (the DB is the source of truth — the same data the Sport `Napló` reads). The `setQueryData` cache-append is **mock-only** (the `mock ? … : …` arm in `trainHooks.ts`): it exists so the UI flips without a backend in tests/prototype, was a silent no-op before, and **never** substitutes for the DB in real mode — mock `sport.week` stats are also not recomputed on append (real mode re-derives them). A timezone edge exists: the backend stamps the log date server-side, so the flip relies on the server "today" matching the browser's local date.
- **Napiv S4 tab rename + pill sub-nav (`mezo-8141`).** The `GYM` tab is now labelled **`Gym`** (`TRAIN_TABS`, `pages/tabs.ts`) — copy-only, no route/id change. `TrainSubNav` switched from the generic `.subnav-item`/`.active` (`--brand-glow` underline) idiom to the coral `.np-pills`/`.np-pill` pill nav (`.on` active state, tinted via the `--pill-accent` custom property, defaulting to `var(--coral)`); Train was the first domain off the old idiom, Fuel (`--sage`) and Me (`--lav`) adopted the same pill markup in S6/S7. **Both `TrainSubNav` and `.np-pills` are since retired (compact-header redesign, `mezo-ugqb`, 2026-07-18)** — Train's sub-nav is now a `SubNavDropdown` popover fed by the unchanged `TRAIN_TABS` array, mounted via `AppHero`'s `utilities` prop (see §2 above).
