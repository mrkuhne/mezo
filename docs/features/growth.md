# Growth — Daily Quests, Activity Log & Gamified Progression

> Surfaces as the **"Napi küldetések"** + **"Tevékenységnapló"** cards on `/today`, a compact **`GrowthSummaryCard`** on `/me` (Profil), the **dedicated `/me/growth` page** (its XP-universe home surface — all-band skills + a 30-day quest/activity journal + computed badges/perks, `mezo-rmhr`), and (E3) the **"Growth — heti"** card on `/insights/weekly`. **Status: ✅ E1 + E2 + E3 + the Growth page done** — daily-quest core shipped 2026-07-11 (`mezo-df7q`, E1); the growth layer (full 8-skill LIFE band + free-text activity log with AI categorization + the GROWTH quest slot + computed discipline/consistency traits) shipped 2026-07-11 (`mezo-jzca`, E2); the **life-integrations polish** (savings aggregate + weekly growth-week aggregate/endpoint + proactive digests + adaptive quest difficulty + companion flavor copy) shipped 2026-07-11 (`mezo-6ng8`, E3); the **Me Growth page** (dedicated `/me/growth` route + quest/activity history reads + a derived-on-read achievements economy; the three former Profil progression cards — athletic radar/muscle levels/LIFE — consolidated into one summary card and **radars retired app-wide**) shipped 2026-07-12 (`mezo-rmhr`). **E4** (shop/coins) remains, under umbrella epic `mezo-52vz`.

## 1. Summary

**Growth** is the gamified personal-growth layer. Two loops feed one XP economy:

1. **Daily side quests** — **3 slots** (BODY + FUELBIO + GROWTH), selected deterministically from a static catalog, completed **by derived evaluation over already-logged data** (never self-claimed), granting XP through the existing progression economy.
2. **Activity log (E2)** — a **free-text mini-journal**: the user types what they did ("olvastam 30 percet", "átraktam 50 ezret megtakarításba"), the **companion cheap-tier LLM proposes** a LIFE skill + XP, and the **server disposes** — clamping/capping the amount deterministically before it lands on one of the **8 LIFE skills**. A confident entry also completes the day's matching open **activity-mode** GROWTH quest.

The psychological contract is [ADR 0010](../decisions/0010-gamified-growth-xp-feedback-not-payment.md): *XP is feedback, not payment* — quests are offers, uncompleted quests expire quietly (no failure state), one reroll/day preserves autonomy, quest XP stays a garnish (15–40/quest), and the **behavior traits are computed/mirrored back, never self-claimed**. The LLM only ever *proposes*; every XP amount that actually lands is deterministic (catalog value for quests, `[xpMin,xpMax]`-clamped + per-skill/per-day-capped for activities).

Status per layer: **backend** ✅ (`feature/quest` + `feature/activity` + progression `QUEST`/`ACTIVITY` sources + full `LIFE` band + `TraitCalculator` + E3's `GrowthWeekService` aggregate, `ActivityLedgerSource` savings port, `QuestSelector` adaptive banding, `QuestFlavor` companion copy, `GrowthDigestBlock` proactive digests), **FE real** ✅ (quest day + reroll, activity log/categorize, the Profil `GrowthSummaryCard` + `/me/growth` page — all-band skills, 30-day journal, badges/perks — incl. the savings stat, Insights Weekly GrowthWeekCard, level-up overlay), **FE mock** ✅ (static seed day, deterministic mock classifier, seed growth-week, seed history + 4/9 badge state). Driving spec: [`2026-07-11-gamified-growth-quests-design.md`](../superpowers/specs/2026-07-11-gamified-growth-quests-design.md) (E3 = §10); plans: [E1 quest-core](../superpowers/plans/2026-07-11-gamified-growth-e1-quest-core.md) · [E2 growth-layer](../superpowers/plans/2026-07-11-gamified-growth-e2-growth-layer.md) · [E3 life-polish](../superpowers/plans/2026-07-11-gamified-growth-e3-life-polish.md).

## 2. User-facing behavior

- **"Napi küldetések" card on Today** (below the check-in strip / companion note, above the workout teaser): each quest shows its HU identity-vote title, benefit-first "why" line, an `+N XP` chip, and a state mark — `◦` offered (brand glow), `✓` completed (success green), `—` expired (dimmed, quiet — no red, per ADR 0010). Header counts `done/total ma`. The GROWTH slot now yields a third quest; an offered **activity-mode** GROWTH quest carries a **`Naplózz` chip** that opens the activity-log sheet pre-banner-ed with the quest (`DailyQuestsCard.tsx:57-63`).
- **Reroll ("Csere"):** offered quests show a `Csere` chip-button while the daily reroll budget (1) lasts; it swaps the quest for the next eligible catalog candidate in the same slot.
- **Derived completion is passive:** the user never taps "done" — logging the underlying thing (weight, water, check-ins, sleep, the planned workout, protein, **a meal cooked from their own recipe**) completes the DERIVED quest on the next Today read (or the nightly cron). A completion detected by the current read fires the global **level-up overlay** exactly once (source chip 📜, headline „Napi győzelem.").
- **"Tevékenységnapló" card on Today (E2)** — always rendered under the quest card (`ActivityLogCard.tsx`). A `+ Bejegyzés` chip opens the **`ActivityLogSheet`**; the day's entries list each show a LIFE-skill emoji, the text, an `+N XP` chip once awarded, and — for an **uncategorized** entry (the AI wasn't confident) — a **`Besorolás?` chip** that reopens the sheet in its picker phase.
  - **`ActivityLogSheet`** is a 3-phase flow (`compose → pick → done`): type free text → **Naplózom** (POST) → if the AI was confident the sheet jumps to `done` (LIFE skill + `+N XP` chip, plus a "Küldetés teljesítve" banner when a matching quest completed), otherwise it drops into `pick` (an 8-button LIFE grid) and the manual choice grants the XP. Any XP-earning write surfaces the **level-up overlay** once. Opened from a quest's `Naplózz` chip it shows the quest banner; opened from `Besorolás?` it starts in `pick`.
- **`GrowthSummaryCard` on Me → Profil + the `/me/growth` page (`mezo-rmhr`)** — the Profil card is a compact whole-card button (`ProfilePage.tsx:35`) → `/me/growth`: `{total} XP` chip (FE-summed across all bands), an Atléta-szint/Streak/Fegyelem line, the **top-3 skills across all bands**, and a "Megtakarítás (30 nap)" row when `savingsHuf30d > 0`; before any XP it renders the **ghost** ("Az élet is edzés." — the retired LIFE card's copy verbatim, still tapping through). The **`/me/growth` page** (`GrowthPage.tsx`) is the full surface: a hero trio (**Össz XP** FE-summed · **Fegyelem** `disciplinePct` · **Ritmus** `consistencyWeeks` hét) over a segmented **Skillek** / **Napló** / **Kitüntetések** control. *Skillek* = 3 `SkillBandCard`s listing ALL LIFE/athletic/muscle skills (icon · name · `Lv n` · progress bar · cumulative XP; LIFE footer = the savings row). *Napló* = a `GrowthJournalCard` over a fixed 30-day window (quests + activities merged by `buildGrowthJournal`). *Kitüntetések* = `BadgesCard` (9 derived badges) + `PerksCard` (perk unlocks). The **radar/octagon diagrams retired app-wide** (user decision); the LIFE band is now meter rows. Me-side detail: [me.md §2 `Growth`](me.md).
- **"Growth — heti" card on Insights → Weekly (E3)** — a third card under the review + tervjavaslat (`GrowthWeekCard.tsx`): the week's **Küldetések** `done/closed`, **LIFE XP** `+N`, **Tevékenységek** count, and a **Megtakarítás** row (`{amount} Ft`, shown only when > 0). A week with no growth data renders the honest empty line „Még nincs growth-adat ezen a héten." (documented under [insights.md §2.2](insights.md)).
- **Adaptive difficulty (E3, invisible):** the quest offered per slot is drawn from a difficulty band derived from the user's trailing-28-day completion ratio for that slot — there is no user-facing control or label; the effect is that a slot the user keeps completing offers harder tiers, one they keep letting expire falls back to the easiest (§3). Availability always wins over difficulty, so a slot never goes empty.
- **Companion flavor copy (E3, morning cron only):** on the morning-generate cron the companion cheap-tier LLM may rewrite each fresh quest's **title/why** in its own voice (targets/XP/skill untouched); lazily-generated quests and rerolls keep the catalog copy. Purely a copy change — no new affordance.
- **Empty states:** the quest card renders nothing on a day with no rows; the activity card always renders (with a "Mi történt ma?" prompt when empty); the Profil `GrowthSummaryCard` ghosts until the first XP; the Growth page's Napló/Kitüntetések tabs render honest empty lines (no journal entries / 0-of-9 badges) before any activity; the Insights GrowthWeekCard shows its honest empty line on a zero week.

## 3. Architecture & data flow

**Quests:** `DailyQuestsCard` → `useDailyQuests(date)` / `useQuestActions(date)` (from `@/data/hooks`) → mock: `questMock.ts` seed · real: `questApi.ts` → `GET /api/quest/day/{date}` → `QuestService.getDay` → lazy `QuestSelector.generate` (today only, guarded against the morning-cron race) + `QuestService.evaluateAndFinalize` (complete/expire + `ProgressionService.applyQuest`) → `daily_quest` table.

- **Generation is deterministic over 3 slots** (`QuestSelector.SLOTS` = BODY, FUELBIO, GROWTH): seeded by `(userId, date, slot)` hash over the eligible catalog pool — filters: slot, day type (planned gym template via `WorkoutService.findPlannedTemplateForDate` → GYM, else REST), goal-prescription requirement, per-key cooldown (yields to availability), distinct metrics per day. No LLM anywhere in the economy (ADR 0010).
- **Evaluation is DERIVED-guarded** (`QuestService.evaluateAndFinalize`): only `MODE_DERIVED` quests auto-complete via `QuestEvaluator.satisfied` (pure reads — check-in count, weight-log presence, water sum, sleep duration, fuel-day protein, gym done-dates, **own-recipe meal**). **`MODE_ACTIVITY` quests never auto-complete** — they complete only through the activity-write hook, and otherwise **expire quietly** like any offered quest once the day passes.
- **XP path (quests):** completion → `ProgressionService.applyQuest(QuestSignal)` → the shared idempotent `award(...)` tail (`source_type=QUEST`, `source_ref_id=questId`) → `skill_progress`/`level_up_event`; gated by `ObjectProvider<ProgressionGate>`.

**Activity log (E2):** `ActivityLogCard`/`ActivityLogSheet` → `useActivities(date)` / `useActivityActions(date)` (from `@/data/hooks`, dual-mode via `useDualQuery`) → `activityApi.ts` → `POST /api/activity` (+ `GET /api/activity/day/{date}`, `POST /api/activity/{id}/category`) → `ActivityService`.

- **Write flow** (`ActivityService.create`): persist the entry → `ActivityClassifier.classify` (companion cheap tier) proposes `{skillKey, confidence, xpSuggestion(5..25), durationMin?, amountHuf?}` → the suggestion is **clamped** into `[xpMin,xpMax]` and stored in `xp_suggested`; **confident** (`skillKey` set, a LIFE key, `confidence ≥ confidenceThreshold` 0.6) → set `skill_key` + `categorized_by = AI` → **`grantXp`** awards `min(xpSuggested, perSkillDailyCap − skillUsedToday, dailyCap − dayUsedToday)` and, when > 0, calls `ProgressionService.applyActivity` (source `ACTIVITY`, idempotent per entry id) → then **`completeMatchingActivityQuest`** completes the day's first open activity-mode quest whose LIFE skill matches, stamping `source_activity_id` (quest XP rides on top). **Low confidence / classifier off** → the entry lands **uncategorized** (`skill_key` null), **no XP yet**.
- **Categorize / override** (`ActivityService.categorize`, `POST /api/activity/{id}/category`): validates the key against `ProgressionTaxonomy.LIFE`. First categorization of an uncategorized entry (`xp_awarded == 0`) → `grantXp` within the day's remaining caps. **Override** of an already-awarded entry → `ProgressionService.moveActivityXp(from, to, xp)` — a **direct skill-row adjustment** (decrement the old LIFE row, increment/create the new one, recompute levels), **no new `level_up_event`**. Either path also runs `completeMatchingActivityQuest`. `categorized_by` becomes `USER`.
- **Response shape** (`ActivityWriteResponse`): `{ entry, completedQuest?, levelUps[] }` — 0–2 level-up payloads (the activity award and/or the completed quest's award). The FE surfaces the first payload with a level-up through the global overlay.

**Traits (E2, read-time):** `ProgressionService.getProfile` calls `TraitCalculator.traits(createdBy, today)` (`TraitCalculator.java`):
- **Discipline** (`disciplinePct`, nullable): a 28-day (`DISCIPLINE_WINDOW_DAYS`) completion ratio, the **average of the available components** — planned-vs-done training (`TrainingCommitmentSource.commitmentStats`, implemented in train by `TrainingCommitmentCalculator`) blended with closed-quest `completed / (completed + expired)` (`QuestLedgerSource.closedQuestStats`, implemented in quest by `QuestLedgerAdapter`, consumed via `ObjectProvider` since the quest switch may be off). **Null** (not 0) when there are no commitments in the window.
- **Consistency** (`consistencyWeeks`): the current streak of consecutive ISO weeks with **≥ 4 active days** (`ACTIVE_DAYS_PER_WEEK`), an active day = any XP-earning row in `level_up_event.occurred_at` (`findOccurredAtSince`, 400-day horizon). The **running week can't break the streak** — it counts only once it already meets the bar (grace, mirrors the robustness streak's tone).

Both ports (`TrainingCommitmentSource`, `QuestLedgerSource`) keep the dependency **one-directional** (train/quest → progression, never back — `feature_slices_are_cycle_free`), the same pattern as `RobustnessSource`.

**E3 — life integrations (`mezo-6ng8`):** three read-time aggregates + two generation-time behaviors, all rule-based (no LLM in the economy — ADR 0010; the LLM only ever rewrites flavor copy):

- **Savings port + 30-day profile stat:** a new `ActivityLedgerSource` port (`feature/progression/ActivityLedgerSource.java`, `Stats(entries, savingsHuf)`) is implemented in `feature/activity` by `ActivityLedgerAdapter` (`ACTIVITY_SWITCH`-gated) — it sums `extracted.amountHuf` over the window's `skill_key='financial'` entries. `ProgressionService.getProfile` consumes it via `ObjectProvider` and adds **`savingsHuf30d`** (trailing 30 days, `now-29..now`; **null** when the activity feature is off) to `ProgressionProfileResponse` — the Me `GrowthSummaryCard` + Growth-page LIFE-band "Megtakarítás (30 nap)" row.
- **Growth-week aggregate + endpoint:** `GrowthWeekService.growthWeek(userId, anyDayOfWeek)` keys the **ISO week** (`anyDayOfWeek.with(MONDAY)`, system zone — the `TraitCalculator` precedent) and composes: quest `completed`/`closed` (= completed+expired) from `QuestLedgerSource.closedQuestStats`, **LIFE XP** = the sum of `level_up_event` payload gains with `kind='LIFE'` occurring in `[weekStart 00:00, weekEnd+1 00:00)`, and activity count + savings from `ActivityLedgerSource` — both ledger ports via `ObjectProvider`, so an off switch yields **honest zeros, never an error**. Served by `GET /api/progression/growth-week/{date}` (§4); also consumed by the proactive `GrowthDigestBlock` (§5).
- **Adaptive difficulty banding** (`QuestSelector.allowedDifficulties`): per slot, count the trailing-`window-days` (28) **closed** rows for that slot — `completed` vs `expired`. `< min-sample` (5) closed → all tiers `{1,2,3}` (v1 behavior, insufficient signal); ratio `completed/closed ≥ high-ratio` (0.85) → `{1,2,3}`; `≤ low-ratio` (0.50) → `{1}`; else `{1,2}`. In `pick`/`replacement` the banded pool is filtered from the eligible pool, but **difficulty yields to availability** (empty banded pool → fall back to the full eligible pool), and only **then** does the existing **cooldown yield to banded**. Reads only terminal statuses — offered/rerolled rows never count.
- **Flavor copy flow** (`QuestFlavor.rewrite`, gated `QUEST` + `COMPANION` + `mezo.quest.flavor.enabled`): called by `QuestJob.runGenerate` via `ObjectProvider` right after `QuestSelector.generate` (the **morning cron path only**). It builds one prompt (marker `KULDETES-IZESITES-FELADAT`) listing the fresh quests, calls `CompanionLlm.complete` (cheap tier), and expects a strict-JSON array of `{title, why}` with **exactly** the same count/order. Any failure — LLM error, unparseable, count mismatch, a blank/over-length entry (`title ≤ 160`, `why ≤ 600`) — **quietly keeps the catalog copy** (the offer is already correct); valid entries are persisted per-quest. `FakeCompanionLlm` mirrors the marker and reads a `[fake-quest-flavor:[…]]` sentinel (default `[]` = no rewrite), so ITs are provider-free.

- **Dual-mode seams:** quests use the `useChallenges` manual shape (`levelUps`/`rerollsLeft` ride beside the list); activities use `useDualQuery` (real → `[]` while unresolved, never the mock seed). `useActivityActions` invalidates `['activities', date]` + `['dailyQuests', date]` + `['progressionProfile']` on a real write so the activity card, the quest card, and the Profil `GrowthSummaryCard` all refresh. Mock mode has a deterministic classifier (`activityHooks.ts:mockWrite` → always `learning`, 15 XP) so the whole flow is demoable offline. **E3's growth-week** rides `useWeekly()` in the Insights domain (`weeklyHooks.ts`, real-only `['insightsWeekly','growth', start]` query → `growthWeekApi.get`, `.catch(()=>null)`; mock = the `insights.ts` seed) — see [insights.md §3](insights.md).
- **Growth page reads (`mezo-rmhr`):** the `/me/growth` Napló + Kitüntetések tabs add three `useDualQuery` reads — **`useQuestHistory(from,to)`** (`data/quest/questHooks.ts`, `['questHistory', from, to]` → `GET /api/quest/history` → `QuestService.history`, rerolled rows filtered server-side) + **`useActivityHistory(from,to)`** (`data/activity/activityHooks.ts`, `['activityHistory', from, to]` → `GET /api/activity/history`), merged by the pure **`buildGrowthJournal`** (`features/me/logic/growthJournal.ts`); and **`useAchievements()`** (`data/progression/progressionHooks.ts`, `['achievements']` → `GET /api/progression/achievements` → `AchievementService`). All barrel-exported, honest-empty in real mode, with deterministic mock seeds (`questMock.mockQuestHistory` / `activityMock.mockActivityHistory` / `achievementsMock` mirroring the mockup's 4/9 badge state) + MSW defaults. The Skillek tab + hero read the existing `useProgressionProfile()` (no new profile field — Össz XP is FE-summed).

**Achievements (`mezo-rmhr`, read-time):** `AchievementService.achievements(createdBy)` (`feature/progression/service/AchievementService.java`) derives **9 badges on read** from the EXISTING seams — no new tables, deterministic, retroactive. Inputs: `QuestLedgerSource.closedQuestStats` (all-time `completed` → first_quest/quests_10/quests_50), `ActivityLedgerSource.stats` (all-time `entries` → first_activity + `savingsHuf` → savings_100k) — both via `ObjectProvider` so a switch-off port reports `current=0` honestly —, `TraitCalculator.traits(...).consistencyWeeks` (rhythm_4w), and `SkillProgressRepository` LIFE rows (all_life_active / life_lv5 / life_xp_10k). Each badge is `{key, icon, name, achieved = current >= target, current, target}`. **Perks** come straight from `PerkUnlockRepository.findByCreatedByOrderByUnlockedAtDesc` joined with `PerkCatalog` copy. **No unlock dates on the badges** (there is no unlock ledger — §9).

## 4. Data model & API

- **Table `daily_quest`** (E1, migration [`202607111300_mezo-df7q_create_daily_quest.sql`](../../backend/src/main/resources/db/changelog/1.0.0/script/202607111300_mezo-df7q_create_daily_quest.sql)): `quest_date`, `slot` (BODY|FUELBIO|GROWTH), `catalog_key`, `skill_key`+`skill_kind` (ATHLETIC|MUSCLE|LIFE), HU `title`/`why`, `completion_mode` (DERIVED|ACTIVITY), `target` jsonb (`QuestTargetEnvelope{metric, threshold}`), `xp`, `coins` (always 0 until the E4 shop), `status` (offered|completed|expired|rerolled), `source_activity_id` (stamped by an activity-completed quest). Identity: partial unique `(created_by, quest_date, slot)` among non-rerolled, non-deleted rows. The E1 migration relaxed two released CHECKs additively: `level_up_event.source_type` += `QUEST`, `skill_progress.skill_kind` += `LIFE`.
- **Table `activity_log`** (E2, migration [`202607112000_mezo-jzca_create_activity_log.sql`](../../backend/src/main/resources/db/changelog/1.0.0/script/202607112000_mezo-jzca_create_activity_log.sql)): `id uuid pk`, `created_by`, soft-delete cols, `occurred_on date`, `text`, `skill_key varchar(40)` (null = uncategorized), `confidence numeric(4,3)`, `xp_awarded int` (clamped+capped, actually granted), `xp_suggested int` (the clamped AI proposal — so a later manual categorization grants a deterministic amount), `extracted jsonb` (`ActivityExtract{durationMin, amountHuf}`), `categorized_by varchar(6)` (`ck_…` AI|USER). Index `idx_activity_log_user_day (created_by, occurred_on) where is_deleted = false`. The **same migration relaxes `level_up_event.source_type` additively: += `ACTIVITY`**.
- **Catalog** (`content/quest-catalog.json`, fail-fast `QuestCatalog` loader — the `PerkCatalog` pattern): **14 entries** — 2 BODY + 5 FUELBIO + **7 GROWTH**. Each `QuestDef` now carries a **`mode`** (DERIVED|ACTIVITY). The 7 GROWTH entries: 6 **ACTIVITY** (`growth_read`/learning, `growth_gratitude`/mindset, `growth_mindfulness`/mindfulness, `growth_finance_review`/financial, `growth_deepwork`/productivity, `growth_connect`/connection — all `metric=activity_match`) + 1 **DERIVED** (`growth_cook_own`/cooking, `metric=own_recipe_meal`). `bio_protein.threshold` still resolves at generation time from the goal prescription's current segment.
- **Quest contract** ([`api/feature/quest/quest.yml`](../../api/feature/quest/quest.yml), tag `Quest`): `GET /api/quest/day/{date}` → `QuestDayResponse{date, quests[], levelUps[], rerollsLeft}`; `POST /api/quest/{id}/reroll` → `QuestResponse` (409: `QUEST_NOT_OFFERED`/`QUEST_NOT_TODAY`/`QUEST_REROLL_EXHAUSTED`/`QUEST_REROLL_NO_ALTERNATIVE`). **`QuestResponse.completionMode`** (DERIVED|ACTIVITY) is now on the wire so the FE can render the `Naplózz` affordance. **`GET /api/quest/history?from&to` (`mezo-rmhr`)** → `QuestResponse[]` — non-rerolled quests in the inclusive range, **newest date first** (`QuestService.history`, `findByCreatedByAndQuestDateBetweenOrderByQuestDateDesc`, rerolled filtered in-service); `from > to` → 400 `QUEST_INVALID_DATE_RANGE`. Feeds the Growth journal.
- **Activity contract** ([`api/feature/activity/activity.yml`](../../api/feature/activity/activity.yml), tag `Activity` → `ActivityApi`, `ActivityController implements ActivityApi`, gated on `mezo.feature.activity.enabled`):
  - `POST /api/activity` — `ActivityCreateRequest{text (≤500), occurredOn?}` → **200** `ActivityWriteResponse{entry, completedQuest?, levelUps[]}`; blank text → 400 `ACTIVITY_TEXT_REQUIRED`.
  - `GET /api/activity/day/{date}` → `ActivityResponse[]` (newest first; **empty array, never 404**).
  - `GET /api/activity/history?from&to` (`mezo-rmhr`) → `ActivityResponse[]` — entries in the inclusive range, newest first (E3's `findByCreatedByAndOccurredOnBetween` + ordering); `from > to` → 400 `ACTIVITY_INVALID_DATE_RANGE`. Feeds the Growth journal.
  - `POST /api/activity/{id}/category` — `ActivityCategoryRequest{skillKey}` → 200 `ActivityWriteResponse`; unknown LIFE key → 400 `ACTIVITY_SKILL_UNKNOWN`; missing/foreign entry → 404 `ACTIVITY_NOT_FOUND`.
- **Profile contract** ([`api/feature/progression/progression.yml`](../../api/feature/progression/progression.yml)): `GET /api/progression/profile` → `ProgressionProfileResponse` carries **`life[]`** (all 8 LIFE `SkillLevel`s in taxonomy order — missing row → level 1, 0 XP), **`traits`** (`ProfileTraits{disciplinePct nullable, consistencyWeeks}`), and (E3) **`savingsHuf30d`** (`int64`, nullable — trailing-30-day sum of AI-extracted `amountHuf` on `financial` activity entries; **null** when the activity feature is off).
- **Growth-week contract** (E3, same fragment): `GET /api/progression/growth-week/{date}` → **`GrowthWeekResponse{weekStart (ISO Monday), questCompleted, questClosed (=completed+expired), lifeXp (int64), activities, savingsHuf (int64)}`** — honest zeros, **never a 404**. Both endpoints ride the switch-gated `ProgressionController` (progression off → 404).
- **Achievements contract** (`mezo-rmhr`, same fragment, tag `Progression`): **`GET /api/progression/achievements`** → **`AchievementsResponse{ badges: BadgeResponse[], perks: PerkUnlockResponse[] }`** — `BadgeResponse{key, icon, name, achieved, current (int64), target (int64)}` (current/target populated even for achieved badges; the FE shows ✓ only), `PerkUnlockResponse{perkKey, name, effectCopy, skillKey, milestoneLevel, unlockedAt}` (from `perk_unlock` joined with `PerkCatalog` copy, newest first). Served by `AchievementService.achievements` — the 9 badges **derived on read** (§9), NO new tables/DDL; also on the switch-gated `ProgressionController` (off → 404 → FE honest-empty).
- **FE types:** `DailyQuest`/`QuestSlot`/`QuestStatus` + `ActivityEntry`/`LifeSkillKey` in `data/types.ts`; wire types off `api.gen.ts` in `data/quest/questApi.ts` + `data/activity/activityApi.ts`.

## 6. How to use it (consume)

```ts
import { useDailyQuests, useQuestActions, useActivities, useActivityActions } from '@/data/hooks'
const { quests, levelUps, rerollsLeft } = useDailyQuests(date)         // date = 'YYYY-MM-DD'
const { reroll } = useQuestActions(date)
const { data: entries } = useActivities(date)                          // ActivityEntry[]
const { logActivity, categorize, pending } = useActivityActions(date)
await logActivity('Olvastam 30 percet')                                // → ActivityWriteResult (entry + completedQuest? + levelUps)
await categorize(entryId, 'learning')                                  // pick/override a LIFE skill
```
`quests` excludes rerolled rows; `quests[i].completionMode === 'ACTIVITY'` marks the `Naplózz`-affordance quests. `levelUps` on the quest read is non-empty only on the read that performed the completion. `logActivity`/`categorize` return the write result — feed `result.levelUps` to `showLevelUp(...)` (the sheet does this) and read `result.completedQuest` for the "quest completed" banner. Real mode returns the empty day while loading (no seed fallback). Query keys: `['dailyQuests', date]`, `['activities', date]`.

## 7. How to extend it

- **New LIFE skill:** add the token to `ProgressionTaxonomy.LIFE` (backend, taxonomy order) **and** the matching `{key, name, icon}` to `LIFE_SKILLS` in `frontend/src/features/progression/logic/levelUpMeta.ts` — the two lists must agree in order and membership. The classifier prompt (`ActivityClassifier.CLASSIFY_PROMPT`) enumerates the keys, so extend it too, and add the new key to the `LifeSkillKey` FE type. The Growth page's LIFE `SkillBandCard` + journal render whatever `LIFE_SKILLS` carries, so no card change is needed (the `all_life_active` badge target `8` in `AchievementService` would need bumping if the LIFE band grows).
- **New GROWTH quest:** add a catalog entry with `slot: "GROWTH"` + a `mode`. An **ACTIVITY** quest (`mode: "ACTIVITY"`, `metric: "activity_match"`) completes via a matching activity-log entry — no evaluator work needed. A **DERIVED** quest needs a new `metric` case in `QuestEvaluator.satisfied` + a label in `QuestDisplay.targetLabel`. Quest XP must stay in the 15–40 band (loader fails fast otherwise).
- **New derived metric:** add the `case` to `QuestEvaluator.satisfied` reading the owning feature's repository/service (pure reads only — honest completion). `own_recipe_meal` is the template for a Fuel-sourced metric.
- **New achievement badge (`mezo-rmhr`):** append a `badge(key, icon, name, current, target)` row to the ordered list in `AchievementService.achievements` — source `current` from an EXISTING seam (`QuestLedgerSource`/`ActivityLedgerSource`/`TraitCalculator`/`SkillProgressRepository`), NO new port/table. The generic `BadgesCard` renders it with zero FE change; update `AchievementsApiIT` + the mock seed `data/progression/achievementsMock.ts` + the MSW default. Unlock **dates** are deliberately out of scope in v1 (badges are derived on read — §9).
- **Tune the activity economy:** all guardrails are `ActivityProperties` (`mezo.activity.*`) — never hardcode XP/caps/thresholds (`configuration_conventions.md`).
- **Tune adaptive difficulty (E3):** `mezo.quest.adaptive.*` (`window-days`/`min-sample`/`high-ratio`/`low-ratio`, bound + validated by `QuestProperties.Adaptive`). To widen/narrow a band edit the ratios; to make difficulty static again set `min-sample` above any realistic closed-count (all slots then fall to `{1,2,3}`). A new difficulty tier needs a wider catalog `difficulty` range (the loader currently pins 1..3) **and** a band that admits it.
- **Disable/adjust flavor copy (E3):** flip `mezo.quest.flavor.enabled` off to keep catalog copy everywhere; the rewrite is otherwise silent-fail-safe. It runs **only** on `QuestJob.runGenerate` (via `ObjectProvider<QuestFlavor>`) — do NOT wire it into the lazy read or reroll paths (that would rewrite copy the user has already seen). Extend the prompt/limits in `QuestFlavor` (`TITLE_MAX`/`WHY_MAX`), never let it touch targets/XP/skill (ADR 0010).
- House standards: contract-first (`api_contract_conventions.md`), backend per `docs/references/*.md`, dual-mode hook recipe in [`_platform-data-layer.md`](_platform-data-layer.md).

## 9. Decisions, gotchas & deferred

- **Mock fidelity note (mezo-rmhr fix wave):** `progressionProfileMock.athletic` carries the full 12-entry band (incl. `robustness`) to mirror the real profile response, so all-band row counts and FE-summed XP literals match between modes.

- **[ADR 0010](../decisions/0010-gamified-growth-xp-feedback-not-payment.md)** is the contract every extension must respect (no failure states, no XP-gated content, no gambling mechanics, coins only with a shop, **the LLM proposes but the server disposes**, traits computed not self-claimed).
- **Gotcha — XP move keeps the original event payload:** an activity **category override** adjusts the LIFE `skill_progress` rows directly (`moveActivityXp`) and writes **no new `level_up_event`**; the original grant's ledger event still names the *old* skill. An accepted correction-history trade-off (documented in `ProgressionService.moveActivityXp`), not a bug.
- **Gotcha — uncategorized entries carry 0 XP until a pick:** a low-confidence (`< 0.6`) or classifier-off entry lands with `skill_key` null and `xp_awarded` 0; the stored `xp_suggested` (already clamped) is what a later manual categorization grants, within the day's remaining caps.
- **Gotcha — the caps make bad-faith logging economically irrelevant:** every activity is clamped to `[5,25]` XP and bounded by `per-skill-daily-cap` (40) and `daily-cap` (100). Spamming the log can't out-earn honest training, so the honor system needs no anti-cheat.
- **Gotcha — ACTIVITY quests don't auto-complete:** `evaluateAndFinalize` completes only DERIVED quests; an activity-mode GROWTH quest completes solely via `completeMatchingActivityQuest` on an activity write, and otherwise **expires quietly** with the day (no failure state).
- **Gotcha — robustness cannot carry quest XP** (E1, still true): the award tail recomputes `robustness` to an absolute streak target, wiping additive deltas — that is why FUELBIO quests route to the `recovery` LIFE skill.
- **Gotcha — flavor is cron-only (E3):** `QuestFlavor.rewrite` runs solely on `QuestJob.runGenerate`; the lazy `GET /api/quest/day/{date}` generation and rerolls **keep catalog copy** by design (re-flavoring copy the user has already seen would be jarring). Any rewrite failure — LLM error, unparseable JSON, count mismatch, blank/over-length entry — silently keeps the catalog copy: flavor is a garnish, never load-bearing.
- **Gotcha — savings counts only AI-extracted `financial` amounts (E3):** `savingsHuf30d` (and the growth-week savings) sum `extracted.amountHuf` **only over entries the classifier categorized to the `financial` LIFE skill** with a non-null extracted amount. A savings-like entry the AI mis-skilled, or one with no amount extracted, contributes nothing; a manual category override to/from `financial` shifts the LIFE XP row but does **not** re-extract an amount (the stored `extracted` envelope is set once at write). Deterministic and honest — no anti-cheat needed.
- **Gotcha — week attribution splits on `occurred_on` vs `occurred_at` (E3):** the growth-week `activities`/`savingsHuf` count by the entry's `occurred_on` date, while `lifeXp` sums `level_up_event.occurred_at` (the instant the XP was granted). A back-dated activity therefore lands in the past week's activity/savings numbers but in the current week's LIFE XP — inherent to the ledger design (mirrors the trait computation), not a bug.
- **Gotcha — adaptive reads only terminal statuses (E3):** the banding counts `completed`/`expired` closed rows in the trailing window; `offered`/`rerolled` never count, and difficulty always yields to availability then to the cooldown rule — so a sparse history simply falls back to all tiers, never an empty slot.
- **Gotcha — badges are derived on read, with NO unlock dates (`mezo-rmhr`):** the 9 achievement badges are computed each read by `AchievementService` from the live ledgers — deterministic, retroactive (a returning user's whole history counts), zero migration cost — so there is **no unlock ledger and no "unlocked on ⟨date⟩" label** (a deliberate deviation from the mockup's `✓ Júl 11` labels). A **persistent achievement table with unlock timestamps + a first-earn celebration overlay** is the recorded **future upgrade**; until then the badge only knows *achieved / current / target*, never *when*. Perks, by contrast, DO carry a real `unlockedAt` (they are persisted `perk_unlock` rows). A switch-off port makes its dependent badges honestly report `current=0`, not an error.
- **Gotcha — Össz XP is FE-summed (`mezo-rmhr`):** the hero "Össz XP" + the summary card's total chip sum `cumulativeXp` across all three bands **on the frontend** from the existing profile response — there is no backend total field. If a new band is added, the FE sum picks it up automatically; a backend total would need a new profile field.
- **✅ E3 shipped (`mezo-6ng8`):** savings aggregate + weekly growth-week endpoint + proactive digests + adaptive difficulty + companion flavor copy. **✅ Me Growth page shipped (`mezo-rmhr`):** dedicated `/me/growth` route (all-band skills + 30-day journal + derived badges/perks) + quest/activity history reads + the `AchievementService`; the three former Profil progression cards consolidated into one `GrowthSummaryCard`, radars retired app-wide. **Deferred (E4):** shop/coins (the `coins` column + catalog field exist, always 0); persistent achievements with unlock dates + celebration overlay. Umbrella epic `mezo-52vz`.
